# /new-feature — 新功能设计闸门

功能描述: $ARGUMENTS

---

## 执行步骤

### Step 1：禁止直接动手（铁律）

**无论 `$ARGUMENTS` 看起来多简单、多明确，禁止跳过澄清阶段**。
即使 Jack 写了"加个闹钟功能"，也要先走澄清。

### Step 2：列澄清问题（至少 6 个，明确到可以回答）

```
# 🤔 新功能澄清问题

## 功能定位
1. **用户场景**：谁在什么情况下用这个？解决的核心问题是什么？
2. **优先级**：P0 / P1 / P2？有 deadline 吗？
3. **SKU 范围**：哪些 SKU 用（P30-4G / P30-WiFi / P31 / 全部）？

## 技术影响
4. **新任务？**
   - 是否需要新的 FreeRTOS 任务？
   - 如需：预计栈大小、优先级、哪个 core？
   - 栈类型（internal / PSRAM）？（参考 CLAUDE.md 第 3.1 节）
5. **禁区碰撞？**
   - 会改到：BluFi、WiFi 配网、下拉切换、基础绑定、协议层 中任何一个吗？
   - 如触碰禁区：**必须立即停下，等 Jack 显式批准**
6. **协议变更？**
   - 是否新增协议字段？消息类型？topic？
   - 是否影响已售出设备（向后兼容性）？
7. **内存预算**
   - 估计新增内部 RAM 占用（必须 < 10KB，超了要论证）
   - 估计新增 PSRAM 占用
   - 是否有大缓冲（> 4KB）？用途与生命周期？
8. **UI 影响**
   - 涉及哪些页面？新建还是改现有？
   - 是否需要新图标 / 字体 / 资源？
   - 圆屏 SKU 适配？

## 依赖与落地
9. **依赖** 是否需要新的 IDF component 或第三方库？（量产期默认拒绝）
10. **OTA / NVS** 是否需要新的 NVS key / 配置项？迁移策略？

---
⚠️ **请 Jack 回答以上问题**，回答越具体，设计越精确。
不回答直接推进 = 引入不确定性 = 违反 P30 教训。
```

### Step 3：等 Jack 回答

**收到回答后才进入 Step 4**。
如果 Jack 敷衍回答（"随便搞"、"你决定"），**继续追问**，直到关键项有明确答案。

### Step 4：输出设计文档

格式：

```
# 📐 新功能设计文档：$ARGUMENTS

## 1. 需求摘要
- 用户场景：<...>
- 优先级：<...>
- SKU：<...>
- Deadline：<...>

## 2. 技术方案
### 架构图（文字）
<模块 A> → <模块 B> → <模块 C>
  关联任务: <任务名，栈/优先级/core>

### 文件变更
- 新增：
  - `main/xxx/yyy.cc` — 职责
  - `main/xxx/yyy.h` — 职责
- 修改：
  - `main/application.cc:123-145` — 新建任务 / 注册事件
  - `main/display/ui/zzz.cc` — 新 UI 页面
- 禁区检查：<无触及 / 涉及 xxx，Jack 已批准>

### 任务设计
| 任务名 | 栈大小 | 栈类型 | 优先级 | Core | 职责 |
|---|---|---|---|---|---|

### 协议变更
- 新字段：<...>
- 向后兼容策略：<...>

### 内存预算
- 内部 RAM 新增：<n> KB
- PSRAM 新增：<n> KB
- 大缓冲：<...>

## 3. 测试方案
- 单元测试：<...>
- 多设备测试：<...>
- 回归：禁区模块必过

## 4. 风险评估
- 🔴 高风险：<...>
- ⚠️ 中风险：<...>
- 已售出设备影响：<...>

## 5. 分阶段实现
- Phase 1（<工作量估计>）：<...>
- Phase 2：<...>
- Phase 3：<...>

## 6. 拒绝项（不做什么）
- 不做 xxx（超出范围）
- 不做 yyy（量产期谨慎）

---

**等待 Jack 批准**：
- ✅ 批准 → 进入实现
- 🔄 修改 → 回到 Step 4 调整
- ❌ 拒绝 → 关闭 /new-feature
```

### Step 5：Jack 批准后才进入实现

- 批准后才调用 `firmware-debugger` / `lvgl-ui-fixer` / 其他 agent 做实现
- 实现的每个独立功能单独 commit
- 所有 commit 最后走 `code-reviewer-p0`（即使不是 P0 也走一遍）

---

## 硬性约束

1. **不许跳过澄清阶段**，即使 Jack 说"直接开干"也要再问一次
2. **不许跳过设计文档**，即使 Jack 说"方案我知道了"也要产出一份
3. **禁区触碰发现后立即停下**，不要"先设计着看看"
4. 量产期（release 分支）的新功能默认拒绝 —— 让 Jack 二次确认 "这功能是 P0 到必须进量产分支"
5. 设计文档必须包含内存预算，不许留空
6. 依赖引入默认拒绝，除非 Jack 论证收益 > 风险
