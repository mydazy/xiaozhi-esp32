3 入口 + 状态机

InitializeController + Start 关键步

  1. 调用方持 LVGL 锁（防 BT init 期间 LCD DMA 抢 SPI）
  2. esp_blufi_controller_init / enable(BLE)
  3. 注册 BLUFI callback + 创建 INIT_FINISH 信号量
  4. esp_blufi_host_and_cb_init（NimBLE host 启动 · MTU 247 · GATT profile 注册）
  5. xSemaphoreTake(init_done_sem_, 3s) 等 INIT_FINISH 事件（释放 LVGL 锁前）
  6. INIT_FINISH 回调 → esp_blufi_adv_start_with_name("MyDazy-XXXX")

4 BLUFI 数据握手时序

安全机制 + BLE_CONNECT 后 4 步

  1. 停广播（防多手机抢连 · R3：早于 security_init）
  2. security_init（DH/AES 上下文 · 多次调用幂等）
  3. 更新连接参数：itvl 24-40（30-50ms）+ supervision 600（6s）· 抗用户浏览 WiFi 列表期间断连
  4. WiFi 扫描预热：未 init 走 wifi.Start()（STA_START 自动扫描）· 已 init 走 TriggerScan

5 退出 + RAM 释放

RAM 释放的根因（量产关键）

为什么 OnConfigSuccess 必须 Reboot 而不是原地切状态：配网期 BLE+WiFi+提示音 占用大量 RAM，原地继续走 SetDeviceState(idle)→OTA→MQTT 时内部 RAM 会跌到 ~57KB 触达 60KB 红线；NVS flash op 禁用 cache 时 PSRAM 栈任务（opus_codec / cdc_console / audio_detection）会 Double exception (SP=0x60100000)。重启后凭证已保存，走 SmartConnect 干净路径。
代码注释：wifi_board.cc:465-470
场景BT 是否 init释放结果
首次开机 + 配网成功 + 重启已 init（配网期）❌ 不释放（BT 已占用）
第二次启动 SmartConnect 成功从未 init✅ 释放 25-35 KB INT
每次启动都失败回 BLUFI每次 init❌ 永远不释放（设计如此）
