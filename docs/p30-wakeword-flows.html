1 概览

精简后两种实现共生于 main/audio/wake_words/，由 AudioService::SetModelsList 按"模型 bin 包含什么"二选一替换。量产 SKU（P30-WiFi/4G/P31）走 AfeWakeWord（S3 + PSRAM 默认路径），具备 AEC + Speaking 中打断能力；CustomWakeWord 为离线自定义唤醒词备选。已删 2026-05-13 · 旧裸 WakeNet 兜底实现已下线（量产板均 S3，非 S3 路径直接 nullptr · 详见 § 2.1 精简纪事）。

1.2 推荐量产配置

推荐 AfeWakeWord（量产 SKU 默认路径）
理由（按权重）：
  1. Speaking 中可打断 · application.cc:1117 显式判 IsAfeWakeWord()，是 P30 用户体验的核心场景
  2. 内建 AEC · 喇叭外放不会自激
  3. 与 AfeAudioProcessor 解耦：唤醒态走 AFE_TYPE_SR，对话态走 AFE_TYPE_VC，两套独立 afe_data（注：1.5MB PSRAM × 2 共存，P1 范围）
  4. SetModelsList 默认选它（模型 bin 不含 MN 即走 AFE）· audio_service.cc:780-789
  5. 上游 xiaozhi-esp32 主战路径，rebase 风险最小
备选 CustomWakeWord（自定义唤醒词路径 · 也属量产可选）
适用场景：① 客户要求设备名/品牌名自定义唤醒（如"嘿小爱灵芯"非标准词）② 离线场景需要命令词分支（虽然本仓库 commands[] 只在 action=="wake" 分支回调）③ 模型 bin 已经包含 MultiNet（如 nihaoxiaozhi_mn7_cn）。
代价：失去 AEC · Speaking 中无法打断 · 阈值语义不同（MultiNet 概率 0.2 vs WakeNet 触发阈 0.6）· Feed 内联 detect 抢占 audio_input P10 时间片。

7.5 自我反驳清单（误判排除）

误判 1：vector::erase O(N²)？反驳：input_buffer_ 上限 chunksize × 2，单次 O(chunk)，total O(N)，可接受。
误判 2：event group 用 portMAX_DELAY 是红线？反驳：event group 主循环 idle wait 不耗 CPU · 红线针对业务同步等待。真正的红线是 fetch_with_delay(portMAX_DELAY)（P0-1）。
误判 3：encode 用 PSRAM 栈违反 esp32-memory.md？反驳：红线是 PSRAM 栈 + Core 0。当前是 PSRAM 栈 + Core 1 + INT TCB · 正是 04-28/29 修复后形态 · 正确。
误判 4：dynamic_cast 性能 P1？反驳：S3 上单次 5~10 μs，状态切换频率极低 · 归 P2。
误判 5：wake_word_pcm_ deque 无锁与 encode task race？反驳：Encode 由 application 在唤醒后调（:1012），那时 detection 已 Stop · 不再写。P0-1 修后顺带防住。
误判 6：Custom 取左通道没考虑 ref？反驳：P30 codec input_channels==2 是 mic+ref，Custom 无 AEC 只需 mic，取通道 0 正确。
误判 7：models_ deinit race？反驳：是问题（未误判），但严重性 P1，量产首发不触发 OTA 重加载。
误判 8：24 KB encode 栈够吗？反驳：Opus 编码器内部主要 PSRAM 工作，编码上下文小（< 8 KB 栈）· 粗估高水位 ~12 KB · 留 50% 余量合理。
