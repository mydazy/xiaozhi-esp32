评判口径 · 三维基线

本报告以三条基线综合判定每条修改的"必要 / 合理 / 过度"，并出具保守＞激进的量产前建议（项目 1 万台呆滞库存教训已沉淀为 CLAUDE.md 铁律）。

基线 A · 上游 10 万开发者验证基线

  • 78/xiaozhi-esp32 已被 10 万以上开发者使用，其原始 wifi_board、wifi_manager、blufi.cpp、websocket_protocol、protocol 抽象都属于"实战 hardened"。
  • 覆盖率：任何对上游"骨干模块"的整体重写 = 把 10 万人压测过的代码替换成无压测的代码，需要强商业理由（双频、CSI、SoftAP 配网等）。

基线 B · 项目铁律（CLAUDE.md / 量产纪律）

  • 不动状态机：12 态 DeviceState 不新增枚举。
  • 不抽基类：不抽 ActivitySession / SceneNavigator，第 4 个会话型出现前不动手。
  • 必登记三维取值：(A: DeviceState) × (B: ActivityType) × (C: SceneType)。
  • main/ 下 0 裸 xTaskCreate；PSRAM 栈不放 Core 0。

基线 C · 商业 SKU 必备

  • P30-4G 必需：ML307 模组、GNSS、双网切换。
  • P31 必需：NFC、iBeacon、耳机检测、加速计。
  • 三板共有：1.83″ 圆角矩形 284×240、AXS5106L 触摸、SC7A20H 加速计、AEC、Opus、按键交互。
  • 商业差异化：教育卡 / GIF 识字、闹钟、番茄钟、待办、Flow 脚本（KPI 拉动）。

判定标签

KEEP 商业必需或显著优于上游、应保留。

WATCH 合理但存在边际成本／心智负担／风险点，需观察。

REVISIT 疑似过度优化、自研重叠上游骨干、应严格回归或回退。

对应 wifi_board.cc 重写 +691/-230，新增 SwitchConfigMode() 配网模式热切换		 KEEP 评审复核（2026-05-19 修正）：本条曾被初判为 REVISIT，复核代码事实后改判 KEEP。
核心证据 — 上游做不到的能力：
① 上游 wifi_board.cc 用的是编译期 #ifdef 二选一（CONFIG_USE_HOTSPOT_WIFI_PROVISIONING #elif CONFIG_USE_ESP_BLUFI_WIFI_PROVISIONING），运行时无法切换。
② 上游 WifiManager 没有 SetMode(STA/APSTA) 运行时切换 API；上游 Blufi 没有 Controller Init 与 Start 解耦的设计——拆不下来、装不回去。
③ 本仓 SwitchConfigMode()（wifi_board.cc:208-303）实现 Blufi ↔ SoftAP 真正的运行时热切换，关键工程细节：
  • BLE 控制器初始化触发 flash read → 禁用 cache → LVGL 渲染访问 PSRAM 会阻塞 10+ 秒 → Task WDT；修复用 lvgl_port_lock(1000) 强制 LVGL 暂停渲染等锁，已注释"修复：lvgl_port_lock 强制 LVGL 暂停渲染，让它在等锁而不是在 PSRAM 阻塞"。
  • BLE ↔ WiFi APSTA 共存冲突回避：BLE 启动前强制 SetMode(STA) + 200ms 隔离窗口。
  • 切换失败自动降级回原模式（new_mode = old_mode + 重启原模式）。
  • NVS 持久化 wifi.blufi 字段，重启保留用户选择。
商业必备：用户在家中老人 / 残障 / 无智能手机蓝牙能力等场景，可在现场不重启切换"蓝牙配网 ↔ 热点配网"，是售后服务的关键能力。
仍要注意的工程点（保留观察）：
① MAX_SCAN_RETRY=4 / SCAN_WAIT_MS=5000 两个魔数是踩坑后的硬编码，建议提到 Settings 可远程下发；
② lvgl_port_lock 超时时仍继续 BLE 初始化（注释自标"有死锁风险"），需要监控期 Crash 日志验证是否触发；
③ 切换状态机 StartConfigMode → StopConfigMode → SwitchConfigMode 的并发安全（防用户连续点击）建议加 mutex / debounce。
main/blufi/blufi.cc/h/blufi_init.c/blufi_security.c：BLE 配网纯数据通道（与凭证验证解耦）
blufi/ibeacon.cc/h：iBeacon 扫描（仅 P31）		 KEEP 上游 blufi.cpp 已删除并迁入 main/blufi/，新加 validator 抽象把"凭证验证"从协议栈剥离，符合 SRP。iBeacon 仅 P31 编译，ifdef 收敛得很干净，属于 SKU 必备。
WiFi CSI 默认禁用（NVS csi.enabled=0），远程命令 {"type":"csi","enabled":true} 开启 KEEP "默认禁用 + 远程下发" 是教科书级的 feature gate，正确做法。CSI 在量产期当作"差异化卖点"按需开启，不影响主路径稳定性。

结论与建议

1. 大盘判定

本次差异整体合理性高、量产化纪律执行良好。三大铁律（不动状态机 / 不抽基类 / 必登记三维取值）实际落地基本到位 —— 12 态 DeviceState 未被新增，三维 enum 正确登记，ActivitySession / SceneNavigator 基类没有被冒进抽象。上游 rebase 友好度方面，新增模块（RemoteCmd、FlowEngine、AlarmManager、PomodoroManager、MusicPlayer）几乎都挂在新文件，对 Application 的"侵入"控制在 +504 行（多为 API 暴露）。

2. "过度优化"候选（按风险降序 · 已剔除原 main/wifi 误判）

评审修订（2026-05-19）：原报告把 main/wifi/ 完整重写 列为最高风险的过度优化候选，判定有误，已撤回。
复核事实：上游 wifi_board.cc 用编译期 #ifdef 二选一，不存在 Blufi ↔ SoftAP 运行时热切换的能力；WifiManager / Blufi 也未提供 Controller Init 与 Start 解耦的接口。本仓 SwitchConfigMode() 是商业必备能力（家中老人 / 残障 / 无蓝牙智能手机现场切换配网方式），属于不得不重写的骨干替换，已改判 KEEP（见 § ③）。

① REVISIT ControlCenter / UI Image Manager 已写未编

CMakeLists.txt 明确注释屏蔽 ui_image_manager.cc + control_center.cc，但 CLAUDE.md 和 docs/p30-architecture.md 中却写 "ControlCenter 已完整实现"。文档与代码漂移是量产期的隐性陷阱。
建议：① 立刻修订文档为"已实现，量产期未编译"；② 给"恢复编译"明确触发条件（如版本 2.3.x、灰度 5%、Crash 率 < 阈值）；③ 否则建议 git revert 删除该路径，待重启时按新需求重写（这是"代码先于需求"的典型，1 万台呆滞库存的教训正出于此）。

② WATCH OPUS 全局帧长动态切换 + AEC 后增益硬编码

百度协议要求 20ms 帧、其它要求 60ms 帧，引入 g_opus_frame_duration_ms 全局变量 + SetFrameDuration()（耗时 200-500ms，重启 AFE）。逻辑合理但切换路径未保护，可能在非 Idle 态被调用引发音频中断。
建议：① 切换收口到 protocol_factory，仅在建链前调用；② 强制 SetFrameDuration() 在 kIdle 态执行，违反则 abort；③ kNoiseGateRmsSq 改为 Settings 可调。

3. 三个 "看上去不必要、但实际正解" 的修改

  • 三维 enum + 查询 API：表面看像抽象提前，实际是 CLAUDE.md 铁律的精确落地，是替代散落 IsXxx flag 的成本最低办法。✅
  • FlowEngine 改名：保留 live_companion.start 命令字段做向后兼容，已发货设备零影响。✅
  • FreeRTOS task 双核重平衡：opus_codec 必须留 Core 0、audio_input + AEC Feed 必须 Core 1，已通过测试解决 Task WDT reboot。方向完全正确，是有踩坑实证的硬核工程修复。✅
