2 推送格式 · 三选一

设备端 RemoteCmd::Handle() 兼容三种包装（remote_cmd.cc:38-51）。云端正式服务推荐用方式 1，调试用方式 2，本地测试用方式 3。

2.1 方式 1 · 双层 string-wrapped（生产）

{"type":"custom","payload":{"message":"{\"type\":\"flow\",\"action\":\"start\",\"url\":\"https://xxx/test.json\"}"}}

2.2 方式 2 · object 直传（调试）

{"type":"custom","payload":{"message":{"type":"flow","action":"start","url":"https://xxx/test.json"}}}

2.3 方式 3 · 顶层格式（本地最简）

{"type":"flow","action":"start","url":"https://xxx/test.json"}

2.4 action 列表

action额外字段用途
starturl 或 script 二选一启动脚本 · URL 走 HTTP 拉取 · script 直传 JSON 对象
stop—停止 + 清 items_
restart—从头开始当前脚本
status—设备屏幕显示 + LOG 当前状态

3 测试脚本范例（4 例）

3.1 T1 短脚本 · 冒烟（10s 内验完）

{
  "name": "T1冒烟",
  "loop": false,
  "items": [
    {"type":"tts","text":"测试一"},
    {"type":"tts","text":"测试二","delay_ms":1000}
  ]
}

期望：屏幕 Alert "开始播放: 2 项" · 第一句完后等 1s · 第二句完后自动 Idle · 共 ~8s。

3.2 T2 长脚本 · F5 压测（30 项循环）

{
  "name": "T2压测",
  "loop": true,
  "resume_delay_s": 5,
  "items": [
    {"type":"tts","text":"第一项，请等待"},
    {"type":"ttai","text":"用一句话夸夸今天的小听众"},
    {"type":"tts","text":"第三项"}
    // ... 复制扩展到 30 项
  ]
}

期望：循环播放不停 · 1 小时后 free heap 稳定（INT > 60KB 红线）· loop_count_ 增长。

3.3 T3 WS 直推 script（不需 HTTP 服务器）

{"type":"flow","action":"start","script":{
  "name":"T3直推","loop":false,
  "items":[{"type":"tts","text":"WS 推送测试"}]
}}

期望：跳过 HTTP 拉取 · 直接 ParseScript → 播放。日志 flow start: inline script (N bytes)。

3.4 T4 远程控制集

{"type":"flow","action":"stop"}
{"type":"flow","action":"restart"}
{"type":"flow","action":"status"}

在 T2 运行中依次推送。期望：stop → "直播伴侣停止" Alert · restart → 从第 1 项重播 · status → 屏幕显示当前进度。

3.5 测试服务器（最简 Python · 拉脚本场景）

python3 -m http.server 8000 --directory /path/to/scripts/
# 然后推送 url 为 http://<your-ip>:8000/T1.json

6 推送工具 · 三选一

6.1 A · Python WS client（最简）

import asyncio, websockets, json

async def push(msg):
    async with websockets.connect("ws://<device_ip>:<port>") as ws:
        await ws.send(json.dumps(msg))
        print("sent:", msg)

# 单次推送
asyncio.run(push({
    "type":"flow","action":"start",
    "script":{"name":"T1","loop":False,
              "items":[{"type":"tts","text":"测试"}]}
}))

# 连发 5 次（F1 重入测试）
async def burst():
    for i in range(5):
        await push({"type":"flow","action":"start",
                    "url":f"http://<ip>:8000/T2.json?n={i}"})
        await asyncio.sleep(0.1)
asyncio.run(burst())

9 MCP 描述瘦身回归（v1.1 新增）

2026-05-13 完成 28 处 MCP 工具描述瘦身（中文口语化），tools/list payload 从 9418B → 预期 ~4KB。本章节为 MCP 瘦身专项回归。

9.1 起因回顾

修复前事故：设备启动时云端下发 tools/list 请求 → 设备 publish 9418B 响应 → MQTT broker 单包上限约 8KB → 直接 RST 连接 → 设备每 10s 重连重发死循环。

关键日志特征（修复前）： 
E transport_base: poll_write select error 104, errno = Connection reset by peer
W mqtt_client: Publish: Losing qos0 data when client not connected
E MQTT: Failed to publish message (9418 B): {"session_id":...
W Application: Alert [circle_xmark] 错误: 发送失败，请检查网络
# 每 10s 重复 · 设备永远连不上

9.2 T-M1 payload 大小回归

步操作期望
1烧录新固件 · 启动—
2等设备连上 MQTT（日志看到 "MQTT: Connected"）30s 内
3grep "tools/list: id=.*payload="找到至少一条
4读 payload=N B 数字N < 5000 · 修复前 9418 · 修复后预期 3700-4500
5读 paged=no不应分页（< 12000 max）
边界判定：
< 5KB：✅ 健康 · 任何 MQTT broker 都能过
5-8KB：⚠ 需测试本环境 broker · 大多数仍能过
> 8KB：❌ 多数 broker RST · 必须再瘦身
> 12KB：❌ 触发本地 cursor 分页 · 云端处理待验证

9.3 T-M2 MQTT 60s 稳定性测试

步操作期望
1启动后连续观察日志 60s—
2grep "Connection reset by peer"0 命中（修复前 6 次/min）
3grep "MQTT disconnected, schedule reconnect"≤ 1 次（启动早期可能 1 次）
4grep "Losing qos0 data"0 命中（mqtt_protocol.cc IsConnected 前置拦截）
5grep "Alert.*发送失败"0 命中（无错误 Alert 浪涌）
610 分钟后 free INT RAM> 60KB（CLAUDE.md 红线）
