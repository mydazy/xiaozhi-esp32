> 主力板：`main/boards/mydazy-p30-4g/mydazy_p30_board.cc`（P30-4G，下文简称 board）。P30-WiFi 板逻辑高度一致，差异在网络/配网部分单独说明。

## 生效配置前提（决定多个流程走哪条分支）

实测当前 `sdkconfig` 关键开关：

| 开关 | 状态 | 影响 |
|---|---|---|
| `CONFIG_USE_AFE_WAKE_WORD` | 开 | 默认词走 AFE wakenet（"你好小智"） |
| `CONFIG_USE_CUSTOM_WAKE_WORD` | 关 | 自定义词（multinet）按 NVS `wakeword.mode` 运行期选择，非编译期 |
| `CONFIG_SEND_WAKE_WORD_DATA` | **关** | 唤醒词音频**不上送服务器**；唤醒后本地播 `wakeup.ogg` |
| `CONFIG_WAKE_WORD_DETECTION_IN_LISTENING` | 关 | 聆听中不并行跑唤醒检测 |
| `CONFIG_USE_DEVICE_AEC` | 开 | 设备端 AEC（但 NVS `deviceAec` 默认 0 = 关）|
| `CONFIG_USE_SERVER_AEC` | 关 | 不用服务器 AEC |
| `CONFIG_EN_SC7A20H_WAKE`（config.h:18） | 1 | 编译期允许陀螺仪唤醒；运行期由 NVS `pickupWake` 控（默认 0=关）|

设备状态枚举（`main/device_state.h:4`）：
`Unknown / Starting / WifiConfiguring / Idle / Connecting / Listening / Speaking / Upgrading / Activating / AudioTesting / FatalError`。
状态转移由 `DeviceStateMachine::IsValidTransition` 强校验（`main/device_state_machine.cc:34`），非法转移被拒绝并打 WARN。

监听模式枚举（`main/protocols/protocol.h:39`）：`kListeningModeAutoStop / kListeningModeManualStop / kListeningModeRealtime`。
打断原因枚举（`protocol.h:34`）：`kAbortReasonNone / kAbortReasonWakeWordDetected`。

## 场景 1：语音唤醒

**流程**（默认词"你好小智"，当前配置 SEND_WAKE_WORD_DATA 关）：

```
用户说唤醒词
→ audio_input 任务读 10ms PCM → wake_word_->Feed()   [main/audio/audio_service.cc:299]
→ AFE fetch → wakeup_state==WAKENET_DETECTED → Stop + 回调   [main/audio/wake_words/afe_wake_word.cc:152]
→ AudioService 汇聚 → callbacks_.on_wake_word_detected   [main/audio/audio_service.cc:882]
→ Application 置事件位 MAIN_EVENT_WAKE_WORD_DETECTED   [main/application.cc:196]
→ 主循环分发 HandleWakeWordDetectedEvent()   [main/application.cc:1072]
→ 先停音乐(若在放) MusicPlayer::Stop + 退 Player UI   [main/application.cc:1078]
→ Idle 分支：#if SEND_WAKE_WORD_DATA→EncodeWakeWord  #else→PlaySound(OGG_WAKEUP)   [main/application.cc:1091]
   ↑ 当前走 #else：本地播 wakeup.ogg
→ skip_next_stt_popup_=true（防唤醒词被 ASR 识别后又响 popup）   [main/application.cc:1099]
→ 通道未开 → 状态切 Connecting → Schedule(ContinueWakeWordInvoke)   [main/application.cc:1101]
→ ContinueWakeWordInvoke → OpenAudioChannel   [main/application.cc:1133]
→ #else 分支：pending_listening_sound_=OGG_POPUP + SetListeningMode(默认)   [main/application.cc:1161]
→ 状态切 Listening   [见场景2]
```

**自定义词（"小精灵"）走另一条链**：multinet 命令词，检测在 `Feed` 内 `multinet_->detect()==ESP_MN_STATE_DETECTED` 且命令 `action=="wake"`（`main/audio/wake_words/custom_wake_word.cc:171`、`:179`）。默认词/自定义词二选一，由 `audio_service.cc:864` 按 NVS `wakeword.mode` + 模型存在性决定，不并存。

**关键点**：
- **唤醒词不上送服务器**（当前配置）：受 `CONFIG_SEND_WAKE_WORD_DATA` 控制，实测关闭。开启时才 `EncodeWakeWord`（`application.cc:1092`）→ 循环 `SendAudio` 上送 opus（`application.cc:1153`）+ 发 `type:listen,state:detect` JSON（`protocols/protocol.cc:51`）。当前关闭，只在 Listening 状态发 `SendStartListening`（`application.cc:1214`）。
- **本地提示音 wakeup.ogg**：唤醒词唤醒走 `OGG_WAKEUP`（`application.cc:1094`，Idle 即时分支）。触摸/按键唤醒是另一路，board 直接 `PlaySound(OGG_WAKEUP)`（`mydazy_p30_board.cc:416`、`:713`）。资源文件 `main/assets/locales/zh-CN/wakeup.ogg`。
- **首句 TTS / UDP 打洞**：仅 MQTT 协议在 `OpenAudioChannel` 拿到 UDP 通道后立即发 payload_len=0 空包打洞（`protocols/mqtt_protocol.cc:302`，日志 "UDP hole punch sent"）。这是协议层固定行为，**不依赖 SEND_WAKE_WORD_DATA**。WebSocket 协议无此逻辑。

**实现位置**：`main/application.cc:1072`（入口 `HandleWakeWordDetectedEvent`）、`main/audio/audio_service.cc:299/864/882`、`main/audio/wake_words/afe_wake_word.cc:152`、`main/audio/wake_words/custom_wake_word.cc:171`、`main/protocols/mqtt_protocol.cc:302`。

## 场景 2：语音对话全流程（聆听→识别→TTS 播放）

**流程**：

```
进入 Listening   [HandleStateChangedEvent(kDeviceStateListening) main/application.cc:1201]
→ AutoStop 模式先等播放队列清空 → SendStartListening(mode) + EnableVoiceProcessing(true)   [main/application.cc:1209]
→ AFE 处理麦克风音频 → 编码 → 上送服务器
   AfeAudioProcessor OnOutput → 编码队列   [main/audio/audio_service.cc:98]
   OpusCodecTask 编码 → 主循环 MAIN_EVENT_SEND_AUDIO → protocol_->SendAudio   [main/application.cc:375]
→ 本地 VAD 仅驱动 LED，不本地停听（停听由服务器判定）   [main/application.cc:394]

识别（服务器侧 ASR）
→ 服务器回 stt JSON → 显示用户文本 + 条件播 OGG_POPUP   [main/application.cc:815]

TTS 播放
→ 服务器回 tts start → 状态切 Speaking (aborted_=false)   [main/application.cc:783]
→ TTS opus 帧 → OnIncomingAudio → 仅 Speaking 时入解码队列   [main/application.cc:690]
→ OpusCodecTask 解码（失败走 PLC 兜底）→ 播放队列 → codec 输出   [main/audio/audio_service.cc:376 / :316]
→ 服务器回 tts stop → manual 模式→Idle，其它→回 Listening   [main/application.cc:789]
```

**关键点**：
- **监听模式默认值**：`GetDefaultListeningMode()` = AEC 关→AutoStop，AEC 开→Realtime（`application.cc:1358`）。本板 `deviceAec` NVS 默认 0 → 实际默认 **AutoStop**。模式名（auto/manual/realtime）随 `SendStartListening` 发给服务器（`protocols/protocol.cc:57`）。
- **AutoStop 是服务器判停**，不是本地 VAD。本地 `on_vad_change` 只更新 LED（`application.cc:394`）。设备持续上送，服务器判定后下发 `tts start` 完成 Listening→Speaking。
- **TTS 解码兜底**：`OnIncomingAudio` 只在 Speaking 接收（`application.cc:691`），避免 Idle/打断后残帧入队；解码失败自动 PLC 补偿（`audio_service.cc:404`），队列欠载有静音淡出兜底。opus 帧长按协议切换（百度 20ms，其它 60ms，`application.cc:875`）。
- **状态广播**：每次状态变化经 `MAIN_EVENT_STATE_CHANGED` 并用 esp_event 广播给 FlowEngine 等订阅方（`application.cc:208` → `device_state_event.cc:25`）。

**实现位置**：`main/application.cc:1201`（状态切换处理）、`main/device_state.h`、`main/device_state_machine.cc`、`main/audio/audio_service.cc:98/316/376`。

## 场景 3：打断

**流程**（四种打断方式）：

```
① 语音打断（Speaking/Listening 时再次说唤醒词）
   HandleWakeWordDetectedEvent → Speaking/Listening 分支   [main/application.cc:1112]
   → AbortSpeaking(kAbortReasonWakeWordDetected) + 清空 send queue   [main/application.cc:1113]
   → 重进 Listening 并播 OGG_WAKEUP   [main/application.cc:1123]

② 触摸单击打断 TTS（不退出对话）
   OnTouchClick → HandleTouchSingleClick   [main/boards/mydazy-p30-4g/mydazy_p30_board.cc:358 / :402]
   → Speaking 态：AbortSpeaking(kAbortReasonNone)，无提示音   [main/boards/mydazy-p30-4g/mydazy_p30_board.cc:418]

③ 触摸双击退出对话
   OnTouchDoubleClick → HandleTouchDoubleClick   [main/boards/mydazy-p30-4g/mydazy_p30_board.cc:371 / :425]
   → Listening/Speaking：AbortAnyConversation + 回 Idle + 播 OGG_EXITCHAT   [main/boards/mydazy-p30-4g/mydazy_p30_board.cc:435]

④ 按键打断（BOOT 单击）
   boot_button OnClick → Speaking:ToggleChatState(打断) / Listening:播 OGG_EXITCHAT+退出   [main/boards/mydazy-p30-4g/mydazy_p30_board.cc:716]

AbortSpeaking 实质：aborted_=true; tts_streaming_=false; SendAbortSpeaking(reason)   [main/application.cc:1266]
   → 发 JSON {type:abort[,reason:wake_word_detected]}   [main/protocols/protocol.cc:42]
```

**关键点**：
- `AbortSpeaking` 本身不立即清解码队列，但状态离开 Speaking 后 `OnIncomingAudio` 不再接收 TTS 帧（`application.cc:691`），真正清空在新状态入口的 `ResetDecoder`（`application.cc:1243`）。
- **打断提示音**：唤醒词打断→`OGG_WAKEUP`（`application.cc:1120/1124`）；触摸单击打断 TTS→**无提示音**（`mydazy_p30_board.cc:418`）；双击退出 / 按键退出 Listening→`OGG_EXITCHAT`（`mydazy_p30_board.cc:439`、`:717`）。
- 只有唤醒词打断带 `reason` 上报服务器，其余用 `kAbortReasonNone`。

**实现位置**：`main/application.cc:1112/1266`、`main/boards/mydazy-p30-4g/mydazy_p30_board.cc:402/425/716`、`main/protocols/protocol.cc:42`。

## 场景 4：触摸手势交互

> 触摸驱动 `components/esp_lcd_touch_axs5106l`。**手势全部在固件软件层算**，不读芯片硬件手势寄存器：驱动只取手指数+X/Y 坐标（`axs5106l_touch.c:699`），方向/单双击/长按由 `recognize_gesture()` 用坐标差+时间算（`axs5106l_touch.c:783`）。

## 场景 7：省电 / 休眠 / 唤醒

> 省电定时器 `PowerSaveTimer`（`main/boards/common/power_save_timer.cc`），每秒 tick。板级实例 `new PowerSaveTimer(120, 60, 300)`（`mydazy_p30_board.cc:489`）= cpu_max_freq 120 / 进省电 60s / 关机 300s。

**休眠分级时序**：
```
无操作 0s ──60s──▶ 一级 OnEnterSleepMode：背光降到 15 + 上报状态   [mydazy_p30_board.cc:490]
          ──300s─▶ 二级 OnShutdownRequest   [mydazy_p30_board.cc:500]
                    充电中? → 跳过深睡，保持降亮
                    deepSleep 开? → ShutdownOrSleep("休眠中","拿起唤醒",gyro=true) → EnterDeepSleep
任意操作（按键/触摸/摇晃/音量）→ WakeUp() → 计时归零 + 恢复背光   [mydazy_p30_board.cc:496]
对话/播放中：CanEnterSleepMode 为假 → 计时不累加   [power_save_timer.cc:64]
```
> 注：cpu_max_freq=120 恰好不满足降频分支的 `<120` 条件（`power_save_timer.cc:78`），所以**一级实际只息屏降亮、不降频不关音频**。若产品预期"60s 后降功耗"，目前只省了背光。

**深睡进入流程**（`EnterDeepSleep` `mydazy_p30_board.cc:573`）：
```

→ AudioService.Stop → 关背光 → ResetProtocol 优雅断 MQTT/WS
```

**三个唤醒源配置位置**（三源并存）：

| 唤醒源 | 配置位置 | 默认状态 |
|---|---|---|
| EXT0 开机键(GPIO0) | `esp_sleep_enable_ext0_wakeup(BOOT, 0)` `mydazy_p30_board.cc:602` | 常开 |

| TIMER 闹钟定时 | `AlarmManager::ConfigureTimerWakeup` `main/alarm_manager.cc:323`（分段策略：≤30min 不睡 / 30min~4h 提前 30min / >4h 分段 2h）| 有闹钟时配 |

**唤醒后开机分流**（`HandleWakeupCause` `mydazy_p30_board.cc:192`，按 `esp_sleep_get_wakeup_cause()`）：

| wakeup_cause | 动作 | 代码位置 |
|---|---|---|
| EXT0 开机键 | 防误醒（距 sleep<500ms 未松手→回睡）；要求按住≥1500ms 才真开机，否则立即回深睡 | `:198`、`:239` |

**摇一摇 / 拿起**：
- 拿起唤醒（EXT1）：见上，默认关（与 memory `gyro-sleep-wake-2026-06` 的扰民问题对应，默认关是规避措施）。
- 摇一摇检测：`sc7a20h_shake(阈值1500mg, 窗1000ms, 4帧, 冷却1500ms)`（`mydazy_p30_board.cc:300`），含死值兜底（连续 20 帧不变判数据卡死，对应 memory `sc7a20h-boot-deadvalue`）。
