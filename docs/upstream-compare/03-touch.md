## 一、内存安全审查（红线）✅ 合格

`read_touch`（681-769）读 I2C 触摸帧，逐项核对：

| 检查点 | 实现 | 结论 |
|--------|------|------|
| 缓冲边界 | `uint8_t buf[6]` 固定 + `read_register(...,6)` 读固定 6 字节 | ✅ 无越界 |
| 点数校验 | `n = AXS_POINT_NUM(buf); if (n==0\|\|n>1) return false` | ✅ 拒绝噪声值 |
| 坐标提取 | `AXS_POINT_X/Y` 读 buf[2..5]，均在 6 字节内 | ✅ 安全 |
| 坐标钳位 | `sx>=width → width-1`；`sy>=height → height-1` | ✅ 无越界写 |
| 无效帧拦截 | 0xFFF / >MAX+50 / buf[3]==0xFF&&buf[5]==0xFF 三重 | ✅ |

**无裸 memcpy、无越界索引、无外部长度信任问题。** 触摸内存安全达标。

## 二、4G-RF 补偿层判定（专题核心）

> 按任务原则：维护成本 vs 实测缓解收益。**有物理依据且有效的补偿 = 🟢保留（脏帧守卫红线）**；**治标无效堆复杂度 = 🔴**。

### 🟢 有效补偿（红线保留，勿删）

| 机制 | 原理（物理依据）| 判定 |
|------|----------------|------|
| **storm 检测器**（`storm_detected` 397-455）| 4G TDD 突发耦合 INT 线 → 1s 窗 INT 边沿暴增(>threshold) → 静默 mute。IRAM ISR 仅计数；触摸活跃期跳过避免误判；HOT 模式应对簇发；storm 触发清 latched press + gesture + last_click 防虚假双击(P0-3.1) | 🟢 可观测信号驱动的脏帧守卫，因果清晰 |
| **INT debounce**（int_low_since + 三次复读）| 单帧伪边沿去抖 | 🟢 |
| **速度门**（max_speed_px_s）| 跳变速度超真实手指上限 → 判噪声丢弃 | 🟢 |
| **2 帧确认门**（CLICK_MIN_FRAMES=2 + SINGLE_FRAME_MAX_TIME 200ms）| 事件型芯片单帧 latch，RF 拉长的单帧噪声被拦 | 🟢 **F1 问题的真修**（memory 实证）|
| **swallow 首触吞噬**（省电唤醒首帧）| 唤醒首帧坐标易被 RF 抖动污染 → 吞掉只点亮屏 | 🟢 防误触 |
| **swipe_traj_ratio 轨迹直线度**| 真滑动轨迹直 / RF 抖动轨迹乱，按 manhattan/直线比区分 | 🟢 |
| **RF 双档 NORMAL/STRICT**（apply_rf_mode）| 4G 板传 STRICT 收紧、WiFi 板 NORMAL 宽容 | 🟢 机制本身必要 |

**结论**：4G-RF 补偿**主体是有效的、有物理依据的脏帧守卫**，对应 memory "固件只能缓解"的有效缓解手段。**全部默认保留，不动。**

## 三、🔴 过度优化候选

### 🔴-03-A · 中值滤波死代码（#if 0 残留）

- **位置**：`axs5106l_touch.c` `#define AXS_MEDIAN_FILTER 0`（671）+ `median3_u16`（674-680, `#if` 包裹）+ read_touch 内 `#if AXS_MEDIAN_FILTER` 块（748-762）+ `touch_state_t.med_x1/med_y1` 字段。
- **我们怎么改的**：曾加 3 点中值滤波压离散离群点，**实测无效已用宏关闭**（=0），但源码全留着。
- **官方原实现**：无（自研芯片）。
- **维护成本/风险**：`#if 0` 预处理器排除，**不占运行时/代码空间**，仅源码噪音 + 结构体多 2 个空字段。属"治标无效的尝试残留"。**不碰红线**。
- **处置建议**：清理整套（宏 + median3_u16 + #if 块 + med_x1/med_y1 字段）。未提交改动已在删其注释但未删干净——建议一并清掉。

### 🔴-03-B · RF 参数进入"反复微调收敛不掉"状态（流程警告，非删除项）

- **位置**：`axs5106l_touch.c` 60-76 行 RF_S_*/RF_N_* 常量，注释带大量 `量产 v2.2.16: 4→6ms / 200→300 / 2500→2000 / 12→25 / 18→14` 演进痕迹。
- **性质**：机制（storm/debounce/速度门）必要保留，但**参数已多轮微调**，正是 memory `touch-jitter-4g-rf` 明确警告的"反复调 RF 参数收敛不掉"。
- **维护成本/风险**：每轮微调耗工但边际收益递减（根因是硬件 RF 耦合，参数只能缓解不能消除）。这不是"删代码"，是**流程纪律**。
- **处置建议**：**冻结当前参数，停止继续微调**。除非有新的实测数据支撑，否则不再投入 RF 参数调校（memory 已定性收益不明）。
