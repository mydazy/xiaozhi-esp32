## ⚠️ 唯一仍存在的真问题：P1-3 remote_cmd 零鉴权

- **现状**：新 main `RemoteCmd::Handle`（`remote_cmd.cc:20`）只校验 JSON 格式，**无 token/签名/设备身份**。命令已精简为 `music_play/music_stop/wakeup/send_text/reboot/ota` 6 个——**最危险的 update_prompt/wakeword 已移除**，但 `reboot`/`ota` 仍是零鉴权危险命令。
- **风险评级**：比旧分支**显著降低**（命令面收窄），但劫持对话连接仍可远程 reboot/触发 ota。非"必崩 P0"，属安全加固。
- **有效修复必须分两层**（详见 `P1-FIX-PLAN.md` P1-3）：
  - **设备端阶段1**（本仓可做）：`ota` 命令限流（防反复 CheckVersion DoS）+ Handle 输入加固。但**挡不住 reboot 零鉴权核心威胁**。
  - **阶段2**（**必须前后台联调**）：危险命令子集 HMAC 签名 + nonce 防重放，需后端按设备密钥分发+签名下发。**这是有效防护的关键，超出本仓/设备端独立可完成的范围。**

**canonical main 是干净的重构基线，旧分支的 P1/安全/死码债绝大多数未带过来**（多是旧分支自研结构引入的、重构时自然甩掉）。

- **已落地**：评审成果文档（扩展功能清单 / 交互流程 / 10 模块评审 / RED 清单 / P1 方案）已带入 `docs/upstream-compare/`，作为新 main 的知识资产。
- **唯一待落地**：P1-3 remote_cmd 零鉴权——设备端阶段1 加固为辅，**核心需后端配合 HMAC 签名联调**（用户已确认"需前后台联调"），排后端协作专项。
