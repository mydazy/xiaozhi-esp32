§ 1 ESP32-S3 优秀设计原则（10 条）

规划基于业内通用 ESP32-S3 best practices，是后续推算的"判据"。

#原则当前项目违反点
1双核分工严格：Core 0 网络/控制 · Core 1 音频实时/UI当前 Core 0 挤 11 任务（含 audio_input P8）· Core 1 利用率 < 30%
2实时任务必须 Pin Core，禁止漂移13 个任务用 xTaskCreate（无 affinity）：audio_output / audio_detection / wake_word_encode / ml307_net / activation / LedEvent 等
3PSRAM 栈不挂 Core 0（cache 共享 SPI · NVS/OTA flash op 时禁用 cache，PSRAM 栈任务被调度即崩 SP=0x60100000）红线 3 项：custom_wake_word / stt_post / mp3_dl
4实时音频用 INT 栈 + DMA 内部 RAM 缓冲（PSRAM 无 DMA 直读）audio_input/output/codec 已用 INT ✅
5临时任务用静态栈（xTaskCreateStatic）减少碎片仅 wake_word_encode_afe 静态 · 其他临时任务（activation/flow_load/stt_post/mp3_dl）都动态
6优先级分层：P12 实时硬中断响应 / P10 实时数据流 / P8 编解码 / P5 UI / P3 后台 / P1 维护当前优先级散乱（main_app P10 偏高 · audio_input P8 偏低 · flow_load P1 偏低）
7Task Watchdog 启用持久任务周期 reset · IWDG 中断超时检测未启用 / 不可知（待 grep 确认）
8事件驱动避免 portMAX_DELAY 裸用已合规 ✅
9资源池化：connection / buffer pool 复用Protocol 实例每次重建 · cJSON 反复 alloc
10WiFi/4G 差异化资源：WiFi 模式不要保留 4G 任务栈当前 ml307_net 在 WiFi 板被编译进但永不启用（死代码污染）

§ 2 三遍评审推演（基于 ESP32-S3 原则）

第一遍 · ESP32-S3 硬件特性视角

ESP32-S3 双核 240MHz · 16MB Flash + 8MB PSRAM（octal SPI 80MHz）· 320KB INT RAM · WiFi+BLE 共射频。关键观察：

发现严重度建议
Core 0 严重过载（11 应用层 + WiFi/lwip/MQTT IDF 任务）🔴 高把 audio_input/audio_output/audio_detection/mp3_dec/wake_word_encode 全挪 Core 1
Core 1 利用率 < 30%（仅 audio_communication + LVGL + mp3_out）🟡 中承接 Core 0 移过来的音频任务
opus_codec 单任务 24KB INT，最大单点占用🟡 中拆成 audio_opus_enc 8KB + audio_opus_dec 8KB 双任务（节省 8KB INT）
13 任务无 affinity 漂移🟡 中全部 Pin Core，禁漂移
PSRAM cache 共享 SPI 风险（NVS/OTA 触发）🔴 高红线 3 项 PSRAM → INT，Core 0 严禁 PSRAM 栈

§ 5 长期稳定运行设计（5 类 Watchdog）

连续运行 7 天 / 30 天 / 90 天的稳定性靠 5 类机制保证。

#机制配置触发动作
1Task WatchdogCONFIG_ESP_TASK_WDT_EN=y · timeout 5s · 持久任务 esp_task_wdt_add()任务卡死 5s → panic + 自动重启
2Interrupt WatchdogCONFIG_ESP_INT_WDT=y · timeout 300ms中断/调度卡死 300ms → panic
3Stack Overflow 检测CONFIG_FREERTOS_CHECK_STACKOVERFLOW=2（canary 模式）栈越界 → panic + 上报任务名
4Brownout 检测CONFIG_ESP_BROWNOUT_DET_LVL=7（2.6V 阈值）电压跌落 → 安全重启
5Heap Integrity Watchdog已加 MonitorMemoryHealth + 加 heap_caps_check_integrity_all(MALLOC_CAP_INTERNAL, true) 每小时一次堆损坏 → 上报 + 软重启

额外稳定性措施

WiFi/4G 自愈30s 心跳超时 → 主动 reset 网络栈 · 5 次失败 → reboot
PSRAM 完整性启动期 PSRAM 测试（写读模式）· 失败 → FatalError
任务 CPU 统计CONFIG_FREERTOS_USE_TRACE_FACILITY=y · 每小时 dump uxTaskGetSystemState · 异常占用任务上报
每周软重启连续运行 7 天主动 esp_restart（量产数据决定阈值）
OTA 回滚已实现 esp_ota_mark_app_invalid_rollback ✅
