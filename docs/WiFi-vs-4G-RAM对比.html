一句话结论

4G 模式 ESP32 内部 RAM 占用大约只有 WiFi 模式的一半。原因是 4G 走 ML307 模组（AT 指令），把 TCP/IP 协议栈和 TLS 加解密整体卸载到了模组芯片自己的内存里——ESP32 这边只剩一个 UART 串口缓冲和两个解析任务。WiFi 模式则要在 ESP32 上自己跑 WiFi 驱动 + LWIP 协议栈 + 片上 TLS，是吃内部 RAM 的大头。

对产品的含义：WiFi 版的内部 RAM 比 4G 版紧张，尤其是配网那一刻（蓝牙 + WiFi 同时开），实测可用内部 RAM 会跌到 ~57KB、踩穿 60KB 红线——这也是代码里"配网成功后必须重启一次"的根本原因。4G 版没有这个配网峰值问题。

组件内部 RAM来源说明
BT 静态 .bss/.data（控制器固件 + 协议栈）30–50 KB代码佐证wifi_board.cc:421 注释明写；配网完调 Blufi::ReleaseStaticMem() 释放

叠加效果：BLE + WiFi + 提示音并存可用内部 RAM 跌到 ~57 KB代码佐证wifi_board.cc:464-469 注释明写，已踩穿 60KB 红线

现状缓解措施（代码已实现）：① BLUFI 模式下 WiFi 延迟到手机连上 BLE 才初始化，错开峰值；② 配网成功后主动重启一次，让 BLE 静态内存彻底释放、RAM 干净后再走 OTA/MQTT；③ 没进配网就直连成功时，立即 ReleaseStaticMem() 释放 30–50KB。结论：配网峰值已被"延迟初始化 + 配网后重启"两招压住，但它仍是 WiFi 版唯一逼近红线的场景，量产前应作为重点压测项。
