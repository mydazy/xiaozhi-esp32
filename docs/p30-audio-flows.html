1 概览

音频子系统 = 3 个核心任务（audio_input · audio_output · opus_codec）+ AFE 处理器（audio_communication）+ 一次性校准 calib · 5 队列 deque（统一在 audio_queue_mutex_ 下）· DMA 缓冲走 INT RAM · opus_codec 走 PSRAM 栈例外（24KB · Core 0）

1.1 设计亮点（保留）

  • PSRAM 栈策略分配正确 · opus_codec 24KB · Core 0 · 不触 flash op（与 wake_encode 红线不冲突）
  • 背压 + 200ms 超时丢帧（audio_service.cc:524）· 实时音频丢一帧远好于 AFE pipeline 雪崩
  • AEC 后软件增益 + 噪声门（audio_service.cc:106-122）· 预算 aec_gain_linear_ 避免每帧 powf · sum-of-squares 避免 sqrt · int32 限幅
  • 三档 MIC 自动校准（box_audio_codec.cc:286-342）· 1kHz tone 两轮闭环 · 量化 3dB 步进 · 跳过 33dB 驱动 bug
  • ES7111 软件回采 AEC（es7111_audio_codec.cc:255-296）· PSRAM 环形 200ms 缓冲 · AFE "MR" 格式启用 AEC · 0 改动适配
  • SetFrameDuration 帧长热切换 · encoder_mutex_ 串行重建 + AFE 重启 · 显式注释"仅 audio 空闲态调用"
  • PlaySound boot 期防御（audio_service.cc:687）· iot_button 先于 Initialize 不空指针
  • AlarmRinger 渐入 + 5min 兜底 · 60/80/100% 三段曲线 · 自动 EnableAutoSleep(false) 抑制深睡
  • OggDemuxer 状态机解析 + 8KB 固定缓冲 · 避免动态分配 · 跨页 Opus 包正确

6 Power Timer + AEC 后增益 + MIC 三档校准

6.1 AEC 后软件增益 + 噪声门

位置 audio_service.cc:106-122 · 每个 AFE 输出 60ms 帧（960 sample）做一次。

  • g = codec_->aec_gain_linear()（预算的 powf(10, aec_gain_db_/20)，缺省 9 dB → 2.82）
  • avg_in = sum_in / N（int64 sum-of-squares，避开 sqrt · 与门限 kNoiseGateRmsSq=20000 比 · ≈ RMS 141 ≈ -47 dBFS）
  • 条件：g > 1.01f && avg_in >= kNoiseGateRmsSq 才放大（低于门限不放大避免抬底噪）
  • 限幅：int32_t v = s * g · clamp -32768..32767
  • P1-10 · 即使不放大，sum_in 仍每帧 O(N) 计算 · 可抽样优化

8 三遍评审结果

三轮法：① 通读 review.md + 对照代码 ② 补全细节（任务优先级 / 上限 / mutex 超时 / PSRAM 宏） ③ 量产分级（P0 必修 · P1 后修 · P2 lean）。共 30+ 项 · 与代码现状对齐 · 不一致点已标注 ⚠。

8.1 与 review.md 的"漂移点"（重点）

⚠ 漂移 1 · P0-3 NoAudioCodec::Read 已部分修：review.md 说 NoAudioCodec::Read 在 :371 portMAX_DELAY。代码现状：NoAudioCodec::Read（:240-255）已用 pdMS_TO_TICKS(200)（kReadTimeoutTicks，:242）；:371 的是 NoAudioCodecSimplexPdm::Read（独立子类 override），仍 portMAX_DELAY。
真正未修：NoAudioCodec::Write :236（portMAX_DELAY） + NoAudioCodecSimplexPdm::Read :371（portMAX_DELAY）。P0-3 范围缩小。
⚠ 漂移 2 · 6.8 AlarmRinger::Start Schedule 嵌套已采用：review.md "误判，安全"的结论正确，但代码已显式 app.Schedule([text = BuildWakeWord(...)]() { app.WakeWordInvoke(text); })（alarm_ringer.cc:90-92）——比 review.md 描述的形式更稳。代码已比 review.md 更进一步。
⚠ 漂移 3 · AFE Feed 已加 input_buffer_ 上限 16000 红线：review.md 未提及 · 代码 2026-05-13 已修（afe_audio_processor.cc:114-127）：chunk_size=0 时清空 + input_buffer_ > 16000 sample 强 reset · 防 AFE 初始化竞态导致 50 万元素 Cache error。代码已比 review.md 更进一步。
⚠ 漂移 4 · AFE Feed 已加 TOCTOU 检查：afe_audio_processor.cc:103-107 在 input_buffer_mutex_ 内做 IsRunning() 检查防 Stop 竞态。review.md 未提。代码已比 review.md 更进一步。
⚠ 漂移 5 · 队列上限注释偏移：review.md 表格中 testing_queue 上限 = 167，实际 = 10000/60 = 166.67（取 int = 166）· review.md 算成 167。微小漂移 · 已修正本视觉版表格。
⚠ 漂移 6 · ResetDecoder 路径已加 audio_queue_mutex_ 包裹：review.md § 6.5 已确认安全 · 代码现状（audio_service.cc:725-737）持续保持。

8.6 自我反驳清单（第一轮误判项 · 重点关注）

评审本身的信心校准：第一轮通读时怀疑、第二轮验证后撤销的 8 项，列在此防止维护者重复误判。
  1. OpusCodecTask PSRAM 栈 + Core 0 = 红线违规？
    否。红线针对"flash op + cache 共享 SPI"，opus 编解码不触 flash op；24KB INT 栈会爆 60KB 红线，必须 PSRAM；Core 0 让 Core 1 留给 AFE。设计正确，保留。
  2. audio_input Core 1 / audio_output Core 0 跨核会 cache miss？
    否。ESP32-S3 L1 cache cluster 共享，跨核 deque 仅 mutex μs 级开销。设计正确，保留。
  3. AudioInputTask AUDIO_TESTING + WAKE_WORD 互冲？
    否。AS_EVENT_* 4 bit 业务互斥（testing 仅配网 / wake_word 仅 idle），EnableWakeWordDetection / EnableVoiceProcessing 都做 input_resampler reset 清缓存。设计合理。
  4. SetFrameDuration g_opus_frame_duration_ms 无锁更新是数据竞争？
    不严格。① 注释"仅 audio 空闲态调用" ② int 写入 ESP32-S3 原子 ③ 过渡期最多丢一两帧。保留 P2-1 作为优化项，非 P0 数据竞争。
  5. ResetDecoder decoder_lock.unlock() 后清队列是 TOCTOU？
    否。audio_queue_mutex_ 全程持有（函数末释放），OpusCodecTask 推 playback_queue 也要拿同锁 → 顺序串行化。安全。
  6. PopWakeWordPacket 没加锁访问 wake_word_ 内部？
    否。wake_word_->GetWakeWordOpus 内部有 wake_word_mutex_（自带保护）。安全。
  7. audio_queue_cv_ 共用于 4 个队列会假唤醒？
    否。std::condition_variable::wait_for 配合 predicate 会重判，假唤醒只多消耗几次循环 · 逻辑正确。符合 std 习惯。
  8. AlarmRinger::Start 死锁？
    否。app.Schedule 是 push 到主循环消费，Start 可在任何线程，不嵌套。安全。
