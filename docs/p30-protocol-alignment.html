P30 四协议接口对齐 · 深度分析

小智 WebSocket / MQTT 是系统默认协议，定义了 Protocol 基类的 12 个虚函数语义。 百度 BRTC、京东 JoyInside 是扩展平台，需要把各自的平台原生协议（前缀字符串 / 嵌套 EVENT 结构）翻译成基类语义，让 application 层零感知调用。

4 协议 × 12 接口 2 套基准（小智 WS · 小智 MQTT/UDP） 2 套扩展（百度 BRTC · 京东 JoyInside） 6 处对齐缺口热点 3 档对齐路线（必修 / 应修 / 选修）

§ 一 接口分层架构

application 层只看到 Protocol 基类。三层调用：上行接口（application → protocol）+ 下行事件（protocol → application）+ 平台原生（protocol ↔ 服务端）

小智 WS（基准） 小智 MQTT（基准 UDP 变体） 百度 BRTC（扩展） 京东 JoyInside（扩展）

核心原则

  • application 层零感知：所有调用走 protocol_->SendXxx()，所有回调走 OnIncomingJson(root) + OnIncomingAudio(packet) 两条 channel。
  • 基类语义就是小智原生 JSON：{"type":"tts"/"stt"/"mcp"/"hello",...} 是基类约定的统一格式。扩展协议必须把自家平台事件 翻译 成这套 JSON 再 emit。
  • 平台差异完全留在协议内：鉴权流程、心跳、超时、命令名、字段名一律在协议内吸收。application 不应包含任何 if (protocol == baidu/joyai) 分支。
当前对齐评分：调用方向（上行）小智/百度/京东都已对齐基类签名 100%； 回调方向（下行）小智天然透传、百度完整翻译 OK、京东本次刚补齐 OK 但仍有 2 处缺口。

§ 三 上行调用 · 泳道图（以 SendAbortSpeaking 为例）

application 一句 protocol_->SendAbortSpeaking(reason)，在四协议内部展开成完全不同的命令序列。

观察：同一个 application 调用，4 协议内部展开规模从 1 步（小智）到 5 步（百度）不等。 application 层始终零感知 — 这就是基类抽象的价值。 扩展协议（百度/京东）的"重写"动作越复杂，平台特性越强；但同时也意味着 application 调用方依赖更多被翻译的语义在协议内部正确实现。

4.2 京东 EMPTY_CONTENT 翻译流程图

为什么 EMPTY_CONTENT 是问题点：服务端每 3-5s 持续推送，但语义是"没识别到任何内容"。当前实现盲目把它当作"TTS 结束"翻译。

不暴露给 application，让基类 IsTimeout() 120s 兜底自然触发会话回收。

§ 六 对齐缺口热点

站在"统一架构、零感知调用"立场，当前实现还有 6 处必须补齐的缺口，按风险级排列。

# 缺口 影响协议 现状 建议 优先级
1 京东 listening 不退出 京东 JoyAI EMPTY_CONTENT 每 3-5s 刷 last_incoming_time_，基类 120s 兜底永远不触发，listening 卡死 HandleEventMessage 内 EMPTY_CONTENT 分支不更新时间戳，且不 emit tts.stop（无 TTS 上下文） P0 必修
2 application 缺统一 listening 超时 小智 WS · MQTT · 京东 仅百度自己有 30s xTimer，其他三协议靠基类 120s 但只在被动调用时检查 application.cc 进 kDeviceStateListening 启 30s esp_timer，state 切走停；超时 SetDeviceState(Idle) P0 必修
3 京东 SendAbortSpeaking 不 ResetDecoder 京东 JoyAI 打断后解码队列残留 1-2 秒，用户感觉"按了打断还在说" 对齐百度：Application::GetInstance().GetAudioService().ResetDecoder() + 区分 reason P1 应修
4 京东缺 PING 心跳 京东 JoyAI JoyInside 文档建议 30s 一次，弱网/4G 网关可能 60s+ 闲连接断 xTimer 30s 周期发 contentType=PING，OnAudioChannelClosed 时停 P1 应修
5 mid 唯一性 京东 JoyAI 文档语义"请求唯一标识符"，代码用 bot_id_ 填充，所有上行 mid 一样 每次 BuildClientEvent 用 UUID 或 esp_random() + 单调计数器 P2 选修
6 京东 ASR textType 区分 京东 JoyAI 每条 ASR 都 emit stt，包括中间识别结果，UI 反复刷新 + 重复 popup 仅 textType=IS_FINAL 才 emit（本次刚补，已修 ✓） 已完成
P0 两条是真因：用户日志里 listening 卡死 130+ 秒就是 #1 + #2 联合作用。 单独修任一条都不够：只修 #1 京东能退（120s 兜底生效）但慢；只修 #2 京东仍会被 EMPTY_CONTENT 阻挡。两个一起做才完整。

§ 十 长期运行风险评估

3 遍推演 + 5 项风险，按"不要过度设计 + 真机长期运行 + 弱网"三维度。

10.1 三遍推演结论

推演 1：网络层已有保活，应用层做的是否冗余？

层级谁来管保什么是否冗余
TCP keepalive内核死连接清理⚠️ 默认禁用，通常不开
WebSocket ping/pong (RFC 6455 §5.5)esp_websocket_client双向心跳 + 网络存活✅ 已有，足够保连接
4G PDP keepaliveML307 modemLTE 链路保活✅ modem 自管
JoyAI 应用层 PING本次新加让服务端知道客户端在不冗余 业务语义
EMPTY_CONTENT 不刷本次新加"用户不说话"自动退出不冗余 产品体验

结论：网络层 ping/pong 保连接活着；应用层 PING + 不刷时间戳保产品体验。两者目标不同。

推演 2：是否需要再加 application 30s timer？

方案退出机制用户体验架构成本
A. 仅 P0-1（当前） 基类 120s IsTimeout（被动检查） 120s 后用户操作时才察觉 0 行
B. P0-1 + application 30s application 主动 SetState(Idle) 30s 自动退出 30 行
C. P0-1 + 协议层 30s 每个协议各自重写 30s 退 每协议 30+ 行

当前选 A，理由：先测基类 120s 兜底是否对真机够用。120s 兜底通过 SendAudio 链路被动检查：

  1. 用户不说话 → application 仍在 listening，audio_service 持续 SendAudio 上行
  2. JoyAI 服务端 EMPTY_CONTENT 不刷时间戳，last_incoming_time_ 老化
  3. 120s 后 IsTimeout 返回 true → IsAudioChannelOpened 返回 false → SendAudio 返回 false
  4. 音频持续发送失败 → 触发 OnDisconnected 或上层察觉 → 切 idle
真机验证后再决定：如果 120s 内真能退（音频发送失败传导切 state），方案 A 够用；如果 120s 后仍卡 listening 不退，需要补 B（30s timer）。

§ 十一 弱网处理策略

P30 量产以 4G modem (ML307) 为主，弱网/抖动/失联是常态，必须分层处理。

11.1 各场景的处理路径

弱网场景 哪一层处理 用户感知
WiFi 临时丢包 100ms ① TCP 重传 语音稍慢，无感知
4G 切基站 1-2s ① ml307 重发 + TCP 重传 语音延迟 1-2s
ws 心跳 60s+ 无响应 ① ws 库 OnDisconnected → ③ 自动回 idle
服务端处理卡死但连接活 ② JoyAI PING 失败 3 次 → CloseAudioChannel → ③ 自动回 idle
用户不说话 + 服务端持续 EMPTY_CONTENT ② 不刷 → ① SendAudio 120s 失败 → ③ 100 帧失败 close → 自动回 idle ~126s
完全失联（飞行模式） ① TCP RST → OnDisconnected → ③ 自动回 idle

11.2 还未验证的边界

  • EMPTY_CONTENT 兜底链路完整性：120s 后 SendAudio 返回 false，audio_service 是否会触发 OnDisconnected 或主动切 state？需真机测
  • 弱网下 PING 串扰：30s 周期 PING 是否与音频上行冲突影响延迟？真机抓 ws 看时序
  • 断路器误触发：3 次 PING 失败阈值是否过紧？弱网下可能正常情况下也失败几次
  • OnDisconnected 重连：京东侧重连流程是否健壮？application 切 idle 后用户再唤醒能否重建连接
