4 License 三态时序（LIC_MUST / PASS / FAILED）

百度服务端鉴权流程 · 与 OpenAudioChannel 10s 等待事件位耦合。

设计要点

  • 永不立即发送 LIC_ACTIVE：WS 刚建立时 ML307R UART 仍在处理 TLS/WS 帧 · 立即发送必失败 · 浪费 TCP 失败名额
  • 5s 延迟（lic_retry_timer）：等 modem UART 空闲 · 大幅降低失败率
  • 机会式重试：TTS_END / REMOTE_PLAYER_END / SendAbortSpeaking 都调 TryFlushPendingLicActive（UART 此时必空闲）
  • FAILED 永久停止重试：服务端拒绝 = 客户端无救 · 立即 SetError
  • license_key 空：跳过整个流程 · 直接置 MEDIA_READY · 用于本地测试或 token-only 鉴权场景

Idle 超时分级降级（保 WS 连接）

CheckIdleTimeout 不直接 reset · 而是"软关闭音频通道，保留 WS 连接"用于服务端推送。

步骤动作含义
1idle > 5 min · idle_goodbye_sent_=false判定空闲
2idle_goodbye_sent_=true · Schedule lambda派发到主线程
3SendTextToAI("我要先去忙了，你跟我说再见吧")云端 TTS 道别
4嵌套创建 idle_dc timer 3 s等 TTS 播放
53 s 后 ResetDecoder + media_ready_=false + on_audio_channel_closed_关音频通道 · WS 不动
6xTimerDelete(idle_dc)清理嵌套 timer
嵌套 timer 析构风险：idle_dc 由 CheckIdleTimeout 内动态创建 · 析构靠回调内 xTimerDelete · 如果 WebsocketBaiduProtocol 在 idle_dc 触发之前析构，会通过 prevent_destroy_guard 防止访问悬空 this，但 timer 本身泄漏。**实际场景**：仅在 esp_restart 前发生，restart 后所有 timer 都被清，无累积。

8 上下行帧长协商（v2.2.12）

详见 总文档 § 6.7 · 此处补充诊断细节。

OPUS TOC byte 解析速查

TOC byte 高 5 位 config 字段定义帧长（RFC 6716 § 3.1）：

config 值模式采样率帧长
0-3SILK NB8 kHz10/20/40/60 ms
4-7SILK MB12 kHz10/20/40/60 ms
8-11SILK WB16 kHz10/20/40/60 ms
12-13Hybrid SWB24 kHz10/20 ms
14-15Hybrid FB48 kHz10/20 ms
16-19CELT NB8 kHz2.5/5/10/20 ms
20-23CELT WB16 kHz2.5/5/10/20 ms
24-27CELT SWB24 kHz2.5/5/10/20 ms ← 多帧情况
28-31CELT FB48 kHz2.5/5/10/20 ms
注意：单帧 OPUS 包只能表示≤20ms · 60ms 必然是 multi-frame packet（CELT 模式不支持 60ms，必须用 SILK / Hybrid 多帧打包）。所以 PROBE 日志 config 字段需要结合 packet length 判断：
  • config 0-11 + 60ms 帧长 = SILK 60ms 单帧（最可能）
  • config 12-15 + ~60ms = Hybrid 多帧（不常见）
  • config 16-31 + ~60ms = CELT 3 × 20ms 多帧（罕见）

预期 PROBE 日志样本

真实下发帧长预期日志解读
20ms @ 24kHzpacket=~40B · TOC=0x68 · config=13 · 20ms?服务端未识别 v2.2.12 协商 → 回滚
60ms @ 24kHz SILKpacket=~120B · TOC=0x38 · config=7 · 60ms?协商成功 ✓
60ms @ 16kHz SILK WBpacket=~100B · TOC=0x58 · config=11 · 60ms?协商成功 + 降采样了

10 断路器 + 快速重试（音频/控制双独立）

百度协议独有 · 设计目标：4G 弱 CSQ 下避免无谓的 Ml307Tcp 5 s 超时堆积。

音频通道断路器（SendAudio）

状态条件行为
正常failures = 0正常发送 · 失败计数+1 + 记 first_audio_failure_time
跳过failures ≥ 3不再尝试发送 · failures 继续累加 · 每 50 次记日志
关连接failures ≥ 10 && elapsed ≥ 5 swebsocket_->Close() · 触发 OnDisconnected 重连
恢复任一帧成功failures 清零 · "Audio recovered after N failures"

控制面断路器（SendText）

独立计数control_send_failures_ · 与音频隔离避免相互污染
恢复策略同音频 · 任一成功清零

快速重试（时间感知）

路径第一次失败耗时动作
SendAudio< 50 ms立即重试一次 · WiFi 瞬时失败常见
SendAudio≥ 50 ms不重试 · 4G 慢超时跳过避免堆积
SendText< 200 ms立即重试一次（控制面阈值更宽松）
SendText≥ 200 ms不重试

11 4G UART 阻塞耦合（ML307R 专属风险）

核心机制：P30-4G 板用 ML307R Modem，其 UART 是单线程串行。任何时刻只能发一帧 PPP 数据。WS 帧 + LIC_ACTIVE + 音频帧若挤在 UART 队列，后到的全部 5 s 超时。

容易踩坑的场景

场景UART 占用百度协议应对
WS 刚连上立即发 LIC_ACTIVEWS 握手 TLS 帧仍在 UART · LIC 排队 → 5 s 超时✅ ScheduleLicRetry 5 s 延迟（避开 TLS 阶段）
TTS 播放中持续上行音频每 60 ms 一帧上行 + 每 60 ms 一帧下行 · UART 双向忙⚠ 上行有断路器（>3 失败跳过）· 下行无控制
TTS_END 回调立即发 LIC_ACTIVETTS_END 文本帧还在收 · 立即发会撞 · 5 s 超时✅ TTS_END 内调 ScheduleLicRetry（不直接 TryFlush）
断网 + 自动重连FetchSessionToken HTTP + WS Reconnect 并发抢 UART⚠ 无显式锁 · 依赖 Schedule 串行

UART 友好的发送时机（机会式重试）

时机UART 状态百度协议利用
TTS_END_SPEAKING 后下行音频流停 → UART 半空HandleEvent 内调 ScheduleLicRetry
REMOTE_PLAYER_END 后同上同上
SendAbortSpeaking 末尾用户主动打断 → 音频流即将停TryFlushPendingLicActive 立即重试（不走 timer）

12 与百度 BRTC SDK 的差异点（我们绕过 SDK 直连）

设计选择：P30 不集成 BRTC SDK，直接用 WebSocket 与百度服务端通信 · 原因：BRTC SDK ~3 MB Flash + ESP32 移植成本高 + 不需要 WebRTC 完整能力。
维度BRTC SDK（标准方式）P30 WebsocketBaiduProtocol（本实现）风险点
连接层QUIC / WebRTC RTPWebSocket无 RTP 重传 · 弱网丢包不可恢复
音频上行RTP 包含 timestamp + ssrc + seq裸 OPUS payload无序列号 · 服务端无法检测乱序丢帧
音频下行RTP 解包裸 OPUS payload + TOC 自解析无 PLC（丢包隐藏）
帧长SDK 内部固定 20 ms（与 WebRTC 对齐）客户端可配（v2.2.12 60 ms）服务端是否兼容需联调
LicenseSDK 内部处理 · 透明客户端手动 LIC_ACTIVE / 重试UART 阻塞需自己规避
TokenSDK 自动刷新客户端 HTTP 预取 + 缓存 + 失效重取HTTP 失败时 fallback 路径有限
AECSDK 内置回声消除设备端 AFE + 服务端 dfda 配合无 SDK 的端到端调优 · 自己做参数
音乐控制SDK 提供 RemotePlayer API手写 CMD_MUSIC_* 文本协议协议升级时需要同步
Flash 占用~3 MB~50 KB（仅 .cc 编译产物）—
调试可视化SDK 内置统计 + 日志手写 audio_tx_frames/bytes + PROBE 日志—
