3 main/flow_engine.cc 评审

FlowEngine 是直播伴侣 / 通用流程引擎，状态机 4 态（kIdle / kPlaying / kDelay / kSuspended），通过 HTTP 拉取 JSON 脚本按序循环播放 TTS/TTAI 内容。核心数据流：HTTP → ParseScript → items_ → PlayCurrentItem → SendTextToAI/Tts → OnDeviceStateChanged → ScheduleNext。

3.1 ① 并发 / 资源（4 项）

#问题实测/估算修复
F1静态栈非重入安全：Start() 内的 s_flow_load_stack / s_flow_load_tcb 是函数 local static。Start 虽先调 Stop，但 Stop 不等 load_task_handle_ 退出；用户连发两次 Start（或脚本未拉完 user 又远程发 reload），第二次 xTaskCreateStaticPinnedToCore 直接复用同一 TCB+栈，两个任务共享栈 → 100% 翻车长脚本 / 弱网 15s 重试时窗口 5-30sStart 入口判 load_task_handle_ != nullptr → return 或先 vTaskDelete+等 100ms；或改 dynamic TCB + 自销毁
F2pending_url_ 跨任务无同步：主线程在 Start() 写、LoadScriptTask 在新任务读。配合 F1，连发两次 Start 时 task A 读 url 中途被 task B 写入新 url → 拿到拼接错乱同 F1把 url 通过 task arg 传入（避免共享成员）
F3ParseScript 持 script_mutex_ 时 Schedule(Alert)（行 322-326 / 347-350）：锁粒度放大，且 Alert 内部 LVGL 可能再触发 schedule，潜在死锁面低概率把 Alert/Schedule 提到锁外执行
F4last_script_json_ 写入无锁：当前仅主线程访问，但没文档约束。若后续 remote_cmd 在协议任务直接调 RestartLast → race设计风险header 注释"仅主线程访问"或加 mutex

3.2 ② 状态机 / 时序（5 项）

#问题实测/估算修复
F5加载期间 state_=Idle 死区：Start 在 LoadScript 完成（200ms-15s 重试）才把 state_ 置 kPlaying。这期间 IsRunning() 返 false，NotifyTtsFinished / Suspend / Resume / OnDeviceStateChanged 全部无响应。极端情况：用户在加载期唤醒说话 → Speaking 状态正常 → 加载完成 task 把 state_ 强切 kPlaying + ForceListeningMode(Manual) → 直接淹没用户当前对话弱网必现入 Start 立即 state_=kLoading（用 kPlaying + loading_ flag 复用 4 态，符合 CLAUDE.md "12 态不新增"），LoadScript 期间 OnDeviceStateChanged 早 return
F6OnDeviceStateChanged TOCTOU：行 495 IsRunning() 通过 → 行 506/513/527 读 state_，期间外部线程 Stop() 可把 state_ 切 Idle，仍走 ScheduleResumeAfterDelay → idle 状态被错误恢复量产偶发关键路径 state_.load() 缓存到本地变量后一次性判断
F7recap prompt 无长度上限（行 206）："刚才直播间讲到「" + recap_text + "」..."，recap_text 直接来自脚本，如果脚本项 2KB → AI 端可能拒绝/超时取决脚本来源截到 200 字 + "..."
F8Listening→Idle 强切打断用户唤醒（行 517-523）：检测到 Speaking→Listening 立即 Schedule(SetDeviceState(Idle))，若用户正在 AutoStop 模式说话 → 被强切回 Idle体验问题加 200ms 宽限等用户真的没说话再切
F9PlayCurrentItem 持锁内网络发送（行 406+441/448/451）：lock_guard 持锁期间调 SendProtocolText/SendTextToTts，锁粒度过大中等把 item 复制出锁，发送在锁外
⚠ F5 状态机时间轴可视化

4 main/remote_cmd.cc 评审

RemoteCmd 是云端 19 类远程命令分发器，全 Protocol 任务上下文同步执行。命令通过 {"type":"custom","payload":{"message":"..."}} 三种格式之一下发（双层 / object / 顶层）→ Handle() 用 strcmp 链 dispatch → 各 On* handler 通过 Schedule 切主线程做实际工作。

4.1 ① 并发 / 资源（5 项）

#问题实测/估算修复
R1main_loop 内 vTaskDelay 2-3s：OnReboot（行 92）/ OnOta（行 111）/ OnSleep（行 324）的 Alert+delay 写在 Schedule lambda 内，main_loop 是 P10 高优先级常驻任务，被阻塞 2-3s → 所有 Schedule（包括 AlarmRinger.Start、UI 刷新、protocol 收发）全部冻结。~/.claude/rules/esp32-memory.md § "禁止 portMAX_DELAY 裸用 / 阻塞实时任务" 红线远程命令必触发用 esp_timer_start_once 延后真正执行 Reboot/EnterDeepSleep；Alert 用回调方式而非 sleep 等渲染
R2PostSttText 静态栈窗口竞态：stt_posting_ atomic_flag 在 task 进入 vTaskDelete 前置 false（行 420），但 TCB/栈实际释放在调度器层。spawn 间隔 < 10μs 时新任务复用同一 TCB → 旧任务上下文未完整销毁罕见但真红线flag clear 改放 vTaskDelete 之前但留 1 tick yield；或动态 TCB（heap_caps_calloc INT）
R3OnDownload cJSON_Duplicate 死代码（行 230）：复制整个 files 数组数 KB → Schedule 进 main_loop → lambda 里只 LOGW 没用 files_copy 直接 Delete。File sync 在 V2 未实现，但代码已分配浪费 INT RAM直接返回 / LOGW 即可，删 cJSON_Duplicate
R4Handle() 全 Protocol 任务上下文同步执行：依赖 19 个 handler 各自 Schedule。若某 handler 同步走 cJSON 解析 + 大 strcmp 链 + 字符串构造 → Protocol 接收线程被拖慢 → 后续 WS 帧延迟持续轻度影响现状可接受，但应做最坏路径压测（OnFlow inline script 4KB cJSON_PrintUnformatted）
R5OnVolume / OnGain 无范围校验：value 传 -1 / 999 直接送 codec→SetOutputVolume，依赖 codec 内部 clamp协议契约弱0-100 / -30~30 clamp
🔴 R1 详解 · main_loop 阻塞链路
RemoteCmd::OnReboot::Schedule(lambda) 投递到 main_loop（Core 0 P10）→ lambda 内 app_->Alert(...)（同步 LVGL）→ vTaskDelay(2000ms) → main_loop 被钉死 2s → 期间 AlarmRinger 到点回调 Schedule(Start) 入队但不执行 → 闹钟延迟 2s 才响 / WS 心跳超时 / UI 卡顿。

正确做法：app_->Alert("重启设备", "正在重启", "", OGG_VIBRATION); esp_timer_start_once(reboot_timer, 2*1000*1000); → 定时器 ISR 调度 app_->Reboot()，主循环不阻塞。

4.2 ② 状态机 / 时序（5 项）

#问题实测/估算修复
R6OnFlow start 双层 Schedule（行 268-270 / 282-284）：lambda 已 Schedule，内部 lc->Start() 又自己 Schedule → 跨两个 main_loop tick 才真正开始下载；期间状态机暴露死区。配合 F5 → P0 联动必发生删 OnFlow 内 Schedule，直接调 lc->Start（Start/StartWithScript 内部已 Schedule）
R7OnMusicPlay 内 Suspend / AbortSpeaking 顺序错（行 467-478）：先 AbortSpeaking(Speaking→Idle) 触发 FlowEngine.OnDeviceStateChanged → 进入 "Speaking→Idle 走 ScheduleNext" 路径 → 推进了 current_index_，之后 Suspend 才打标 → 恢复时跳了一项已知漏跳 bug 候选顺序倒过来：先 Suspend(打标 kSuspended) → 再 AbortSpeaking → 再 CloseAudioChannel
R8OnReload SwitchProtocol 析构同步性未声明（行 247）：注释说"避免 lambda 在 protocol/remote_cmd 重建期间被析构"，所以捕获 app = app_ 而非 this。实现正确，但 Handle() 调用者也需要这种意识——任何 lambda 捕 this 都得想清楚设计风险header 注：所有 On* lambda 必须 capture by value，不得依赖 RemoteCmd 实例存活
R9OnEduPool 异步执行无错误回传（行 558-562）：Schedule lambda 内 UpdateFromString 失败仅 ShowNotification，不向云端 ACK 失败。云端无从得知 names 格式错误远程运维盲点加 ACK 通道（与 mcp-flows § 9 D 类 diag 工具一脉相承）
R10Handle() top-level 兼容路径 const_cast（行 50）：把 const payload 当 msg 用，是因为云端有三种下发格式（双层 / object / 顶层）。实现正确但注释仅一句，新手难定位演进债头文件画 3 种格式对照表（已有但不全）
