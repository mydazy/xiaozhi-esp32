2.4 三遍交叉视图（哪些 bug 互相加重）

B6 + B12 双爆链：手机发 RECV_CUSTOM_DATA → BlufiCallback 写 NVS → flash op 禁用 cache → opus_codec PSRAM 栈访问失败 → SP=0x60100000 · 同时 B6 的 ble_connected_ 裸读写让 SendData 主线程读到 stale true，再调 esp_blufi_send_custom_data 触发更多 BT controller 操作 → 加重 race。
B3 + B16 双爆链：Stop 不调 security_deinit → blufi_sec 残留 · OnConfigSuccess 后 reboot 干净避开 · 但若用户走 SwitchConfigMode 在不重启情况下从 BLUFI 切 AP 再切回 BLUFI · blufi_security_init 第二次调 short-circuit · DH 第一帧用旧 dhm 上下文 → 握手失败手机端报"配网失败"无具体原因。
T8 + B12 安全风险链：恶意手机连上 BLUFI（无配对）→ 发大量 RECV_CUSTOM_DATA → 每包都写 NVS → Flash 写入次数耗尽（NVS 寿命 100k 次擦写）+ 期间频繁 cache 禁用导致 Core 0 PSRAM 栈任务 crash → 设备 brick。

InitializeController + Start 关键步

  1. 调用方持 LVGL 锁（防 BT init 期间 LCD DMA 抢 SPI）
  2. esp_blufi_controller_init / enable(BLE)

  4. esp_blufi_host_and_cb_init（NimBLE host 启动 · MTU 247 · GATT profile 注册）

6 退出 + RAM 释放

RAM 释放的根因（量产关键）

为什么 OnConfigSuccess 必须 Reboot 而不是原地切状态：配网期 BLE+WiFi+提示音 占用大量 RAM，原地继续走 SetDeviceState(idle)→OTA→MQTT 时内部 RAM 会跌到 ~57KB 触达 60KB 红线；NVS flash op 禁用 cache 时 PSRAM 栈任务（opus_codec / cdc_console / audio_detection）会 Double exception (SP=0x60100000)。重启后凭证已保存，走 SmartConnect 干净路径。
代码注释：wifi_board.cc:465-470
场景BT 是否 init释放结果
首次开机 + 配网成功 + 重启已 init（配网期）❌ 不释放（BT 已占用）
第二次启动 SmartConnect 成功从未 init✅ 释放 25-35 KB INT
每次启动都失败回 BLUFI每次 init❌ 永远不释放（设计如此）
