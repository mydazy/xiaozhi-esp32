1 模块定位

FlowEngine 是 P30 三个会话型 ActivityType 之一（kFlow）。承载"按脚本编排的多步骤交互"——区别于即时对话（kChat）与单一播放（kMusic）。命名沿革：LiveCompanion（直播伴侣） → 2026-04 抽象重命名为 FlowEngine（流程引擎），仍保留 live_companion.start 等远程命令字段向后兼容。

1.1 在 P30 三维心智模型的落位

维度取值说明
A · DeviceState复用 12 态 · 不新增播放时占 kSpeaking + kIdle 来回，等用户打断走 kListening
B · ActivityTypekFlowmain/activity_type.h · 与 kChat/kMusic 并列
C · SceneType不绑定特定 SceneUI 沿用 Emoji / Player / Clock · 不抽 FlowScene
AudioSourcekTts所有 item 走 TTS 通道 · 不占用 kMp3/kAlarmBell 等独立源

1.2 模块边界（不做什么）

✓ 做脚本解析 / 状态机推进 / TTS-TTAI-SET 三类指令分发 / 用户交互打断与恢复 / 循环控制
✗ 不做TTS 合成本身（交 Application）· LVGL 渲染（交 UiDisplay）· 网络协议（交 Protocol）· 唤醒词检测（交 AudioService）
⚠ 边界灰HTTP 下载（自己起 task · 用 Board::GetNetwork）· DeviceState 监听（订阅 DeviceStateEventManager · 不主动改 state 但会 SetDeviceState(Idle) 兜底）
📐 抽象层级：FlowEngine 是"会话型 Activity 的脚本驱动器"——它本身不是 ActivitySession 基类的实例（CLAUDE.md § 一.5.5 已明确"不抽 ActivitySession 基类"）。它通过 ActivityType::kFlow 在三维落位表登记，运行时由 Application 持有单一 unique_ptr 实例。

步动作state_说明
1用户唤醒说话（或外部 MCP 触发）kPlaying / kDelay—

✨ 设计亮点 · recap 回顾机制
直接续播被打断的项体验生硬（用户已经"出戏"）。FlowEngine 通过 构造一句 TTAI 提示（拼当前 item.text）让 AI 用自己的话先回顾再继续，无缝衔接。这是 LiveCompanion 时代积累的核心 UX 优化。

6 现有互动能力盘点

基于当前 4 状态机 + 3 ScriptItem 类型 + 双路加载 + 暂停恢复机制，FlowEngine 已能支持以下 6 类互动。

① 顺序播报

直播伴侣最经典场景：脚本预编排 N 条 TTS 文案，按序播 + 等待 + 循环。延时可逐项配置。

② AI 互动对话

kTtai item 让脚本写入 prompt 串，AI 自行生成回复 + TTS 播报。比如"用一句话夸夸今天的小听众"。

③ 配置切换

kSet item 中途切音色 / 切渠道 / 切角色 prompt。SET 不等待，链式立即推进。

④ 用户打断 + 自然恢复

用户随时唤醒说话 → Suspend → AI 对话 → 5s 静默后 recap 回顾上文 + 续播。体验无缝。

⑤ 循环播放

loop=true 支持长直播守夜场景。loop_count_ 暴露轮次，便于云端监控。

⑥ 跨场景脚本互换

URL 加载（运营更新）+ WS 推送（实时下发）两路并行，云端可针对设备状态动态切脚本。RestartLast 支持唤醒续。

6.1 当前能力 vs 三维心智模型

能力占用 DeviceState占用 ActivityType占用 AudioSource
顺序 TTS 播报Speaking↔IdlekFlowkTts
TTAI 对话Speaking↔IdlekFlowkTts（AI 回复也走 TTS）
Suspend 用户打断Listening / SpeakingkFlowkTts
SET 配置切换无切换kFlow无

9 风险与边界

9.1 不要扩展的方向

方向不做理由
抽 ActivitySession 基类CLAUDE.md § 一.5.5 已锁定。FlowEngine / MusicPlayer / AudioService 形状差异大，强行抽公共接口会违 SRP
新增 DeviceState 枚举CLAUDE.md 量产纪律 · 12 态封死。kWaitUser 应复用 kListening + state_=kSuspended 组合实现
FlowEngine 内做 NLP / 模糊匹配设备侧只做命中关键词；理解交云端 TTAI
FlowEngine 并行多脚本Application 仅持单 FlowEngine 实例。多脚本需求改服务端编排为单脚本

9.2 ScriptItem 扩展的兼容策略

新字段兼容做法
新增 type 枚举（kWaitUser / kMcp）ParseScript 中未知 type → 当 kTts 兜底（不丢项）
新增 emotion / led_pattern / chapter_end 等字段缺字段 = 默认值 · 旧脚本无感
新增 priority / next_id 等结构性字段引入 schema_version · 服务端按设备 version 下不同结构

9.3 关键内存 / 性能预算

资源当前扩展后预算
脚本最大字节8 KB不变（H2 流式拉取实现前不放宽）
items_ 数组vector · 无上限建议加 256 上限 · 防 OOM
last_script_json_~8 KB 字符串不变 · 加 H1 NVS 缓存后 NVS 占用 +4 KB
flow_load 任务栈INT 6 KB · static不变（F1 修后 dynamic 也仍 6KB）
delay_timer_esp_timer · 1 个不变（书签 / Pomodoro 新增 timer 独立）

9.4 验证清单（每加一项互动 PR 自检）

  • 新加的 ScriptItem 字段在 ParseScript 中有缺省值
  • 新 type 枚举在 ParseScript 中默认回退 kTts
  • 状态机 4 态转换图无新增边（如不可避免 → 文档评审）
  • 不新增 DeviceState 枚举
  • FlowEngine 不直调 LVGL / cJSON 之外的第三方库
  • 新增 task 走 docs/p30-architecture.html § 四.1 落位表登记
  • 修改 ScriptItem 结构 → bump schema_version
  • 云端脚本可在 nvs 4 KB 装下（H1）
✅ 终极判据：每个扩展项问自己——能否仅通过修改服务端脚本 + 增加 ScriptItem 字段实现？ 如果能 → 优先做；如果必须改 FlowState 4 态机或拆 ActivitySession 基类 → 触发文档评审，宁可不做。
