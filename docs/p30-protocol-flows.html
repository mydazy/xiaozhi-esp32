2.4 IsTimeout 机制

关键：基类 IsTimeout() 默认 120 s 没收到任何消息即超时。IsAudioChannelOpened() 实现统一为 (transport 在 + !error + !IsTimeout)。 意味着：3 分钟无服务端心跳 → 通道判超时 → 用户下次唤醒需要重 OpenAudioChannel。这给云端"长时间无对话自动关闭 session"留了观察窗口。

3.2 启动与重连

3.3 OpenAudioChannel（hello + UDP 拉起）

  1. 检查 mqtt_ 已连 · 否则 StartMqttClient(true)
  2. error_occurred_ = false + 清 session_id_ + 清 SERVER_HELLO event bit
  3. 发 hello 消息（含 features.aec/mcp + audio_params 16k opus 60ms）
  4. EventGroupWaitBits 10 s 等 SERVER_HELLO_EVENT（阻塞调用方任务）
  5. 失败 → SetError "服务端超时" → return false
  6. 成功 → lock channel_mutex_ → 创建 UDP · Connect server:port
  7. 注册 UDP OnMessage（含 sequence 校验 + AES 解密）
  8. 调 on_audio_channel_opened_ 回调

3.4 服务端 goodbye 兵乓机制

服务端发 goodbyesession_id 匹配时 Schedule(CloseAudioChannel(false))
客户端主动 closeCloseAudioChannel(true) 发 goodbye 给服务端
避免兵乓send_goodbye 参数 · 服务端发来的 close 不回发

9.3 OpenAudioChannel 在 main_app 阻塞 10 s 风险

关键：ContinueOpenAudioChannel / ContinueWakeWordInvoke 通过 Schedule 派发到 main_app 任务。它们内部调 OpenAudioChannel · 而 OpenAudioChannel 内部含 xEventGroupWaitBits(... 10000) 等服务端 hello。 意味着：4G CSQ 差或服务端慢，main_app 任务卡 10 s。期间所有 Schedule lambda 都不能执行（UI 刷新、其它按键、闹钟检查等都被推迟）。

12 评审二 · 切换流程

12.1 ResetProtocol 异步陷阱

核心问题：Application::ResetProtocol 是 异步 Schedule（application.cc:1423）· 调用方调它后立即返回 · 但实际 CloseAudioChannel + protocol_.reset() 要等 main_app 任务处理下一个 MAIN_EVENT_SCHEDULE 才执行。

潜在后果：
  • EnterDeepSleep 流程：ResetProtocol() → vTaskDelay(500) → GracefulShutdownModem → 断 LDO。若 main_app 在这 500 ms 内被其他 Schedule lambda 卡住（如 SetPowerSaveLevel 的同步路径），CloseAudioChannel 不会及时跑 → 服务端 session 未优雅 close · 同时 4G modem 仍在 PPP → GracefulShutdownModem AT+CFUN=4 多等几个包
  • SwitchNetworkType 流程：1500 ms 余量较多，相对安全

12.2 触发点对比

12.4 ResetProtocol 是否应该改同步？
