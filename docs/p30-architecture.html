§ 一 四层架构鸟瞰

P30 采用经典的「业务 → 服务 → 硬件 → 平台」四层结构。L4.5 / L3.5 不存在——三遍评审后明确：业务/UI 抽象用 enum 收拢即可，不引入新基类。Application 是单例编排中枢，Board 工厂在编译期决定具体硬件实现。

L4 业务编排 L3 服务抽象 L2 硬件抽象 L1 IDF / 第三方

§ 一.1 分层原则（量产期纪律）

  • L4 只与 L3 接口对话，不直接操作硬件（穿层是事故母体）
  • L3 通过 L2 的基类接口屏蔽板间差异（Board / AudioCodec / Display 都是工厂模式 + 虚函数）
  • L2 同时承载抽象基类（Board / Display / AudioCodec / NetworkInterface）和具体 driver（LED / Backlight / typec_headset）——这是 ESP-IDF 嵌入式的工程现实，不视为分层错误
  • L2 基类定义合同，子类落地；新增板型只复制 boards/ 目录，不动基类
  • L1 是 IDF 与第三方组件，不动；上游 xiaozhi-esp32 改动的部分要登记到 § 十"上游 vs 自研"对照表
  • 三个 enum（DeviceState / ActivityType / SceneType）横切L4，不构成新层

§ 二.3 启动链路三遍评审决策记录

本节为 § 二 启动链路的三遍独立评审结果，与 § 一.5.5 状态机评审、§ 一.5.7 四层架构评审形成评审三联节。14 条发现 + 4 个候选优化方向 + 推荐路径，决策可追溯。代码层重构留作下一波 PR-A，本节先固化心智共识。

已落地 · 1500ms 神秘延迟修复（方案 B）

2026-04-28 已修复 · 原 application.cc:180 的 vTaskDelay(1500ms) 已迁移到 ml307_board.cc::NetworkTask 开头。根因：GPIO9（AUDPWR-EN）级联开关同时上电 LCD + 音频 + 4G VDD_EXT 三负载，ML307R boot 完成（典型 800-1200ms）后才能响应 AT；不等会触发 AtModem::Detect 重试 30 次（最坏 30 秒浪费）。收益：① WiFi 板（mydazy-p30-wifi）启动同步段省 1.5s；② 4G 板复用 NetworkTask 独立 xTask，Application 主流程提前 1.5s 进入 Run() 主循环。

Phase 化重构 · 顶层结构（候选 1 设计草案）

// main/startup_phases.h（拟新增 · 量产前 PR-A）
class StartupPhases {
public:
    enum class Phase : uint8_t {
        Hardware,      // GPIO / 电源 / I2C
        UI,            // LCD / Backlight / boot animation
        Service,       // Audio / Timer / MusicPlayer
        Registration,  // Callbacks / Listeners / MCP
        Network,       // 异步起飞（不阻塞）
        Activation,    // 异步起飞（不阻塞）
        Ready          // 进 Run() 主循环
    };

    bool RunPhase(Phase p);
    Phase GetCurrent() const;
    void ReportProgress(int percent, const char* msg);

    // BootMode 决定走哪些 phase
    void ConfigureForMode(BootMode mode);
};

// main.cc（精简后）
extern "C" void app_main(void) {
    esp_event_loop_create_default();
    nvs_flash_init();

    auto mode = DetectBootMode();        // 5 种唤醒入口分支
    StartupPhases phases;
    phases.ConfigureForMode(mode);

    for (auto p : phases.Sequence()) {
        if (!phases.RunPhase(p)) {
            EnterFatalError(p);          // phase-level error
            return;
        }
    }
    Application::GetInstance().Run();
}
关键架构演进价值 · 这套 Phase 化重构同时解决 S6/S7/S12 三个高严重度问题：① Initialize God Function 拆解 → 层次正确性 ② 隐式串行 → 显式 7 阶段 ③ main.cc 担当启动调度 → 关注点分离。BootMode 入口分支（候选 2）让 5 种唤醒路径各自跳过不需要的 phase，闹钟/加速计唤醒响应从 ~2s 降到 < 500ms。
量产期纪律 · ① S1/S2 已修复（方案 B），其余 12 项纯文档登记，代码层下一波再改；② 新增启动期代码必须挂在某个 Phase 下，禁止再加散落 init；③ 任何打破 phase 边界的"穿层 init"在 PR review 时直接拒绝；④ Phase 内部失败必须返回 false + 错误码，不允许 ESP_ERROR_CHECK panic（量产期不能直接挂掉）；⑤ 候选 1 真要落地前先做 git branch + 三板编译矩阵验证。

③ 复用 Schedule 队列 Activity 不挂独立任务。Start()/Stop()/OnInterrupt() 必须 <10ms 返回，长 IO 走 Application::Schedule(lambda) 每 Activity 一任务 → 6×8KB = 48KB 内部 RAM 红线爆
④ 需求拉动渐进抽 不预先抽。出现第 N 个同类才抽（4 个会话型抽 LongSession / 6 个 Scene 抽 Factory / 6 个 AudioSource 抽 AudioOwner） 预先抽错 → 重写代价 10×；渐进 → 单阶段独立可回滚

§ 五 音频数据流（端到端）

录制和播放是两条独立链路，通过 AudioService 内部队列串接 5 个任务。采样率 24 kHz mono 16 bit，Opus 帧长 60/80/100ms 自适应。
