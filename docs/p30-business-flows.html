综合 RAM 占用表（基线 + 业务流 Δ · v2.1 已对齐最终代码）

基线 4G Idle = 145 KB INT · PSRAM ~4.5 MB（MultiNet 已禁用 · 实测）。任意时刻总占用 = 基线 + 该业务流 Δ。
P0+P1 完成态修正：(a) 红线 3 项已修（PSRAM→INT 多用 ~22 KB INT，已并入 ① 199K）(b) 14 处漂移 Pin Core 不影响 RAM (c) HTTP Binary 模式带宽翻倍但 RAM 占用持平 (d) MonitorMemoryHealth 60s 监控 INT/PSRAM/largest_block 三项。

项目流程合理性结论（v2.1 · P0+P1 完成态） · 主流程图反映的 6 业务分支 + Idle 中枢架构 整体合理且已进入量产可控状态。原阻塞项 (a) 红线 3 项 (b) 14 漂移 (c) FatalError bug (d) 弱网 W4 timeout (e) ML307 弱信号注册 — 全部修复 ✅。剩余风险：对话+唤醒词叠加场景距 60K 红线 30K（量产前 24h 实测验证）。
后续项目对齐原则 · ① 任意新业务流必须能落进上图某个分支或新增分支 ② 新业务流必须填 RAM Δ + Core 分配 ③ 不动 Idle 中枢架构 ④ 业务流互斥关系参照 § 9 矩阵 ⑤ 新功能优先复用现有基线任务 ⑥ 新增任务必须 Pin Core（main/ 下 grep xTaskCreate\b 必须保持 0 命中）⑦ 任何 HTTP 调用自动受益 Binary 模式（无需改业务代码）。
