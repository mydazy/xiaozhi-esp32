04 · 跨线程数据流

3 个生产线程 · Schedule() 屏障 · main loop 单线程串行化

教育卡 / 笔画 GIF 数据来自 3 个 FreeRTOS 任务，全部通过 Application::Schedule() 投回 main loop，避免跨线程 race 和 UAF。

Network/HTTP 任务
Timer 任务
Schedule 屏障
Main loop（唯一可访问 g_edu_queue）

关键不变量

  • g_edu_queue 只在 main loop 读写（无 mutex 需求）
  • active_scene_ / in_font_mode_ 只在 LVGL 持锁的 main loop 内更新
  • LVGL 渲染调用必须持 DisplayLockGuard（除 timer/LV 内部异步）
  • FontGif 内 PSRAM buffer 所有权转移给 EmojiCollection（不再 free）
  • OnFontExitClicked / OnEduCardClicked 用 Schedule() 异步退出，避免 LVGL 事件回调内修改 src 触发链问题

07 · 流程交互合理性评审

每环节是否合理 · 哪些可精简 · 哪些是必要 hack

逐路径审视交互链每一步：是否原子？是否冗余？副作用是否隐式？hack 的根因是什么？给出量产期保留 / 未来打磨建议。

路径环节判定评审说明
a · Chat→EduCard EnsureDisplayFonts 延迟加载 ✅ 合理 cbin 字体启动后才挂载，懒加载避开 boot 卡顿
if (in_font_mode_) 预盖 守卫 ✅ 合理 统一入口检查 · 路径 a 走 false 分支零开销 · 路径 f 复用同代码
RenderEduCardLayout 末尾把 bottom_bar 提到 overlay 之上 ✅ 合理 产品决策（教育卡时保留"长按说话"提示），LVGL 缺少"反向 z 锁定"语义，move_foreground 是标准做法
b · EduCard→Chat overlay 单例 add_flag(HIDDEN)，不 lv_obj_del ✅ 合理（必要） 避免 lv_obj_del 与 label 异步 layout cb 的 race（LVGL 9 已知 UAF），是验证过的关键设计
层级复位：bottom_bar / status_bar 重新 move_foreground ✅ 合理 对称于 ShowEduCard 内的提顶，HideEduCard 必须复位否则下次显示层级错乱
c · Chat→FontGIF FontGif 内 in_font_mode_ = false 后才 SetEmotion("font") ⚠ 必要 hack 根因：SetEmotion 检测"相同 emotion"会 short-circuit，导致 GIF 不重启；先清标志强制走完整切换
SetEmotion("font") 内含 HideEduCard / HideQrCode / move_foreground(emoji_box) ⚠ 副作用隐式 违反单一职责，但收敛了 font 互斥逻辑。未来重构方向：SetEmotion 仅切表情 src，互斥副作用搬到 FontGif 显式调用
emoji_collection->ReplaceEmoji 原子 swap ✅ 合理 PSRAM buffer 所有权转移给 EmojiCollection，调用方不再 free，零内存泄漏
d · FontGIF→Chat ResetFontMode 先 in_font_mode_ = false 再 SetEmotion("neutral") ⚠ 必要 hack 根因：SetEmotion 守护 if (in_font_mode_ && neutral) return 会反噬，先清标志才能切回 neutral · 与路径 c 同源根因
SetEmotion 内 is_font != in_font_mode_ 检测切回 bottom_bar 显示 ✅ 合理 边沿检测避免重复 set/remove HIDDEN flag · LVGL 操作幂等但少做总比多做好
e · EduCard→FontGIF SetEmotion("font") 隐式触发 HideEduCard ⚠ 隐式 清场被埋在 SetEmotion 内，调用方不显式可见。同 c 路径的副作用问题
触发条件：教育卡显示中 LLM 调 show_stroke ✅ 合理 产品层决策（自然流），UI 层只需保证切换流畅
f · FontGIF→EduCard 预盖 overlay 后再 ResetFontMode ⚠ 必要 workaround 根因：ResetFontMode 切 neutral 会渲染 1-2 帧 PNG · 预盖让切换发生在黑屏下不可见。当前 workaround 已 stable，无需重构
预盖仅 in_font_mode_=true 时执行 ✅ 合理 路径 a 跳过预盖（无 neutral 闪烁问题），路径 f 才执行 · 守卫精确

⚠ 3 处必要 hack 同源根因

路径 c / d / e 的 3 处 hack 都源于同一耦合点：SetEmotion 一个函数承担了 4 个职责：
  1. 切表情 src（父类 LcdDisplay::SetEmotion）
  2. 是 font 时屏蔽 neutral 切换（in_font_mode_ 守护）
  3. 是 font 时清教育卡 / QR 互斥 overlay
  4. 是 font 时隐藏 bottom_bar
职责 2-4 都依赖 in_font_mode_ 字段，导致调用方必须"前置清标志"才能绕开守护——这就是 3 处 hack 的根源。

重构方向（量产后）：把 2-4 的副作用从 SetEmotion 内移出，由 FontGif / ResetFontMode 显式调用： 
void FontGif(uint8_t* buf, size_t size) {
    DisplayLockGuard lock(this);
    HideEduCard(); HideQrCode();        // 显式互斥清场
    bottom_bar HIDDEN;
    emoji_collection->ReplaceEmoji("font", ...);
    LcdDisplay::SetEmotion("font");     // 父类仅切 src，无守护
    move_foreground(emoji_box);
    in_font_mode_ = true;
}
void ResetFontMode() {
    if (!in_font_mode_) return;
    DisplayLockGuard lock(this);
    in_font_mode_ = false;
    bottom_bar SHOW;
    LcdDisplay::SetEmotion("neutral");  // 直接调父类，无守护
}
void SetEmotion(const char* e) override {
    if (in_font_mode_) return;  // 唯一守护：font 模式不响应外部 SetEmotion
    LcdDisplay::SetEmotion(e);
}
取舍：行为零变更 · 路径 c/d/e 三 hack 消除 · SetEmotion 回归"仅切表情" · FontGif 长度从 16 行涨到 22 行（副作用显式化）

量产期建议：当前 hack 已 stable 半年，无新风险。重构属于"打磨项"，建议量产烧板 3 个月稳定后第 2 个迭代再做。

✅ 6 条路径总评

功能完整性：6/6 全部覆盖 · 状态闭合
视觉流畅性：路径 f 预盖后零闪 · 其他 5 路径本来流畅
线程安全：DisplayLockGuard 全覆盖 · 跨线程 race 已修
清场动作：每条路径都明确（HideEduCard / ResetFontMode / 层级复位）
代码可读：3 处 hack 都有 root-cause 注释，未来重构有据可循

结论：6 条路径作为量产架构合理 · 3 处 hack 不影响功能 · 重构是优化非必要
08 · 命名规范 · 二轮精简后

4 项保留 · 2 项回退（避免过度设计）

分类命名选择理由
SceneType kChat（而非 kEmoji） 产品语义对齐 SwitchToChatMode，不用 UI 元素名
笔画 GIF 入口 FontGif()（而非 EnterFontMode / UpdateFontGif） 精简名词命名，与 ShowEduCard / ShowQrCode 风格统一；"Update" 误导，"Enter" 多余
笔画 GIF 退出 ResetFontMode() 私有（无 ExitFontMode 公共 wrapper） 0 外部调用方，仅内部 5 处使用；wrapper 是死代码
教育卡分类 const char* category 字符串（无 EduCategory enum） 调用入口都是 LLM JSON 字符串；9 enum + 双映射函数是死代码
教育卡渲染主体 RenderEduCardLayout() "已知数据 → 画屏"语义清晰，避免与 ShowEduCard 公共 API 混淆
状态字段 active_scene_ / in_font_mode_ 消除 "current_xx" 命名重复

回退记录

首轮命名重构引入了 ExitFontMode 公共 wrapper 和 EduCategory enum + 双重载， 复盘发现属于过度设计——前者 0 外部调用方，后者所有调用入口都是 LLM 字符串， 9 个 enum 值实际只用 1 个 bool 信号（是否触发 88px 主秀）。 二轮回退共删除 ~50 行死代码 + 历史修复注释，公共 API 从 7 个收敛到 5 个。

设计原则：量产稳定期，实际使用驱动 API 形态，不为了"对称性 / 编译期类型安全"提前抽象。
