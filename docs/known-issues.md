# xiaozhi-esp32 已知问题 Known Issues

> 本文件持续维护所有已知问题（P0/P1/P2），是 `/pre-flash-check` 放行的必过项之一。
>
> **维护规则**：
> - 发现新 bug → 立即新增条目（由 `/diagnose-bug` 触发的报告自动提醒）
> - bug 修复后 → 更新状态为 `已修复 - <commit hash>`
> - 每次 release 前 → 清理已修复且已发版 ≥ 2 个版本的老条目到 `docs/archive/`
> - 复现率、SKU、影响必须写清楚，不接受 "偶发" / "部分设备"

---

## 状态说明

| 状态 | 含义 |
|---|---|
| 🔴 修复中 | 正在调查或已有补丁待验证 |
| 🟡 待修复 | 已确认，未排期 |
| 🟢 已修复 | 已合入，带 commit |
| ⚫ 已归档 | 老版本遗留，当前版本无影响 |
| ⚠️ 不修复 | 成本 > 收益，记录绕过方案 |

## 优先级

| 级别 | 定义 |
|---|---|
| **P0** | 阻断量产 / 影响核心功能 / 安全合规 |
| **P1** | 体验缺陷 / 功能不全但可绕过 |
| **P2** | 优化项 / 非功能性 |

## 模板（新增条目时复制）

```markdown
| #<id> | P<n> | <模块> | <现象一句话> | <100%/偶发-X%/只一次> | <SKU / 批次> | 🔴/🟡/🟢/⚫/⚠️ | <commit hash> | <备注，复现步骤、根因线索> |
```

### 新增条目需包含
- **模块**：UI / 网络 / 音频 / OTA / RTOS / BLE / ...
- **现象**：用户视角，一句话
- **复现率**：100% / 偶发-30% / 只一次
- **SKU / 批次**：必填，没有具体就写"全 SKU 未限批次"
- **关联 commit**：修复后补上
- **备注**：复现步骤、根因、discussion 链接、相关日志文件名

---

## 本文件使用

### 读取
- `/pre-flash-check` 会读取本文件，核对本版本是否有更新
- `/diagnose-bug` 完成后会建议新增条目
- `/fix-p0` 完成后会要求更新条目状态

### 写入
- **必须人工维护**（Jack 或开发者）
- Agent 可以**建议**条目，但不自动写入（避免污染）
- 提交时纳入 commit：`[docs] 更新 known-issues #<id>`
