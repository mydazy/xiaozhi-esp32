> - 本提示词 = **决定能不能出院**（出货门禁裁决）。不重新大海捞针，而是把已知结论"挤到见底"，

## 〇、为什么要"精选"而不是再跑一次全量

全量审计已经做过，再跑一遍只会复述同样的 152 条，不产生出货决策价值。量产真正缺的是：
1. **去伪**：16 个 P0 里，`量产必修清单.md` 单 agent 一遍已挤到约 8 条，但那只是一遍主观判断——精选评审要派**独立对抗 agent** 复验，防漏报也防"为凑数拔高"。
2. **闭环**：每条不只说"改哪行"，还要给"**怎么测才算真修好**"（正常+异常+边界+压力），否则改了等于没改（Karpathy 铁律 4：定义可验证成功标准）。
3. **裁决**：最终要一张**产品语言的 go/no-go 表**给产品负责人拍板，而不是一堆技术条目。

## 三、每条评审的强制动作（逐条产出，缺一不可）

```
对 <编号> 这一条，按以下五步裁决，每步给证据，不脑补：

1. 真伪复验（对抗视角）
   - 拿真实代码确认：文件:行号 是否真如描述？贴 ≤10 行现场代码。
   - 在量产构建目标(P30-4G)下、真实用户场景里【会不会真触发】？
     给结论：必触发 / 高频条件触发 / 边缘条件触发 / 不触发(误报)，并写触发路径。
   - 反向尝试证伪：找一条"它其实不会发生"的理由；找不到才算坐实。

2. 影响裁定
   - 后果落到产品层：烧硬件 / 砖机 / 假死 / 体验退化 / 仅可诊断性。
   - 影响面：整批 / 部分批次 / 个例；哪种板型；哪个场景。

3. 闭环修复方案（可验证）
   - 改哪个文件哪一行 → 改成什么（给具体代码，不是"建议优化"）。
   - 验证方法四档：正常路径 / 异常路径 / 边界 / 压力——每档写"观察到什么=修好了"。
   - 估改动量 + 回归影响面（会动到哪些功能）。

4. 红线特判（仅 08 电源 / 04 OTA / 涉分区命中时）
   - 标注【需真机实测双确认】，给实测步骤（量什么、预期值、判定阈值）。
   - 这类不接受"代码看着对就行"，必须实测。

5. go/no-go 裁决
   - 是否 ship-blocker：是 / 否 / 有条件(写明条件)。
   - 一句产品语言结论（给产品负责人看，不堆术语）。

不确定标 [待确认] 并说明；禁止编造行号或代码。
```

## 四、多 agent 编排（执行时）

1. **并行复验**：A/B/C/D 四组拆条，派 N 个复验 agent 并行，每个 agent 对抗式只啃 1~2 条，跑满"三、五步"。
   - 红线两条（08 电源、04 OTA）各单独一个 agent，深挖到底。
2. **裁决合表**：全部回报后，派 1 个裁决 agent：
   - 汇总成「精选评审表」+「出货 go/no-go 裁决」
   - 跨条去重、定级对齐、标出真正的 ship-blocker 集合
3. **产品回报**：主控用四句话 + 一张 go/no-go 表回报产品负责人。

2. **出货裁决**：
   - 结论：✅可出货 / ❌不可出货 / ⚠️有条件出货
   - 「出货前必清零」最终清单（预计 ≤8 条）
   - 「可缓解/可后放」清单 + 缓解措施
   - 误报/可降级清单（从 P0 摘除的，附理由）

3. **给产品负责人的四句话**：做了什么 / 卡点是什么 / 要你拍板的取舍 / 下一步排期。

## 六、纪律（与本仓 CLAUDE.md 红线一致）

- 电源域 / OTA / 分区表相关结论，**只评审、只列 diff 清单，不自动改代码**——改这些要 Jack 双确认 + 实测。
- 每条必须有「文件:行号 + 现场代码」证据，无证据的判定一律标 [待确认]。
- 评审本身不改任何代码；这是"决定要不要修、怎么验"的决策环节，修复是确认后的独立动作。
