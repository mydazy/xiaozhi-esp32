# 根因修复深度推演提示词（v1）

> 定位：上一轮评审已**定性坐实**根因（`量产后评审_2026-06-07.md`）；本提示词把每个根因**量化推演到数值级**，再产出可落地断根方案 + 推演修复效果。不停留"看着像 bug"，要"算给你看"。

## 输入
- 已坐实根因（优先 3 个，代码可证、不需现场日志）：过放保护失效 P0、弱网音频发送阻塞 P1、弱网重连阻塞 P1。
- 当前代码 `main/` `components/`（git 原生 Bash 查 blame）。

## 每个根因 · 五步深度推演

1. **量化坐实**：从代码精确推导数值，把"高置信"升级为"数值证明"——
   - 阻塞类：单次操作耗时(ms) × 次数 = 最坏阻塞时长；列出每段 timeout/retry 的来源行号。
   - 电源类：采样周期、阈值(mV)、ADC 抖动幅度、迟滞 N 次设计、关机触发时机。
   - 内存类：分配峰值(KB)、INT-RAM 余量、队列/缓冲上限。
   - 并发类：竞态时间窗口(μs/ms)、共享变量、读写任务。
2. **断根方案**：代码级改法（哪个文件哪段、改成什么结构），说清为什么这是治本不是再加超时。
3. **修复后推演**：改后的关键数值（主循环最大延迟、队列深度、内存增量、关机响应时间），证明方案有效。
4. **回归推演**：改动影响面，是否引入新竞态/新阻塞/新分配/新 UAF；列出受影响的调用方。
5. **红线特判**：命中电源/分区/OTA/并发/内存安全 → 标【plan mode + 真机实测点】，只出方案不直接改。

## 对抗审查
每个方案派独立 agent 找：副作用、新 bug、边界漏洞、是否真断根。证伪不掉才算方案可行。

## 输出（`docs/audit/根因修复方案_2026-06-07.md`）
- 每根因：量化坐实数值 | 断根代码方案 | 修复后推演 | 回归风险 | 验证测试(正常/异常/边界/压力) | 红线?
- 实施顺序：止血小改(可直接) → 红线项(plan mode+实测) → 架构断根(里程碑)
- 给产品负责人四句话

## 纪律
- 量化必须从代码实证，不脑补数值；推不出的标 [待确认 + 需现场实测]。
- 红线（电源/并发/内存安全）只推演+方案，plan mode + 双确认 + 实测后才改。
- 断根 ≠ 再加一层防御。
