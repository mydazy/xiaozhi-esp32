# 量产后稳定性 / 时序根因评审提示词（v1）

> **它和 `固件审计提示词.md`（出货前 v2）的本质区别：**
> - 出货前审计 = 从代码正向找缺陷，预防，全量 10 子系统铺开。
> - 本提示词 = **已量产、现场频发问题** → 从**现象 + 一个月修补史**倒推根因，聚焦稳定性/时序，治本不治标。
> - 一句话：不是再扫一遍代码，而是回答"**为什么修了一个月还反复发，真根因在哪，怎么断根**"。

---

## 〇、为什么不能重跑出货前审计

- 出货前审计基于 1 个月前的代码；这一个月 **202 次提交**，代码已大幅变化，旧的 P0/P1 编号与行号多已失效。
- 量产后问题的性质是"**反复修补仍复发**"——根因往往是时序/并发/内存这类**非确定性**问题，正向扫代码扫不出，必须**现象驱动 + 修补史驱动**。
- 重跑全量只会复述过时结论，浪费且误导。

---

## 一、三路输入（缺一不可，现象是灵魂）

1. **现场症状**（最高优先，Jack/返修/客服提供）：频发的具体现象——重启?死机?触摸失灵?音频杂音/卡顿?4G 掉线?摇一摇误触发?开关机异常?发热?。每条尽量带：**复现场景 + 频率 + 批次 + 有无串口日志/coredump**。
2. **一个月修补史**（`git log --since` 202 条）：**重点找"同一模块反复修"**——反复修 = 没根治。已知热区：I2C/触摸时序、内存省用、音频/唤醒词、异步化补救、关机重启。
3. **当前代码**（`main/` `components/` 为准，不看旧审计行号）。

> 若现场症状暂缺，退化为"按六维度 + 修补热区盲扫高风险点"，但精度大降——**优先催 Jack 给症状**。

## 二、评审维度（6 个，量产后高发，从修补史提炼）

| # | 维度 | 重点（为什么是它） | 路径锚点 |
|---|---|---|---|
| D1 | **内存稳定性** | 一个月大量 perf 省内存=RAM 紧张→OOM/碎片/长稳泄漏致随机崩溃重启 | 全局 `min_free` 谷值、PSRAM/INT 分配、ring/queue 上限、长稳泄漏、`heap_caps` |

## 三、方法：现象 → 根因（每条问题都走）

```
对每个现场症状 / 每个修补热区：
1. 现象固化：复现条件、频率、批次、日志特征（什么时候崩/卡/异常）。
2. 修补史回查：git log/blame 这块这一个月改了几次?都怎么改的?是治标(加超时/重试/异步/降内存)还是治本?
   —— 反复 patch 同一处 = 强烈的"根因未除"信号。
3. 根因假设：基于当前代码提出机制级假设（时序窗口/竞态点/泄漏路径/分配峰值），给文件:行号+代码片段证据。
4. 对抗验证：派独立 agent 试图证伪该假设（"它其实不会发生"），证伪不掉才算坐实根因。
5. 断根方案：治本改法（不是再加一层超时/重试）+ 可验证标准（正常+异常+边界+压力/老化怎么测才算真断根）。
6. 红线特判：命中电源/分区/OTA → 标【plan mode + 真机实测双确认】。
不确定标 [待确认]，禁止编造行号。
```

## 四、Claude Code 新特性编排（复用本仓已验证的多 agent 模式）

1. **Workflow fan-out**：D1~D6 六维度并行，各派 1 个排查 agent（pipeline：排查→根因→对抗验证，不设 barrier）。
2. **对抗式根因验证**：每个根因派 2~3 个 skeptic agent 从不同视角（时序/复现/泄漏量化）尝试证伪，多数证伪则降级。
3. **修补史交叉**：一个 agent 专做 `git log`/`blame` 反复修补热区识别，喂给各维度。
4. **plan mode 红线**：电源/分区/OTA 类断根改动，先 plan mode 出方案+回滚点，Jack 确认+真机实测再改（见 `rules/esp32-memory.md` 红线流程）。
5. **原生 memory 留痕**：每个坐实的根因写 `<模块>-<现象>.md` topic（如 `i2c-touch-timing-race.md`），下次复发可复用，直接服务返修。
6. **synthesis**：去重、按"现场影响×复发频率×断根成本"排优先级，出整改路线图。

## 五、输出（写到 `docs/audit/量产后评审_2026-06-07.md`）

1. **根因表**：现象 | 根因(机制) | 文件:行号 | 反复修补史 | 治标 or 治本 | 断根改法 | 验证方法 | 红线?
2. **治标清单**：这一个月哪些 patch 是治标（加超时/重试/降内存/异步），背后真根因是什么，是否需回补断根。
3. **稳定性整改路线图**：按 现场影响×复发频率×成本 排序；区分"现网热修(快)"vs"架构断根(稳)"。
4. **给产品负责人四句话**：现在最伤口碑的是什么 / 根因 / 断根要动什么(风险) / 排期。

## 六、纪律

- 现场症状优先；无症状的纯代码推断标 [待确认]，不当结论。
- 断根 ≠ 再加一层防御；治标修复要明确标注并回查真根因。
- 电源/分区/OTA 断根属红线，只评审+列方案，plan mode + 实测双确认后才改。
- 评审只读不改代码；修复是确认后的独立动作。
