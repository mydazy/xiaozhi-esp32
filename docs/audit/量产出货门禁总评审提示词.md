## 〇、GOAL（可验证的成功标准 · 评审完成前必须全部回答）

本评审产出一份能让产品负责人直接拍板的裁决，达成标准：

1. **回归零漏判**：本轮改动里**每一处与"已确认根因记忆"冲突的改动**都被识别、定级、给处置建议（撞根因 = 红旗，必须显式裁决，不许默认放行）。
2. **已知 P0 现状清晰**：量产必修 8 条（`量产必修清单.md`）在当前代码里**逐条**给出「已修 / 未修 / 被本轮改动影响」三态结论 + 文件:行号证据。
3. **四条红线状态可判**：电源域 / 内存安全 / 并发 / OTA 四条红线在当前代码各给一句「过 / 不过 / 有条件」结论。
4. **闭环可验证**：每条 ship-blocker 给「怎么测才算真修好」（正常 + 异常 + 边界 + 压力/老化），缺验证方法的不算修好。
5. **裁决唯一**：最终输出 ✅可出货 / ❌不可出货 / ⚠️有条件出货 之一 + 出货前必清零清单（预计 ≤10 条）+ 产品语言四句话。

> 全部 5 条回答完才算评审完成；任何一条标 [待确认] 必须写清"卡在哪、需谁补什么"。

## 三、每条评审的强制动作（缺一不可）

```
对 <每处改动 / 每条 P0>，按五步裁决，每步给证据，不脑补：

1. 真伪/回归复验（对抗视角）
   - 拿真实代码确认：文件:行号 现状如何？贴 ≤10 行现场代码。
   - 若属本轮改动：对照已确认根因记忆，它是否撞回已断根的问题？给"撞/不撞/部分撞"+理由。
   - 在量产构建目标(P30-4G 为主, WiFi 同核)、真实用户场景下【会不会真触发】？必触发/高频/边缘/不触发。
   - 反向证伪：找一条"它其实不会发生/不算回归"的理由，找不到才坐实。
2. 影响裁定：落到产品层——烧硬件/砖机/假死/扰民/体验退化/仅可诊断性；影响面整批/部分/个例；哪板型哪场景。
3. 闭环修复/处置方案：改哪文件哪行→改成什么(具体代码或配置)；验证四档(正常/异常/边界/压力老化)每档写"观察到什么=修好了"；估改动量+回归面。
4. 红线特判(命中 08 电源/休眠唤醒/04 OTA/分区)：标【需真机实测双确认】，给实测步骤(量什么、预期值、判定阈值)。不接受"看着对"。
5. go/no-go：是否 ship-blocker(是/否/有条件+条件)；一句产品语言结论。

不确定标 [待确认] 并写清卡点；禁止编造行号或代码。
```
