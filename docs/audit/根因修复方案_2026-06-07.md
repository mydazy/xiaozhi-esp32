# 根因修复方案（2026-06-07）

> 适用：mydazy-p30-v31（P30-4G / P30-WiFi）。以**当前代码**为准，所有行号/常量均经本轮 grep + Read 实证。
> 红线项（电源 / 并发 / 内存安全）**planModeRequired=true**：本文件只出方案，实施走 plan mode + 真机实测。
> 上一轮坐实见 `docs/audit/量产后评审_2026-06-07.md`；本文件在其基础上吸收方案对抗审查的 refinedFix，补齐缺陷。

## 0. 现场症状 → 三条根因 + 三个止血小改

现场两类故障：①长期放置/久用后电芯损坏批量返修（过放）；②弱网下整机卡死/反复假死/重启。

| 编号 | 根因 | 分级 | 红线 |
|---|---|---|---|
| R1 | 运行期过放保护链路不闭环（开机才检查一次，运行期只弹"请充电"不关机） | P0 | 电源 |
| R2 | 弱网音频上行在主事件循环内同步抽干发送，阻塞整条主循环 | P0 | 并发 |
| R3 | 弱网断线重连在主任务事件循环内同步阻塞执行，冻结主循环数十秒~分钟 | P0 | 并发 |
| S1 | AFE `create_from_config` 返回未判空，PSRAM 不足时空指针解引用 | P1 | 内存安全 |
| S2 | 唤醒词编码静态任务：`encode_in_progress_` 早于 `vTaskDelete` 清零 + 析构裸 free 活动任务栈 | P1 | 并发 |
| S3 | `service_stopped_` 裸 `bool` 跨任务读写（data race / UB） | P1 | 并发 |

S1/S2/S3 是**止血小改**：单点、外科手术式、不改架构，可直接做（仍属并发/内存红线，需走 plan mode 双确认，但改动极小、风险可控）。R1/R2/R3 是**架构断根**，需独立任务 / 生命周期同步设计。

## R1 · 运行期过放保护链路不闭环（P0 · 电源红线）

### 量化坐实（代码实证）

- 定时器周期 1s：`power_manager.h:244` `esp_timer_start_periodic(timer_handle_, 1000000)`，跑在 `ESP_TIMER_TASK`（:239），任务名 `battery_check_timer`（:240）。
- 稳态下**每 60s 才真正重采样一次**：`CheckBatteryStatus`（:116-119）攒满 3 个样本（`kBatteryAdcDataCount=3` :35）后，仅 `ticks_ % kBatteryAdcInterval == 0`（`kBatteryAdcInterval=60` :34）才调 `ReadBatteryAdcData`。
- 关机阈值：`kBatteryShutdownMv=3400`（:37）。`ReadBatteryAdcData` 末尾 :209-214：`voltage < 3400 且非充电 → is_off_battery_=true`，否则 false（即时置位）。
- **唯一消费者只在开机一次性快照**：全仓 grep `IsOffBatteryLevel` 仅 2 处——`mydazy_p30_board.cc(4g):464` / `(wifi):459`，都在 `InitializePowerManager` 开机时读一次。运行期 PowerManager 的回调出口 `OnLowBatteryStatusChanged`（4g:452 / wifi:447）**只弹"请充电"Alert，不关机**。
- 结论：**开机后只要不重启，电压跌破 3400mV 设备永不自动关机 → 持续过放伤电芯**，正是现场批量返修机制根因。
- ADC：12bit（:255）、`ATTEN_DB_12`（:254）、×2 分压（:172/176）。3 点滑均（:134-141），**无离群剔除** → 4G PA 发射地弹毛刺可污染均值。
- Schedule 切主线程链路安全：`application.cc:1255` `xEventGroupSetBits(MAIN_EVENT_SCHEDULE)`（入队加锁见 Schedule 实现），主循环 :398-405 move 出队后解锁再执行回调。esp_timer 任务侧仅 push_back，O(1) 无阻塞。
- 关机流程 `ShutdownOrSleep`（board.cc）：Alert + vTaskDelay(3000) + EnterDeepSleep（AudioService.Stop / ResetProtocol / GracefulShutdownModem 各段 delay）≈ 端到端 8s 量级。

### 断根方案（吸收对抗审查 4 项补丁）

把"置位"和"关机消费"在运行期闭环，跨任务安全切回主线程，并修复 3 个会致新 bug 的缺陷：

**(1) 新增运行期关机回调 + 独立连续确认计数器（不动 is_off_battery_ 即时置位）**
`power_manager.h` 新增：
```cpp
std::function<void()> on_shutdown_request_;
int off_battery_streak_ = 0;
bool off_battery_fired_ = false;
bool ready_ = false;                 // 构造期抑制运行期关机触发
const int kBatteryRecoverMv = 3450;  // 迟滞回滞点
const int kOffBatteryConfirmCount = 3;
void OnShutdownRequest(std::function<void()> cb) { on_shutdown_request_ = cb; ready_ = true; }
void ResetOffBatteryGuard() { off_battery_streak_ = 0; off_battery_fired_ = false; }
```
`ReadBatteryAdcData` 末尾（:209 段）**保留** `is_off_battery_` 即时置位不动（消除开机兜底语义歧义，:464 行为零回归），**新增**独立 streak 状态机：
```cpp
if (is_charging_) {
    off_battery_streak_ = 0;          // 充电清零（缺陷2 重置点之一）
} else if (voltage < kBatteryShutdownMv) {
    off_battery_streak_++;
} else if (voltage >= kBatteryRecoverMv) {
    off_battery_streak_ = 0;          // 迟滞：回到 3450 才清零，防 3400 附近横跳
}
// 仅在 ready_（构造期不触发）、达确认次数、回调非空、未触发过时，触发一次
if (ready_ && off_battery_streak_ >= kOffBatteryConfirmCount
    && on_shutdown_request_ && !off_battery_fired_) {
    off_battery_fired_ = true;
    on_shutdown_request_();           // 回调体内只能 Application::Schedule，不碰 codec/UI
}
```
> **缺陷1（构造期空回调崩溃）根治**：构造函数 :261-267 在 `new PowerManager` 内即跑 `CheckBatteryStatus`，开机即过放时循环 4 次可攒满 streak=3；但 `on_shutdown_request_` 由板级在 new 之后才注入。`CONFIG_COMPILER_CXX_EXCEPTIONS=y` 下调用空 `std::function` 抛 `bad_function_call` → 开机崩溃/砖。`ready_=false`（仅 `OnShutdownRequest` 内置 true）+ 回调判空双保险，构造期绝不触发回调。

**(2) 板级注入运行期关机回调（4g `InitializePowerManager` :449 / wifi :444）**
```cpp
power_manager_->OnShutdownRequest([this]() {
    Application::GetInstance().Schedule([this]() {
        if (!power_manager_->IsCharging()) {                 // 二次确认，防确认窗口内插充电器
            ShutdownOrSleep("电量过低", "强制关机",
                            Lang::Sounds::OGG_LOW_BATTERY, 3000, false);
        }
    });
});
power_manager_->OnChargingStatusChanged([this](bool charging) {
    if (charging) power_manager_->ResetOffBatteryGuard();    // 缺陷2：充放电循环后重置
});
```
> **缺陷2（充放电循环后保护永久哑火）根治**：原方案 off_battery_fired_ 置 true 后无重置路径，经一次充电循环后过放保护永久失效。`OnChargingStatusChanged` 板级当前未注册（接口 :318 存在但全仓未调用），现注册并在充电时 `ResetOffBatteryGuard`；ReadBatteryAdcData 内 `is_charging_` 分支也清 streak，双重保险。
> 开机一次性兜底（:464 `IsOffBatteryLevel()`）**保留不删**，专管"开机即过放"，与运行期路径解耦。

### 修复后推演

- 检测周期仍 60s/次（不动采样频率，避免增加 ADC/4G 干扰耦合）。
- 确认延迟 = 3×60s = 180s；迟滞带宽 50mV（3400~3450）。
- 关机端到端：正常路径 ≈ 3~4 分钟（检测+确认+8s 关机流程）；弱网最坏因 Schedule 排队 +数十秒（受 R2 主循环阻塞拖累，二者应一并修）。
- 过放余量：3400mV 触发距硬件 BMS 截止 ~2800mV 有 ~600mV / 数小时缓冲，180s 确认延迟远小于该窗口 → 电芯不再持续过放。
- 内存增量：2 int + 1 std::function(~32B) + 2 bool < 64B，可忽略。

### 回归风险

- 仅新增运行期关机路径 + 充电重置，**不改采样/电量计算/充电检测/开机兜底/is_off_battery_ 即时置位** → 现有功能零回归。
- 新增并发点：回调在 esp_timer 任务调用，但体内只做 `Application::Schedule`（已线程安全），不直接碰 codec/UI/protocol → 不引入新竞态。
- 残余风险：迟滞+3次确认误关概率 <0.1% 量级 **[需真机标定]**；4G PA 地弹毛刺若实测超 ±50mV，需在 `ReadBatteryAdcData` 加中值滤波/离群剔除（当前仅滑均），可选增强。

### 验证测试（红线 · plan mode + 真机双确认）

- 正常：稳压源从 3.7V 缓降至 3.39V → 确认 streak 累积 180s 后串口打关机日志并真 deep sleep，测端到端时间。
- 异常：降到 3.39V 立即回升 3.46V（>3450）→ streak 清零、不关机；3.41V 在 3400~3450 带内反复横跳 → 不触发（验迟滞）。
- 边界：恰好 3400mV 持续 → 验跨阈；关机倒计时 3s 内插充电器 → Schedule 内二次确认 `IsCharging` 取消关机（验竞争窗口）；**触发关机倒计时内插充电→拔出→再次降压 → 确认能二次触发关机（验缺陷2）**。
- 压力：3.39V + 弱网（4G 差，主循环被 SendAudio 占用）→ 验关机 Schedule 不丢失、最终仍关机；连续上电/断电 ADC 死值 → 验不误关（复用 sc7a20h 死值经验）。
- 串口抓 `Cali Voltage` 日志（:216）坐实电压-码值标定。

**redline=true · planModeRequired=true**

## R2 · 弱网音频上行同步阻塞主循环（P0 · 并发红线）

### 量化坐实（代码实证）

- 主任务优先级 10：`application.cc:319` `vTaskPrioritySet(nullptr, 10)`；唯一事件循环 :336 `xEventGroupWaitBits(...portMAX_DELAY)`，时钟/状态/调度/音频发送共用（`MAIN_EVENT_SEND_AUDIO` 在 ALL_EVENTS :323）。
- 同步抽干：:373 `while (auto packet = audio_service_.PopPacketFromSendQueue())` 逐帧 `protocol_->SendAudio`；:375 `++audio_send_fail_count_ >= 100` 才退出对话。
- `Ml307Tcp::Send`（ml307_tcp.cc:151-214）：每块 `SendCommand(timeout=tx+1500ms)` + 失败 `vTaskDelay(200)` 重试一次 + `xEventGroupWaitBits(SEND_COMPLETE, TCP_CONNECT_TIMEOUT_MS=10000)` 全程同步阻塞。`at_uart.cc:684 try_lock_for(timeout_ms)` + :710 `xEventGroupWaitBits(timeout_ms)` 双重阻塞。
- **最锋利发现（已证实）**：`consecutive_send_failures_` 只在 SendCommand 返回值分支增减；当 SendCommand 成功（清零）、仅卡在末尾 `xEventGroupWaitBits(SEND_COMPLETE)` 超时 `return -1` 时，该计数器**完全不参与** → 13.2s/帧 路径绕过 `kMaxConsecutiveSendFailures=5` 拦截，可连续发生。
- 单帧最坏阻塞 = 1516×2 + 200 + 10000 ≈ **13.2s/帧**；100 帧理论上界 ≈ 22 分钟主循环停摆（喂狗若在主循环则先 TWDT 复位）。最痛区间 = "AT 成功但等不到 SEND_COMPLETE"的 10s/帧 × N。
- 队列上限：`MAX_SEND_PACKETS_IN_QUEUE = 2400/60 = 40 帧`（audio_service.h:45），生产者侧已判 size<40 背压（:367/:521）；队列锁 `audio_queue_mutex_`（:182）同时守 push（:551）/pop（:652/657）→ **锁已就位可复用**。

### 断根方案（吸收对抗审查 4 项补丁 —— 缺陷1 是核心 claim 证伪）

原"只搬 SendAudio 出主循环"**未真正断根**：主任务上还有大量同走 `Ml307Tcp::Send` / 同一把 AT 锁的阻塞发送，仍留主循环。

**(1) 独立上行任务 + 有界队列丢最旧帧**
新建 AudioSender 任务（优先级 5，低于主任务）：`while(true){ packet=BlockingPop(超时); protocol_->SendAudio(packet); }`。删除 application.cc:372-385 `MAIN_EVENT_SEND_AUDIO` 整段；on_send_queue_available（:190-191）改为唤醒该任务。生产者侧（audio_service.cc:521 入队处）队列满改"丢最旧帧（pop_front 后 push_back，**必须在 audio_queue_mutex_ 内完成 RMW**）"，丢帧计数打 metric。

**(2) 补全主循环去阻塞（缺陷1 · 否则"<10ms"是假的）**
主任务上所有走 `Ml307Tcp::Send` 的阻塞发送需统一处理：`SendStartListening`(:1110) / `SendStopListening` / `SendAbortSpeaking` / `SendWakeWordDetected` / `SendTextToTts` / `SendMcpMessage`(注释明写"Always schedule to run in main task") / `Open|CloseAudioChannel`。改为投递到网络发送任务异步执行（主循环只入队不等回执），或至少给这些控制命令也设短超时上限，杜绝单条 AT 在主循环卡 >1 个 tick。

**(3) 音频路径短超时（缺陷6 关联）**
`Ml307Tcp::Send` 给 `Tcp::Send` 增可选 `send_timeout` 参数：音频上行路径等 SEND_COMPLETE 改 500ms（约 8 帧节奏），超时即丢帧 `return -1`；控制连接 Connect/Disconnect 仍用 10s。

**(4) AT 锁可抢占 / 控制命令优先（缺陷2 · 并发新副作用）**
sender 任务弱网可持 AT 锁 ~3.7s（SendCommand×2+200）。此时主任务调 `SendAbortSpeaking`(打断说话) / `CloseAudioChannel`(挂断) 会在 `try_lock_for` 上等到超时才失败 → 打断延迟甚至失败。`SendAbortSpeaking` 是实时交互关键路径，必须：控制类 AT 优先拿锁，或音频帧发送支持被打断信号中断退出。

**(5) protocol_ 生命周期硬同步（缺陷3 · UAF 必做项）**
`protocol_` 是 `unique_ptr`（application.h:158），`reset()` 在 Reboot(:1363)/ResetProtocol(:1558) 主任务调。sender 任务并行调 `protocol_->SendAudio()` 与主任务 reset 是裸 unique_ptr 跨线程读写 = **明确 UAF**。必做：reset 前确定性停 sender 任务对 protocol_ 的访问（任务通知停+确认已停 / 或 protocol_ 改 `shared_ptr` 且 sender 内 `auto p = atomic_load(&protocol_); if(p) p->SendAudio();` 本地持引用，对象延后到用完才析构）。

**(6) 退出策略统一（缺陷5）**
明确：丢帧处理瞬时拥塞（维持连接）；保留"彻底断网/连续 N 秒发不出去"退出对话兜底（可复用 tcp 层 `kMaxConsecutiveSendFailures=5` 触发 Disconnect 上抛），两者条件不重叠，避免"该断不断"。

### 修复后推演

- 主循环单次最大延迟：从最坏 13.2s/帧、累计分钟级 → 纯事件分发 **<10ms**。
- 上行队列：维持 40 帧上界（2400ms 缓冲，约 40×120B≈4.8KB，本就已分配），弱网贴满后丢最旧，**内存增量 0**。
- sender 任务最坏 ~3.7s/帧（仅影响上行丢帧，不卡主循环；注：缺陷4 的 AT 锁争用会反向影响主任务控制命令，故须做 (4)）。
- 用户体验：从"整机卡死/重启"退化为"这段话上行丢几帧、AI 偶尔漏听"，可恢复、不崩。

### 回归风险

- 触并发红线：新增上行消费者 + "队列满丢最旧"RMW，**必须在 audio_queue_mutex_ 内完成**，复用既有锁，不与 OpusCodecTask 入队点(:551) 形成双重持锁顺序问题。
- 丢帧 vs opus 流连续性：websocket v2 带 timestamp，中间丢帧致时间戳跳变，服务端 VAD/AEC 是否容忍 **[需实测]**。
- protocol_ 跨线程 UAF（见 (5)，必做）；AT 锁跨任务争用（见 (4)）；行为变更"弱网维持连接"需产品确认。
- 回归测试覆盖：正常网（不丢帧）、弱网（丢帧不卡死、主循环响应正常）、断网恢复（任务安全停起、protocol_ 切换无崩）、压力（满队列丢帧无 OOM 无锁竞争崩溃）。

### 验证测试（红线 · plan mode + 真机弱网/断网/打断实测）

- 正常：好网对话无丢帧、时序正常、AI 正常应答；监控主循环 tick 稳定。
- 异常：拔天线/进电梯弱网 → 主循环仍按 1s 出 clock tick、UI 不卡、不重启（TWDT 不触发），仅日志报上行丢帧。
- 边界：队列恰 40 帧满持续丢最旧 → 丢帧计数正确、无越界、无 OOM。
- 压力：>10min 弱网持续对话 + 反复断网重连 → 无锁竞争崩溃、无内存增长、protocol_ 切换无野指针；老化观察偶发 crash。
- **打断专项**：弱网持续对话中触发打断 → 验打断在 X ms 内生效（不被 sender 持锁拖延）。

**redline=true · planModeRequired=true**

## R3 · 弱网断线重连同步阻塞主循环（P0 · 并发红线）

### 量化坐实（代码实证）

- `OnAudioChannelClosed`（application.cc:706）Listening/Speaking 分支走 :748 `Schedule(...)`，lambda 含 `for attempt 1..3 { vTaskDelay(attempt*500); OpenAudioChannel(); }`（:748-762）。
- 关键错误：Schedule 投递的 lambda 在**主任务事件循环里同步执行**（:398-405 `MAIN_EVENT_SCHEDULE` 分支 `for(task) task()`）→ 重连整段跑在主任务（优先级 10）上下文。
- 主循环 :337 单一 `xEventGroupWaitBits(portMAX_DELAY)` 串行无抢占 → 重连占用期间 TOGGLE_CHAT/WAKE_WORD/CLOCK_TICK/SEND_AUDIO 全积压。
- 4G 路径 `OpenAudioChannel` 内 `TCP_CONNECT_TIMEOUT_MS=10000`（ml307_tcp.h:17）多段 WaitBits（MIPSTATE/MIPCLOSE/MIPOPEN/WS 握手/hello）串联：单次病态上界 ≈ 50s（各段同时满超时，理论极值，真机多落 20-33s）；弱网典型 ≈ 10s。
- 退避空转 `vTaskDelay(500+1000+1500)=3000ms`。最坏总冻结 = 3000 + 3×50000 ≈ **2分33秒**（病态）；弱网典型 ≈ 33s；握手卡型 ≈ 63s。
- 二次放大：`kMaxReconnectInWindow=3 / kReconnectWindowUs=300s`（application.h:184-185），5 分钟内最多 3 整轮 → 用户感受"反复假死"。
- 订正前提：WS 路径 `OpenAudioChannel`（websocket_protocol.cc:83）确为"单次 Connect + 单次 10s hello"，无 500ms warmup、无内层 3 次 Connect。"app3×协议3=9次"不成立。

### 断根方案（吸收对抗审查 7 项补丁 —— 第1、2条不补=红线倒退）

把执行上下文从主任务剥离是唯一结构性解；调小超时/减 attempt 只缩短卡死时长（治标）。

**(1) 独立可取消重连任务**
新建重连任务（优先级 5，独占栈），`OnAudioChannelClosed` 只做"投递重连请求 + 立即 SetDeviceState 给 UI 反馈"，替换 :748-762 整段。重连状态机在独立任务跑：`Idle → Backoff(attempt, 任务自身 vTaskDelay) → Connecting(OpenAudioChannel) → Success(Schedule 回主任务 SetListeningMode) | Fail(attempt<3 回 Backoff / ≥3 Schedule 回主任务 SetDeviceState Idle)`。每段等待前后检查 `atomic<bool> reconnect_cancel_`。

**(2) protocol_ 生命周期硬同步（缺陷1 · 必须先定结构 · UAF 阻断项）**
现有 :751-753 裸指针判空在新模型下**完全失效**（判空通过后、解引用前另一任务 reset → UAF；reset 后对象已析构，OpenAudioChannel 内还摸 websocket_/event_group_）。建议 `protocol_` 改 `shared_ptr<Protocol>`，重连任务调用 OpenAudioChannel 全程持局部 shared_ptr 拷贝（引用计数保对象在解引用期间不析构），主任务 reset 只释放自己那份，对象延后到重连任务用完才析构。若不引入 shared_ptr 则须一把覆盖"判空→解引用→OpenAudioChannel 全过程"的 mutex 且 reset 同锁——但会让 reset 阻塞在 10s 段上=把卡死搬到关机路径，不可取。**倾向 shared_ptr。**

**(3) 统一销毁入口 join（缺陷2 · Reboot 绕过 Schedule）**
`Reboot()`(:1363) 直接在调用任务 reset protocol_（调用上下文 remote_cmd/mcp/wifi_ap 等非主任务）。`Reboot()` 与 `ResetProtocol()` 都必须先 set cancel + 唤醒重连任务，等重连任务退出到安全点（join 语义）后才 reset/重启；Reboot 现有 `vTaskDelay(1000)` 改为"等重连任务退出信号或超时"。

**(4) 范围补齐（缺陷3 · 同类阻塞点）**
全仓 app 层 `OpenAudioChannel` 共 5 处（application.cc:754/1006/1133/1283/1319）。除 :754(reconnect) 外，:1283(TTS)、:1319(教育卡切 prompt) 属非强用户预期触发，纳入同一异步开通道机制；:1006(ToggleChat)/:1133(StartListening) 用户主动触发可保留同步但需给"连接中"UI 反馈避免误判卡死。文档明确哪些保留同步、为何。

**(5) window 计数归属单一任务（缺陷4 · 裸 int RMW）**
`reconnect_attempts_in_window_`（application.h 裸 int）读改写现全在 `OnAudioChannelClosed`（:717/726/734-740）。移任务后收敛到单一任务（建议全移进重连任务，OnAudioChannelClosed 只投递不算账），或改 atomic+临界区，消除跨任务裸 int RMW。

**(6) SetListeningMode/SetDeviceState 仍 Schedule 回主任务**（状态机单线程，原方案已对）。

**(7) 栈水位标定**：新增重连任务建议 4-6KB 栈，落地前 `uxTaskGetStackHighWaterMark` 标定真实水位（OpenAudioChannel→CreateWebSocket→ml307/at_uart 调用链栈深需实测），否则栈溢出又一类崩溃 **[需实测]**。

### 修复后推演

- 主循环最大单次阻塞：33000~153000ms → ~0ms（主任务只投递+Schedule 回调，<1ms 量级）。
- 触摸/唤醒/状态栏响应延迟：冻结整轮（秒~分钟）→ 正常调度 <50ms。
- 重连本身耗时不变（弱网典型 33s），但后台独立任务执行，用户无感、UI 持续可交互、可随时取消。
- 内存增量：1 个重连任务栈 4-6KB **[需标定]**；window 计数复用现有，几乎零额外堆。

### 回归风险

- 并发红线最大新风险=protocol_ 跨任务 UAF（见 (2)，不补=把"必卡死"换成"偶发 UAF 崩溃"，更贵）。
- Reboot 绕 Schedule 竞态（见 (3)）；window 裸 int RMW（见 (5)）；取消延迟最长滞后一个 10s 段（可接受，需文档说明 + Reboot join 点）。
- 回归测试：断线重连、关机时断线、切 4G/WiFi 时断线、重连中按键、连续弱网抖动触发 window 限流。

### 验证测试（红线 · plan mode + 真机实测）

- 正常：WiFi/4G 正常断线一次 → 后台重连成功；重连期间反复触屏/喊唤醒词全程有响应（对比改前必卡死）。
- 异常：重连中拔网/关机 → 开 heap poisoning + assert，无崩溃、无 UAF，protocol_ 销毁与重连访问无竞态；重连中用户主动按键结束 → 取消生效，状态机回 Idle 不残留。
- 边界：3 次全失败 → 干净回 Idle，UI 清空；5 分钟窗口第 4 次断线 → window 限流不再重连。
- 压力：制造 MIPOPEN/握手/hello 三种超时分支，逻辑分析仪/日志时间戳量主循环 CLOCK_TICK 间隔保持 1s±100ms、触摸到状态变化 <200ms；**重连任务正卡 10s WaitBits 时执行 Reboot/切网，开 heap poisoning+assert 跑 100 次确认无 UAF**；连续断-连 30 分钟老化，`heap_caps_get_minimum_free_size` 无单调下降、`uxTaskGetStackHighWaterMark` 标定栈水位。

## 止血小改（可直接做的外科手术，仍走 plan mode 双确认但改动极小）

### S1 · AFE create 后判 nullptr（内存安全）

`afe_audio_processor.cc:72` `afe_data_ = afe_iface_->create_from_config(afe_config);` 结果未判空即在 :74 起任务并后续解引用。PSRAM 不足（`AFE_MEMORY_ALLOC_MORE_PSRAM` :61）时 `afe_data_` 可能为 null → 处理任务空指针解引用崩溃。
**简方案**：:72 后加
```cpp
if (afe_data_ == nullptr) {
    ESP_LOGE(TAG, "AFE create_from_config failed (PSRAM 不足?), audio processor disabled");
    return;   // 不创建任务、task_created_ 保持 false，析构安全
}
```

### S2 · 唤醒词编码任务 join（并发）

`custom_wake_word.cc`：`encode_in_progress_ = false`（:318）在 `vTaskDelete(NULL)`（:319）**之前**清零 → 存在窗口：新一轮 `EncodeWakeWordData` 通过 :241 的 `exchange(true)` 后，立刻在**同一静态栈/buffer**（:320 复用 `wake_word_encode_task_stack_/buffer_`）上 `xTaskCreateStatic` → 旧任务尚未 vTaskDelete = UB。析构（:30-36）也裸 `heap_caps_free` 这块栈，若编码任务在跑 = free 活动任务栈 UB。
**简方案**：
- 用任务退出信号量/事件位标记"任务真正结束"，`encode_in_progress_` 改为在任务最末（vTaskDelete 前不可清，因清零即开放重入）——正确做法：保留 `encode_in_progress_` 作"逻辑忙"标志，**另加** `task_done_sem_`（参考 afe_audio_processor 析构 :85-87 的 `xSemaphoreTake(2000ms)` 模式），任务体最后 `xSemaphoreGive` 再 `vTaskDelete(NULL)`；下一轮创建任务前 `xSemaphoreTake` 确认上一任务已退出再复用静态栈。
- 析构 :30 free 静态栈前，先 `xSemaphoreTake(task_done_sem_, 2000ms)` 确认编码任务已退出，再 free（与 afe 析构同模式）。

### S3 · service_stopped_ 改 std::atomic（并发 · data race）

`audio_service.h:195` `bool service_stopped_ = true;` 跨任务读写：写在 :135（false）/:179（true）持锁；但读在 :262（无锁，AudioInputTask 主循环判退出）= 裸 bool 并发读 = data race / UB（违反 esp32-memory 红线）。:319/366/830 的读在 cv 谓词内持锁，:262 是唯一裸读。
**简方案**：`bool service_stopped_` → `std::atomic<bool> service_stopped_{true}`。原持锁写/读保持兼容（atomic 兼容锁内访问），:262 裸读变合法。改动一行声明，零行为变更。
