### 详述

**#1 运行期过放保护失效（P0·红线·已坐实）**
- 机制：`CheckBatteryStatus` 每 1s 跑（power_manager.h:244），跌破 3400mV 置 `is_off_battery_=true`（:209-211）；但消费它去关机的代码**只在开机初始化 InitializePowerManager 检查一次**（4g board.cc:464 / wifi board.cc:459）。运行期 `OnLowBatteryStatusChanged` 只弹"请充电"，不关机。
- 后果：开机后长时间使用/放置直到电压跌破 3400mV，设备不自动关机，持续过放伤电芯。比 #5 Vref 偏高更直接的过放路径。
- 断根：`is_off_battery_` 置位时由 PowerManager 经 `Application::Schedule` 切回主线程回调板级 `ShutdownOrSleep`；加迟滞/连续 N 次确认防 ADC 抖动误关。**属电源红线，改动需双确认 + 真机实测。**

**#2 音频发送主任务同步阻塞（P1·已坐实）**
- 机制：`on_send_queue_available` 仅置 MAIN_EVENT_SEND_AUDIO 事件位（application.cc:189-191），主循环 :372-385 同步抽干发送队列，每帧 `protocol_->SendAudio`。4G 下逐 730B 块 `SendCommand(timeout=tx+1500ms)`（ml307_tcp.cc:182），失败再 200ms 重试，外加 :205 `xEventGroupWaitBits(SEND_COMPLETE, 10000ms)`。单块最坏 ~1.6s 到 ~10s；外层容忍连续失败 100 帧（:375）→ 主循环可被网络 I/O 串行占用上百秒。期间 Schedule/触摸/UI/时钟 tick 全部排队 → "点不动/卡死"。
- 断根：音频上行改独立发送任务 + 有界队列，主循环只入队不阻塞；SendAudio 对上层非阻塞，弱网丢帧在发送任务侧限速/丢弃。

**#3 弱网重连主任务嵌套阻塞（P1·已坐实）**
- 机制：OnAudioChannelClosed 弱网重连用 Schedule（=主任务）跑 `for attempt 1..3 { vTaskDelay(attempt*500); OpenAudioChannel(); }`（application.cc:748-762）；而 OpenAudioChannel 自带 500ms warmup + 内层 3 次 Connect（每次 TCP 10s + 握手 10s，web_socket.cc）+ 10s media 等待。app 3 × 协议 3 = 最多 9 次 Connect，主循环最坏冻结数十秒~分钟级，UI 全程冻结无法唤醒/操作。
- 断根：重连移独立任务/状态机，主任务零阻塞；消除 3×3 嵌套，统一一处重试；OpenAudioChannel 改可取消异步。

**#4 AFE 引擎反复重建无兜底（P2·存疑，需现场日志）**
- 坐实部分：`CONFIG_WAKE_WORD_DETECTION_IN_LISTENING is not set`（sdkconfig）→ 每轮对话进 Listening 即 Release()/destroy afe_data_，回 Idle 再 create_from_config，**每轮硬重建一次**。afe_wake_word.cc:100 create_from_config 后**无 nullptr 判断**即 :105 set_wakenet_threshold(null) → 失败即崩；且 audio_service.cc Initialize 返回 true 后不再重建 → 失败需重启恢复。
- 存疑部分："PSRAM 碎片化 → 必然分配失败"缺直接证据，且更可能以崩溃（esp-sr 内部 abort）而非静默挂死暴露。**碎片发生率需现场长稳日志坐实。**
- 断根：create_from_config 后判 nullptr，失败回退常驻不释放并告警；评估 reset_buffer 软复位替代 destroy/create 硬重建。

**#5 未校准 Vref 偏高（P2·存疑·机制已修正，非红线，需真机标定）**
- 坐实部分：power_manager.h:170-177 回退分支 `voltage=3600*1000/4096*average_adc*2`，eFuse 未烧仅 LOGW skip（:87-88），无出厂校准、无"仅供参考"告警。
- 机制修正：3600 不是内部参考电压（~1100mV），而是 ADC_ATTEN_DB_12 的满量程换算常数。ESP32-S3 DB_12 真实满量程约 3100~3300mV，3600 相对其偏高约 **9%**（非 16%），方向为高估。原嫌疑"Vref 1100→3600 致 16% 偏高 + 过放红线"机理错误且危害链被 #1（运行期本就无软件过放关机）截断。
- 断根：未校准时回退常数贴近实测满量程（~3300）+ 加"未校准仅供参考"告警 + 出厂逐台校准写 NVS 系数。**需真机串口实测满电/低电 Cali Voltage 标定后定数值。**

**#6 service_stopped_ 裸 bool data race（P2·并发红线）**
- audio_service.h:195 普通 bool，被 audio_input/output/opus_codec 三任务当退出条件读、Stop() 写，跨任务裸读写 UB；join_task 500ms 超时即报 handle leaked 掩盖退出不可靠。改 std::atomic<bool>，join 超时视为严重错误而非静默放弃。

**#7 编码任务不 join → PSRAM 栈 UAF（P2·内存安全红线）**
- afe_wake_word.cc 析构只 join 检测任务（detection_done_sem_），对 encode_wake_word 任务无 join；Release 注释明说"故意不动编码任务资源"（回避而非解决），随后 heap_caps_free 编码栈。对象在编码任务仍跑时销毁 → 已 free 栈上继续执行 → UAF。custom_wake_word.cc 同款。是 5bd2b9cc/248f3e95"join 后 destroy"补丁唯一漏覆盖的任务。断根：给编码任务加句柄 + done 信号量，Release/析构先 join 再 free。

**#8 codec EnableInput 阻塞结构（P2·技术债）**
- 当前两板 `PowerSaveTimer(120,60,300)`，cpu_max_freq_==120 使 WakeUp 的 `if(cpu_max_freq_<120)` 恒假 → EnableInput 路径为死代码；CalibrateMicOnce/SetInputGain 都走 Schedule 到 app 任务，非 UI 任务。故"24s 卡死在 UI 任务"当前不复现。但 EnableInput 持锁 + 同步 codec open 无超时的结构仍在，改 cpu_max_freq_<120 启用深度降频即复活。断根：codec open/close 加超时 + 失败降级。

### 已被对抗验证证伪 / 降级（不再列为根因，避免误工）

- **AXS read_register 原子 write_read 被回退** → 证伪。I2C 寻址机制决定跨设备插队不会污染 AXS 寄存器指针；唯一读触摸的是单一 LVGL tick 任务，无并发 AXS op。降 P3 技术债（建议恢复原子版仅为驱动一致性）。
- **session_mutex 孤儿锁致 init 序列裸奔** → 证伪。worker 单队列 FIFO 保序，跨设备插队只延迟不重排、不改 SC7A20H 自身寄存器。降 P3（删孤儿锁 + 改误导注释）。
- **shutting_down_ 永不复位致切平台后重连永久失效** → 证伪（非缺陷）。ResetProtocol 仅 reset 不重建，3 个调用点全部紧跟 esp_restart/deep_sleep，进程重启后旗标默认 false，无跨重启载体。
- **双连唤醒 encode_in_progress_ 丢第二段** → 证伪（防御性守卫非缺陷）。打断走 Speaking 分支不调 EncodeWakeWord；单次 Idle 唤醒内编码被同步消费，离开前已清旗标。守卫保护单份静态任务栈，去掉反而内存破坏。真丢上传点在 GetWakeWordOpus 的 2s 超时放弃路径（内存压力下编码慢）。
- **AXS reset_chip 无 ready 探测** → 证伪。create() 已在 reset_chip 返回后读 chip_id ×5 + 100ms 重试做 ready-gate（axs5106l_touch.c:278-296）。降 P3（可读性，闭环点分散两函数）。
- **INT-RAM 持续砍配 + cherry-pick 重复** → 存疑降 P3。三对重复提交是 divergent-branch + merge 历史噪声（git 卫生）；INT-RAM 有 60KB 红线 + 主动回收（BT 释放/AFE 释放/配网后重启）+ system_info.cc 周期 min_free 监控，非"无治本盲砍"。
  ⚠ 订正（2026-06-10 全模块评审）：上行"BT 释放"的回收量实测仅 ~1.5KB，非代码注释声称的 30-50KB（ESP32-S3 的 BLE controller 大头是动态分配，esp_bt_mem_release 只回收静态区；map 符号实测 _bt_*/_bt_controller_* 合计 1480B+ROM 48B）。凡引用"BT 释放"作红线裕量的结论需按 1.5KB 重算；真正的回收主力是 wake_word_pcm_ 清理（~64KB，见 docs/audit/全模块量产评审_RAM排查_2026-06-10.md）。
- **弱网重连 main_tasks_ deque 无上限堆积致内存尖刺/OOM** → 存疑。阻塞侧（D2）坐实应升级；但内存侧（D1）证伪：音频下行/发送走独立队列不经 Schedule，阻塞期 main_tasks_ 至多攒个位数低频 UI 任务，不构成 OOM。deque 无上限按 D3 技术债 P3 记。
- **err_streak bus_reset 风暴** → 存疑降 P2。节流确为下游治标，但 RF 源已在 touch 层处理；codec 掉电钳位根因缺证据。可坐实项仅：bus_worker 未区分 NACK（不该 reset 整条总线）与 bus-hang。

## 三、整改路线（roadmap）

### 现网快修（一两行守卫止血，本周可上）
1. **#4 AFE 重建加 nullptr 兜底**：create_from_config 后判 null，失败回退"不释放、常驻"+告警。防崩，最高性价比。
2. **#1 运行期过放闭环（P0）**：is_off_battery_ 置位 → Schedule 切主线程 → ShutdownOrSleep，加连续 N 次确认。**红线·需双确认 + 真机实测。**
3. **#7 编码任务加 join**：Release/析构先 join 编码任务再 free 栈。防 UAF。
4. **#6 service_stopped_ 改 atomic**：消除 data race。
5. **#5 未校准告警 + 回退常数贴近实测**（数值待真机标定）。

### 架构断根（排下个里程碑）
1. **网络 I/O 全面移出主任务（#2 + #3）**：音频上行独立发送任务 + 有界队列；重连独立任务/状态机；消除 app×协议 3×3 嵌套；OpenAudioChannel 可取消异步。**这是弱网"卡死/点不动"的唯一断根，用户最有感。**
2. **protocol_ 生命周期收敛单一所有权**：所有创建/销毁/访问归主任务，替代散落的 shutting_down_/user_initiated_close_/server_initiated_close_ 旗标为显式状态机。
3. **唤醒引擎生命周期解耦**：评估 reset_buffer 软复位替代每轮 destroy/create；或扩 INT-RAM 预算让唤醒词可常驻，消除 cf0f1672↔0bc897af 翻烧。
4. **codec 慢 I2C 操作加超时 + 异步化（#8）**，并决策 light_sleep 去留（当前被 cpu_max_freq=120 锁成死代码）。
5. **出厂逐台 ADC 校准流程**（写 eFuse 或 NVS 系数）。

### 需现场日志/复现才能坐实的项
- #4 PSRAM 碎片化的实际发生率（长稳多轮对话日志：create_from_config 返回 null / 重建崩溃栈）。
- #5 满电/低电真实 Cali Voltage（真机串口）+ 硬件分压点标定，才能定回退常数。
- #6 / #7 偶发退出超时 / 切板关机瞬间崩溃的现场栈（LoadProhibited / handle leaked）。
