## ① Overview（总览）

把 6 个长跑场景拍平后，全项目的累积型崩溃风险呈现一个清晰结构：

**一个真·必崩单调泄漏**统治全局——`afe_config` 每次对话泄漏。D1「通话期保活唤醒引擎」改造（commit 0bc897af）把原本 init-once 的 `afe_config_init()` 变成**每轮对话 Release→重新 Initialize** 的循环，而全项目从未调用 `afe_config_free()`（ESP-SR 2.1.4 `esp_afe_config.h:284` 明确要求）。每轮进 Speaking 漏一份 afe_config（含 model-name 字符串数组，数百 B~1KB+）。儿童高频场景每天几十~上百次唤醒对话，数天内单调啃光内部 RAM/PSRAM → 对话时内部 RAM 跌破 60KB 红线 → AFE `create_from_config` 失败或 OOM 崩溃。**这是本次推演唯一确定的、量产级、随使用时长单调逼近的崩溃源。** 证据：`afe_wake_word.cc:88`（无配对 free）+ `audio_service.cc:687-712`（每轮 Release 后 `wake_word_initialized_=false` 致必重建）。

**一条架构性放大器**贯穿四个独立场景——`main_tasks_` 无上限 + 主循环同步阻塞。`Application::Schedule` 向 `std::deque main_tasks_`（`application.h:157`）裸 `push_back`，**无任何 size 上界**，直接违反项目内存红线「所有容器必须有上限」。同时弱网重连 lambda 在主事件循环里同步跑 `vTaskDelay(500+1000+1500)=3s`（`application.cc:751-765, 402-404`），TLS 卡死时切歌可顶格阻塞 22s。**阻塞 × 无界队列 = 生产持续、消费冻结**，弱网/快速切歌现场可让队列分钟级堆到内部 RAM 耗尽，极端下 TWDT 复位。它本身不是泄漏，但把所有「正常会自限」的队列变成「会击穿」的队列。

**一批竞态/UAF 临界**随运行时长概率累积——R2/R3 未做的 41 处 `protocol_->` 裸访问（`OnAudioChannelClosed` 跑在协议任务上下文是真并发）、S2 编码任务裸用 `this_`、`EspSsl::Disconnect` 的 `ESP_ERROR_CHECK` 必崩点、AfeWakeWord Release 2s join 超时后强行 destroy。这些不是单调内存增长，而是「窗口随重连/切网/Reboot 次数线性增多」，量产规模（成千上万台 × 长时运行）下从「偶发」变「必有几台命中」。

**好消息**（穷尽排除、确认不累积，避免误修）：Mp3Player 三 ringbuffer 成对 delete+置空、task 用 active_tasks_ join；所有 esp_timer（pause/audio_power/reconnect/idle_dc/wifi reconnect）均懒创建复用或回调内无条件自删；LVGL 对象全 create-once 守卫；ReplaceEmoji delete 旧 image；教学卡 GIF 下载各分支配对 free + in-flight 去重；唤醒静态任务 stack/buffer 复用 + vTaskDelete(NULL) 自删。深睡唤醒走 `esp_deep_sleep_start` 整链冷启动，heap 类泄漏**不跨循环累积**（最大减压阀）。`timestamp_queue_` 无界 push 被 `CONFIG_USE_SERVER_AEC is not set`（sdkconfig:658）编译掉，当前不触发。

## ② TopCrashRisks（按 累积速度 × 触发频率 × 严重度 排序）

排序口径：单调 > 条件累积 > 概率窗口；高频日常 > 弱网/切歌 > 切网/OTA/Reboot；必崩 > 高 > 中。

**T1【必崩·单调·最高频】afe_config 每轮对话泄漏**
- 累积速度：单调，每轮对话漏 1 份（数百 B~1KB+）；触发频率：最高（每次唤醒对话必经 Speaking 重建唤醒引擎）；严重度：必崩。
- 链路：`audio_service.cc:688` re-Initialize → `afe_wake_word.cc:88` afe_config_init → `:100` create_from_config 仅拷贝不接管 → 函数返回指针丢失，全项目无 `afe_config_free`。
- 量级：每天几十~上百次对话 → 每天漏数十~上百 KB；几百次对话即逼近耗尽，对话态先破 60KB 红线。
- 修向：`create_from_config` 之后立即 `afe_config_free(afe_config)`；`afe_audio_processor.cc:42` 同补。验证：连续 N 次 Enable(false)/(true)，`heap_caps_get_free_size(MALLOC_CAP_INTERNAL)` 应平稳不单调降。

**T2【高·条件累积·全局放大器】main_tasks_ 无界 + 主循环同步阻塞**
- 累积速度：条件触发但一旦触发即快速堆积；触发频率：弱网断连/快速切歌/番茄钟 1Hz tick 期间；严重度：高（OOM / UI 冻结 / TWDT）。
- 链路：`application.cc:1260-1266` 无界 push_back + `:398-405` 单批 drain + `:751-765` 重连 lambda 内 3s vTaskDelay（+OpenAudioChannel 10s 级阻塞，`websocket_protocol.cc:205`）+ `mp3_player.cc:92-102` 切歌 22s 顶格阻塞。
- 量级：阻塞窗口内服务器持续下推 tts/stt/json + 断连双路径（`application.cc:454` + `websocket_protocol.cc:184`）反复 Schedule，分钟级堆数十~上百~上千 closure（各含 std::string 拷贝）。
- 修向：① main_tasks_ 加软上限（256/512）超限丢最旧/拒绝+WARN 计数；② 重连退避移出主循环（esp_timer 或独立低优先级任务，禁止 Schedule 回调里 vTaskDelay）；③ 切歌 AbortAndJoin 移出主任务（deferred-cleanup）。

**T3【中·概率窗口·随重连线性增】R2/R3 未做的 protocol_ 并发裸访问 + EspSsl 必崩点**
- 累积速度：非内存累积，竞态窗口随弱网重连/切网次数线性增多；触发频率：弱网现场高；严重度：中（偶发崩溃返修）。
- 链路：`OnAudioChannelClosed` 跑在协议任务上下文，`application.cc:754 if(!protocol_)` 与 `:757 protocol_->OpenAudioChannel()` 之间若另一上下文 atomic_store 置空 → 非原子读旧值；`esp_ssl.cc:88` `ESP_ERROR_CHECK(esp_tls_get_conn_sockfd)` 弱网坏句柄直接 abort()。
- 修向：落实 R2/R3，运行期 `auto p = std::atomic_load(&protocol_); if(p) p->...`；`esp_ssl.cc:88` 改软判断不 abort。

**T4【中·概率窗口·随切网/OTA/Reboot 增】S2 唤醒编码 UAF + AfeWakeWord Release join 超时**
- 链路：`afe_wake_word.cc:255-330` encode 任务全程裸用 `this_`，析构（`:21-48`）不等 encode 完成 → 切协议/Reboot 致对象析构时 this_ 悬空；`:142-148` Release 2s join 超时仍继续 destroy afe_data_ → 旧任务可能访问已 destroy 的 afe_data_（AFE fetch 卡 >2s 时命中）。
- 修向：析构前 join encode 任务或持 shared_ptr 守护；Release join 超时视为致命重启，或超时则放弃本轮 destroy 保留旧实例。

**T5【中·条件累积·反复切协议时】afe_audio_processor 句柄覆盖泄漏**
- 链路：`audio_service.cc:782-787` SetFrameDuration running 时 Stop 后再 Initialize，`afe_audio_processor.cc:72` `afe_data_=create` 直接覆盖旧指针不 destroy + `:42` afe_config 不 free + `:78` 无守卫 xTaskCreate 起第二个常驻 audio_communication 任务。每次漏 1 afe_config + 1 afe_data_（数十~上百 KB）+ 1 task(~4KB 栈)。当前单协议仅触发一次（低危），加协议切换即变高危。
- 修向：Initialize 入口 `if(task_created_)` 先 deinit（PROCESSOR_EXIT+等 sem+destroy）；补 afe_config_free。

**T6【中·条件累积·SERVER_AEC 开启时】timestamp_queue_ 无界 push（当前编译掉）**
- `audio_service.cc:341-347` `#if CONFIG_USE_SERVER_AEC` push 无上限，pop 仅上行编码（`:618-624`）；纯播放只 push 不 pop 单调增。`sdkconfig:658` 当前未开启 → 不触发，**启用前必须修**。

## ③ ByResource（按资源类型归类所有泄漏/累积点）

**heap（内部 RAM）**
- 【必崩·单调】afe_config 每轮泄漏 `afe_wake_word.cc:88`（无 free）。
- 【中·条件】afe_audio_processor afe_config 不 free `afe_audio_processor.cc:42`。
- 【中·条件】realtime(D1) Speaking 期 AEC+WakeNet 内部 RAM 峰值叠加 ~6-10KB（非泄漏，碎片+峰值）`application.cc:1241-1244` 自承「须真机复核」未做。
- 【低·概率】MQTT `aes_ctx_` 每次 hello mbedtls_aes_init 无配对 free `mqtt_protocol.cc:381`（软件 mbedtls 几乎无害，硬件 ALT 累积硬件资源）。

**PSRAM**
- 【中·条件】afe_audio_processor afe_data_ 句柄覆盖泄漏（含 AEC/NS 模型缓冲，数十 KB）`afe_audio_processor.cc:72`。
- 【中·碎片化】Mp3Player DecodeLoop 每帧 new/free 2-3 个 std::vector<int16_t>（>2KB 落 PSRAM），~38 次/秒，连播数小时 churn → 碎片化 → 下一首 512KB compressed_ring 分配失败 `mp3_player.cc:607-643, 123-130`，`sdkconfig:1703-1705`。

**task（任务 + 栈）**
- 【中·概率】AfeWakeWord audio_detection 任务（6144 内部栈）Release 2s join 超时仍 destroy → 任务+栈泄漏 + UAF `afe_wake_word.cc:110-114, 142-148`。
- 【中·条件】afe_audio_processor 无守卫 xTaskCreate audio_communication（4096 栈）重复 init 起第二个常驻任务 `afe_audio_processor.cc:78`。
- 【低·概率】唤醒编码静态任务 encode_in_progress_=false 与 vTaskDelete(NULL) 之间�narrow窗口，新唤醒抢同一静态 TCB/stack `custom_wake_word.cc:241,318-320` / `afe_wake_word.cc:255-330`（S2 相邻面）。

**timer（esp_timer / FreeRTOS timer）**
- 经核查**全部无泄漏**（穷尽排除）：pause_timeout/audio_power/ring/timeout/pomodoro 懒创建复用；idle_dc `websocket_baidu_protocol.cc:1520` 回调无条件自删；wifi reconnect `wifi_station.cc:606,674-678` nullptr 守卫 + 连上即 delete。
- 【低·竞态】wake_chat_timer start_once 重入窄竞态（旧回调已排队时 stop 撤不回）`mydazy_p30_board.cc:1026-1043`（误 toggle，非泄漏）。

**semaphore / EventGroup**
- 无单调泄漏点。AfeWakeWord Release 用 2s 信号量 join（超时处理见 task 段）。

**socket / TLS 句柄**
- 【低·条件】EspSsl 被动断连只置 connected_=false 不 destroy tls_client_，句柄+fd 滞留到对象析构 `esp_ssl.cc:148-153` + `web_socket.cc:48-55`（~WebSocket connected_=false 跳过 Disconnect）；重连 lambda 被主循环冻结时滞留窗口拉长，接近 LWIP MAX_SOCKETS / mbedtls 会话上限时新连接失败。
- 【低·待查】ReportStatus（省电模式每次息屏调，同进程反复）HTTP 句柄释放/失败路径未核 `mydazy_p30_board.cc:496-497`。

**句柄（AES / AFE / 硬件会话）**
- MQTT aes_ctx_（见 heap 段，硬件 ALT 下属句柄类）。

**queue（容器无上限 / 残留）**
- 【架构红线】main_tasks_ 无界 std::deque `application.h:157` / `application.cc:1260-1266`（4 场景共同放大器）。
- 【中·条件·编译掉】timestamp_queue_ 无界 push `audio_service.cc:341-347`（SERVER_AEC 未开）。
- 【低·残留】wake_word_opus_ deque 被 ContinueWakeWordInvoke 提前 return 跳过排空 `application.cc:1133-1137`（下次 clear 兜底，非单调）。

## ④ MissedScenarios（完整性 critic · 冰山法则）

6 场景覆盖了：对话循环 / 音频长播 / 网络弱网 / 电源休眠 / 外设定时器 / 资源生命周期。**以下交互场景与资源类型未被推演到，按疑点优先级列出（宁可多列）：**

1. **OTA 升级长跑**（最大盲区）：未推演反复检查更新 / 下载分片 / 校验 / 回滚的累积。OTA 任务栈、下载缓冲、`esp_https_ota` 句柄在「检查→失败→重试」循环下是否泄漏？OTA 期间 main_tasks_ 是否被 OTA 进度回调灌爆？回滚保护 + Secure Boot（红线 OTA 安全）未在长跑维度核查。**建议追加专项推演。**

2. **远程命令 remote_cmd.cc**：仅在音乐场景顺带提到 `remote_cmd.cc:385` Play 在 Schedule 上下文。服务器反复下推 remote command（音量/亮度/播放/自定义）数百次，命令解析缓冲、JSON cJSON 句柄、命令队列是否有上限与配对释放？未独立推演。

3. **配网 / BluFi / BLE**：`docs/audit/05_wifi_provision.md`、`06_blufi_ble.md` 子系统未进本次长跑推演。反复进出配网模式、BLE 连接/断开数百次的 GATT 句柄、广播缓冲、配网超时 timer 累积未核。

4. **MCP 工具反复调用**：教育卡 show_stroke 顺带覆盖了一部分，但 MCP server（`mcp_server.cc`）注册的全部工具（设备控制、查询、自定义）被 LLM 高频反复调用时，工具调用上下文、返回 JSON 缓冲、`mcp_server.cc:215` Schedule 是否累积，未系统推演。

5. **相机 / NFC**：红线明确点名「相机、NFC 外部输入先校验再用」。本批 6 场景**完全未触及**相机帧缓冲反复申请释放、NFC 读卡循环的句柄/缓冲累积。若本板型有相机/NFC，是空白区。

6. **闹钟/番茄钟跨天累积**：AlarmManager FlushNvs 每圈写 NVS（电源场景提到 NVS 磨损），但「数十个闹钟 × 每天反复响铃」的 alarm items_ 容器、NVS 写频次、AlarmRinger saved_volume_ 漂移（C2 未闭环）未做跨天定量。

7. **日志 / 上报通道**：ReportStatus（已标待查）、可能的 metric 上报、日志缓冲在长跑下是否有无界堆积，未核 HTTP 客户端生命周期。

8. **时间/时区/NTP 二阶效应**：电源场景发现 NowRtcUs 用墙钟受 NTP 跳变影响死值窗口失效。同类「依赖墙钟做差值」的逻辑（闹钟触发判断、超时判断、限流窗口 `application.cc:736-749` 跨窗口抖动绕过）可能在别处也有，未全局扫描墙钟依赖。

9. **multi-codec / 协议切换**（百度 ↔ websocket ↔ mqtt）：T5 提到 SetFrameDuration 重入泄漏，但完整的「运行中切协议」路径（protocol_ 重建 + afe 重 init + decoder 重置）的资源配对未端到端推演。

10. **看门狗 / 任务优先级反转**：主循环 prio10 长占（T2）已点出 TWDT 风险，但未系统核查各任务优先级在长跑负载下是否存在优先级反转致低优先级清理任务饿死（间接泄漏）。

## ⑤ Monitoring（长跑监测埋点建议）

**核心三指标（最小集，先埋这三个就能抓住 T1/T2）：**

1. **内部 RAM 最小水位**（抓 T1 afe_config 单调泄漏）：
   - 埋点：`audio_service.cc` 每轮对话进/出 Speaking 时打 `heap_caps_get_minimum_free_size(MALLOC_CAP_INTERNAL)` 与 `heap_caps_get_free_size(MALLOC_CAP_INTERNAL)`，带对话轮次计数。
   - 判据：连续 N 轮 min_free 单调下降 = afe_config 泄漏复现；稳态平台 = 已修。对话态低于 60KB 红线即告警。

2. **main_tasks_ 队列深度**（抓 T2 队列溢出）：
   - 埋点：`application.cc:1263` Schedule push 后记录 `main_tasks_.size()`，主循环 drain 前后记录峰值；超阈值（如 64/128）打 WARN + 计数。
   - 配套：主循环单次 drain 耗时（`application.cc:398-405` 前后 esp_timer_get_time 差），>1s 告警（定位 vTaskDelay/OpenAudioChannel 阻塞源）。

3. **PSRAM 最大空闲块**（抓 T5/Mp3 碎片化）：
   - 埋点：每首歌起播前记 `heap_caps_get_largest_free_block(MALLOC_CAP_SPIRAM)`；连播场景定期采样。低于 512KB（compressed_ring 需求）告警。

**任务栈水位（抓 task 泄漏/逼近溢出）：**
- 在常驻任务（main loop / audio_communication / audio_detection / Mp3 三 task / 协议接收任务）周期打 `uxTaskGetStackHighWaterMark(NULL)`，趋势下降 = 接近溢出。重点盯 audio_detection（6144）与 audio_communication（4096）在 D1 共存期。

**句柄/任务计数（抓 T4/T5 重复创建 + socket 滞留）：**
- AFE：在 AfeWakeWord/AfeAudioProcessor Initialize/Release 处自增/自减一个 `afe_instance_count`，恒等性破坏即泄漏。
- task：`uxTaskGetNumberOfTasks()` 周期采样，单调增 = 任务泄漏（定位 T4/T5）。
- socket：弱网期周期采样已用 LWIP socket 数（或 `lwip_stats`），接近 MAX_SOCKETS 告警（T3/EspSsl 滞留）。

**埋点落地建议**：复用现有 `Application::Run` 主循环的 CLOCK_TICK 分支挂一个低频（如 30s）监测回调，统一吐上述指标到日志/上报，避免散落。真机连续跑 ≥ 数百轮对话 + ≥ 数小时连播 + 弱网注入，观察单调趋势。

## ⑥ FourLines（四句话回报）

1. **做了什么**：把 6 个长跑场景拍平汇总成一份「反复用会不会越用越崩」的体检报告，标了每个隐患在代码哪一行。
2. **影响什么**：发现 1 个会随使用天数稳步逼近崩溃的真问题（每次对话漏一点内存，孩子高频用几天就可能对话崩），外加一条放大所有弱网/切歌卡顿的架构缺口（任务队列没设上限 + 主循环被卡几秒）；其余多是弱网/切网次数多了才偶发的概率问题。
3. **要不要你拍板**：建议优先级——先修「每次对话漏内存」（一行 free，必修，性价比最高），再给「任务队列加上限 + 重连不卡主循环」（防弱网现场假死/崩溃）。这两条修完长跑稳定性上一个台阶，是否本版就做请你定。
4. **下一步**：OTA/远程命令/配网/相机/NFC 这 5 类长跑场景本次没覆盖（尤其 OTA 反复升级是空白区），建议下一轮补推演；同时按报告⑤埋 3 个监测点，真机连续跑数百轮 + 弱网注入验证趋势。
