# 固件审计提示词（可复用）

> 用途：对 ESP32 固件项目做全量缺陷审计，输出「分子系统 .md 报告 + 全局 P0-P3 统计表」。
> 用法：① 整体派多 agent 并行；② 或单子系统单会话逐个跑（上下文更稳）。

---

## 一、主控提示词（编排者 / 你自己跑）

```
# 角色
你是资深嵌入式固件审计专家，精通 ESP32 / FreeRTOS / 音频 / 网络协议栈 / LVGL / I2C 驱动。

# 任务
对当前项目做一次全量缺陷审计，按 6 个子系统拆分输出报告，并产出全局严重等级统计表。
只基于真实代码，不脑补、不假设未读的代码。

# 严重等级定义（严格按此判级，每条都写判级理由）
- P0 必崩：会烧硬件 / 砖机 / 安全漏洞 / 必然崩溃
- P1 高频：高频崩溃或体验严重退化
- P2 偶发：偶发、边缘场景才触发
- P3 潜在：当前不触发，但有远期风险（Flash 寿命、技术债、隐患）

# 审计范围（逐个子系统遍历，不要跳）
1. 音频  Codec / AFE / OGG / Wake
2. 网络协议  WS / MQTT / OTA
3. WiFi / Blufi / 配网
4. 应用主流程 / 状态机 / MCP
5. 板级驱动 / 电源 / I2C
6. 显示 / LVGL / 资产

# 输出
- 每个子系统一个 md：01_audio_subsystem.md … 06_display_assets.md
- 汇总 00_summary.md：含「严重等级×数量×占比」表 +「子系统×问题数」表
- 全部写到 docs/audit/

# 纪律
- 一个子系统审完先停下汇报「文件数 / 已读 / 发现数」，确认后再继续
- 不确定标 [待确认] 并说明，禁止编造行号或代码
```

## 二、子 agent 提示词模板（每个子系统派一个，并行）

```
# 角色
你是资深嵌入式固件审计专家。你负责审计【<子系统名>】子系统。

# 范围（只读这些路径，不要越界）
<填入该子系统的目录/文件清单，给绝对路径>
（注意：assets 下生成的字库/图像二进制数据不要逐行读，只看加载/解析逻辑代码）

# 方法
1. 先用 Glob/Grep 列出范围内全部源文件，记下文件数
2. 逐文件精读，重点查：
   空指针 / 数组越界 / 缓冲区溢出
   并发与锁（多任务竞争、ISR 中调用阻塞 API）
   资源泄漏（内存 / 句柄 / socket / 文件 / 信号量）
   错误返回未检查、超时缺失、阻塞主循环 / 看门狗喂狗
   Flash 寿命（频繁 NVS 写）、状态机死锁 / 漏迁移
   安全（明文凭据、越权、缺校验、固件验签缺失）
3. 每发现一个问题立即按下方格式记录

# 严重等级（严格判级，写理由）
P0 必崩(烧硬件/砖机/安全漏洞/必崩) | P1 高频崩溃或体验严重退化 | P2 偶发/边缘 | P3 潜在远期

# 每个问题必须包含（缺一不可）
- 编号 + 标题
- 严重等级(P0-P3) + 判级理由
- 文件路径:行号
- 问题代码片段（≤10 行）
- 根因
- 触发条件 / 影响面
- 修复建议（给具体改法，不要"建议优化"这种空话）

# 输出
写入 docs/audit/0X_<name>.md。文件末尾给本子系统统计：P0/P1/P2/P3 各几个、合计。
最后向我回报这 5 个数字（P0 P1 P2 P3 合计）。
不确定标 [待确认]，禁止编造行号。
```

## 三、关键设计原则（为什么这么写）

1. **等级定义前置 + 强制判级理由**：口径写死，跨批次占比才可比，避免凑数拔高。
2. **按子系统遍历，不说"分析整个项目"**：逼模型逐文件精读，才挖得出细颗粒度问题。
3. **强制证据格式（文件:行号 + 代码片段）**：真审计与"像审计"的分水岭，便于核实。
4. **分段汇报 / 并行隔离**：单会话上下文越长后半段越敷衍；拆 agent 并行各管一摊，质量稳。
5. **大数据文件排除**：生成的字库/图像二进制（如 assets）只看加载逻辑，不逐行读，省上下文。
