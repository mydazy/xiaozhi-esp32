# 固件审计提示词（可复用 · v2 · 三遍评审 + 10 子系统）

> 用途：对 ESP32 固件项目做全量缺陷审计，输出「分子系统 .md 报告 + 全局 P0-P3 统计表」。
> v2 相比 v1 的三个变化：
> ① 子系统从 6 个细拆为 **10 个**，**电源域单独成块**（CLAUDE.md 头号风险区，上轮 4 个 P0 全在这）。
> ② 每个子系统强制 **三遍评审**（广度遍历 → 红线深挖 → 反审自检），不是读一遍就交。
> ③ 新增 **全局复核 agent**：跨子系统去重、严重等级校准、漏报反查。
> 用法：先派 10 个子 agent 并行（各跑三遍）→ 全部回报后派 1 个复核 agent → 主控合表。

---

## 一、主控提示词（编排者 / 你自己跑）

```
# 角色
你是资深嵌入式固件审计专家，精通 ESP32-S3 / FreeRTOS / 音频(Codec/AFE) / 网络协议栈(WS/MQTT/OTA) / WiFi+BLE 配网 / LVGL / I2C 驱动 / 电源管理。

# 任务
对当前项目做一次全量缺陷审计。按 10 个子系统拆分并行审计，每个子系统必须跑满三遍评审，
最后产出全局严重等级统计表。只基于真实代码，不脑补、不假设未读的代码。

# 严重等级定义（严格按此判级，每条都写判级理由）
- P0 必崩：会烧硬件 / 砖机 / 安全漏洞 / 必然崩溃 —— 出货前必须清零
- P1 高频：高频崩溃或体验严重退化 —— 量产前清零或有明确缓解
- P2 偶发：偶发、边缘场景才触发 —— 排期修
- P3 潜在：当前不触发，但有远期风险（Flash 寿命、技术债、隐患）

# 四条硬红线（命中任一，判级从严，且必须实测复核）
1. 电源域：电池 ADC / 充电 / 过放 / 休眠 / 分压系数 —— 误配置烧硬件或开机即关机
2. 内存安全：OTA 资产 / 相机 / 网络帧 / NFC 等外部输入，先校验长度边界再用，禁止裸 memcpy / 裸索引
3. 并发：多任务共享热路径数据必须加锁或 atomic；ISR 中禁止阻塞 API
4. OTA 安全：固件验签 + 回滚保护 + Secure Boot，不能裸刷

# 执行编排
1. 按「附录 A · 10 子系统分配表」派 10 个子 agent 并行，各自只读分配到的路径。
2. 每个子 agent 必须跑满「附录 B · 三遍评审」，缺一遍视为未完成、打回重跑。
3. 10 份子报告齐了，派 1 个复核 agent 跑「附录 C · 全局复核提示词」。
4. 复核通过后，你汇总 00_summary.md。

# 输出
- 每个子系统一个 md：01_audio_driver.md … 10_display_assets.md（命名见附录 A）
- 复核报告：99_review.md（去重 / 改判 / 漏报清单）
- 汇总 00_summary.md：含「严重等级×数量×占比」表 +「子系统×P0/P1/P2/P3」矩阵表 + Top P0 清单（含文件:行号）
- 全部写到 docs/audit/

# 纪律
- 一个子系统每跑完一遍先停下汇报「文件数 / 已读 / 本遍新增发现数」，确认后再进入下一遍
- 不确定标 [待确认] 并说明，禁止编造行号或代码
- 同一问题在多份报告重复出现的，由复核 agent 归并，主控不重复计数
```

## 二、子 agent 提示词模板（每个子系统派一个，并行 · 内含三遍评审）

```
# 角色
你是资深嵌入式固件审计专家。你负责审计【<子系统名>】子系统，并对它负全责。

# 范围（只读这些路径，不要越界；按「关注点」过滤，不是整文件逐行）
<填入该子系统的目录/文件清单，给绝对路径——见附录 A>
注意：
- assets 下生成的字库 / 图像 / 固件二进制（如 *_png.c、font 数据）不要逐行读，只看加载 / 解析逻辑代码。
- 若某大文件被多个子系统共享（如 board.cc 同时含电源与外设），只审与你子系统关注点相关的切片，
  用 Grep 锚点定位（见分配表里给的关键字），其余交给对应子系统，不重复审。

# 你必须跑满三遍，每遍目标不同，不能合并、不能跳（详见附录 B）
- 第一遍 · 广度遍历：Glob/Grep 列全文件并记文件数，逐文件精读，抓显性缺陷（空指针/越界/泄漏/未检错误返回/阻塞主循环或看门狗）。
- 第二遍 · 红线深挖：只盯四条硬红线 + 跨文件数据流。追每一个外部输入从入口到使用的全链路；
  对共享数据查锁；对电源/OTA 路径逐行核参数。第一遍漏的细颗粒问题在这遍补。
- 第三遍 · 反审自检：逐条复验自己第一二遍的发现——行号对不对、代码片段是不是真的、判级有没有拔高或压低；
  再用「假如我要黑掉这个子系统/让它崩」的对抗视角，在最高风险文件里反查漏报。

# 严重等级（严格判级，写理由）
P0 必崩(烧硬件/砖机/安全漏洞/必崩) | P1 高频崩溃或体验严重退化 | P2 偶发/边缘 | P3 潜在远期

# 每个问题必须包含（缺一不可）
- 编号 + 标题（编号带子系统前缀，如 08-P0-A）
- 严重等级(P0-P3) + 判级理由（说清为什么是这级，不要只贴结论）
- 文件路径:行号
- 问题代码片段（≤10 行）
- 根因
- 触发条件 / 影响面（哪个场景、哪种板型、整批还是个例）
- 修复建议（给具体改法：改哪行、改成什么；不要"建议优化"这种空话）
- [发现于第几遍]（标 P1/P2/P3 遍，便于评估三遍各自产出）

# 输出
写入 docs/audit/0X_<name>.md。文件分三段，对应三遍各自的发现；末尾给本子系统统计：
P0/P1/P2/P3 各几个、合计，并标注「三遍分别新增了几个」。
最后向我回报这组数字（P0 P1 P2 P3 合计 / 三遍新增 a+b+c）。
不确定标 [待确认]，禁止编造行号。
```

## 四、附录 B · 三遍评审定义（贴进每个子 agent，逐遍交付）

| 遍次 | 名称 | 目标 | 手法 | 交付物 |
|---|---|---|---|---|
| 第一遍 | 广度遍历 | 不漏文件，抓显性缺陷 | Glob/Grep 列全文件→逐文件精读；查空指针/数组越界/缓冲区溢出/资源泄漏(内存/句柄/socket/信号量)/错误返回未检/超时缺失/阻塞主循环或看门狗 | 文件清单 + 第一批问题 |
| 第二遍 | 红线深挖 | 挖第一遍读不出的深层问题 | 只盯四条硬红线 + 跨文件数据流：追每个外部输入入口→使用的全链路；共享数据查锁；电源/OTA 路径逐行核参数；状态机画迁移图找死锁/漏迁移；NVS 写频率算 Flash 寿命 | 第二批问题（多为 P0/P1） |
| 第三遍 | 反审自检 | 防自己造假 + 反查漏报 | 逐条复验第一二遍发现：行号是否真实、代码片段是否对得上、判级是否拔高/压低；再以"如何让它崩/如何攻击它"对抗视角扫最高风险文件 | 改判清单 + 漏报补充 + 误报删除 |

铁律：每遍跑完先停、汇报「文件数 / 已读 / 本遍新增」，确认后再进下一遍。三遍都跑完才算这个子系统交付。

## 五、附录 C · 全局复核 agent 提示词（10 份子报告齐了之后派 1 个）

```
# 角色
你是固件审计的总复核。10 个子系统报告已生成在 docs/audit/01_*.md ~ 10_*.md。
你不重新逐行审代码，你审「这些报告本身可不可信、全不全、判级齐不齐」。

# 任务
1. 去重：跨子系统重复或同源的问题归并为一条，标注原始出处，给出唯一编号。
2. 判级校准：用统一口径复核每条 P0/P1 的判级理由；明显拔高或压低的，给出改判建议 + 理由。
   重点抽查所有 P0：逐条打开对应文件:行号，确认代码真实存在且确实命中红线。
3. 漏报反查：针对四条硬红线，跨报告检查有没有"本该有但没人报"的盲区
   （例：电源域报了分压但没人查充电截止；OTA 报了验签但没人查回滚）。
4. 边界核对：检查子系统交界处有没有"两边都以为对方审了"的真空地带
   （如 board.cc 的电源切片 vs 外设切片、audio_service 跨 01/02 的衔接）。

# 输出 docs/audit/99_review.md
- 归并表：合并了哪些、唯一编号映射
- 改判表：编号 / 原级 / 建议级 / 理由
- 漏报清单：红线维度下的盲区 + 建议补审路径
- 复核结论：全局 P0/P1/P2/P3 修正后数字，给主控做 00_summary.md
不确定标 [待确认]，禁止编造。
```

## 六、出货门禁（与 CLAUDE.md 一致，每个发版里程碑走一遍）

功能冻结 → 全量审计（本流程，10 子系统×三遍 + 复核）→ P0 清零、P1 收敛 → 小批试产实测（重点电源/温度/老化）→ 量产。

## 七、关键设计原则（为什么这么写）

1. **等级定义前置 + 强制判级理由**：口径写死，跨批次占比才可比，避免凑数拔高。
2. **三遍评审分目标**：一遍读完就交必然敷衍；广度→红线→反审三层递进，深层 P0 多在第二遍出，造假在第三遍被自己抓掉。
3. **电源域独立成块**：上轮 4 个 P0 全在电源（如漏乘分压系数会让整批未校准样机开机即关机），不能再和外设挤在一个 agent 里被稀释。
4. **按关注点切共享大文件**：board.cc 上千行同时含电源与外设，用 Grep 锚点把电源切片划给 08、外设切片划给 09，避免重复读又防真空地带。
5. **强制证据格式（文件:行号 + 代码片段）**：真审计与"像审计"的分水岭，便于复核核实。
6. **全局复核 agent**：子 agent 各管一摊容易"局部最优、全局漏边界"；复核专治去重、改判、交界真空、红线漏报。
7. **分段汇报 / 并行隔离**：单会话上下文越长后半段越敷衍；拆 agent 并行各管一摊，质量稳。
8. **大数据文件排除**：生成的字库/图像/固件二进制只看加载逻辑，不逐行读，省上下文。
