# P30 全面压测流程（测什么 · 怎么测 · 自动化 · 多设备联动）

> 目标：① 回归本轮 P0 全清 + 数十条 P1 修复；② 量产门禁压测/老化；③ 多设备联动验证。
> 适用：P30-4G、P30-WiFi 两板型。4G 专项只在 P30-4G。
> 全程录串口日志；每环节标注【验证】对应修复项，便于追溯。

---

## 一、前置（必须先过）

1. 两板型各 `idf.py fullclean && idf.py build` 通过、烧录开机。
2. 冒烟：开机→配网→唤醒→对话→TTS→打断→息屏→充电，全程无崩溃/卡死。
3. 串口确认开机后可用内部 RAM > 60KB（`heap_caps_get_free_size(MALLOC_CAP_INTERNAL)`），并接入"每 10s 打点"用于趋势监控。
4. 准备测试夹具（见 §四工具栏）。

---

## 三、怎么测（方法论）

- **单项功能**：按 §二步骤跑一遍，看现象 + 串口日志。
- **异常注入**：用 mock 服务器/改包/损坏资产主动喂坏数据（测"喂坏不崩"，§二 5-8、20）。
- **压测**：高频循环（200~500 次）暴露偶发竞态/泄漏。
- **老化**：72h 连续使用循环 + 内存趋势 + 看门狗监控（§五）。
- **指标采集**：崩溃/重启次数、coredump、可用内部 RAM 曲线、堆最大空闲块、任务高水位、关键日志（adc read failed / invalid index / 模组未检出 等出现即记录但不应崩）。

## 四、哪些可自动化（工具 + 范围）

**可自动化（推荐建 CI/夹具）：**

| 能自动化的 | 工具/做法 |
|---|---|
| 编译 + 烧录 + 冒烟 | `idf.py build` + esptool 批量烧录，串口抓 "boot ok" 标志 |
| 异常输入压测（JSON/WS/MQTT/资产） | Python mock 服务器自动喂畸形数据 + 串口断言不崩 |
| 重连/OTA/切协议循环 | 脚本驱动 remote_cmd 下发 + 网络损伤（`tc qdisc netem` 模拟弱网/丢包） |
| 唤醒/打断循环 | 音频信号发生器播唤醒词 + 或 remote_cmd `wakeup` 注入；继电器/GPIO 模拟按键 |
| 充电插拔/电量/过放 | 可编程电源（Source Meter）按曲线供电，自动读串口电量日志 |
| 内存/崩溃监控 | 串口日志自动解析：grep panic/coredump、抽取 RAM 曲线、阈值告警 |
| 长稳老化 | 上述循环串成 72h 脚本 + 定时采样 |
| 框架 | **pytest-embedded**（ESP-IDF 官方）统一编排串口断言 |

**难自动化（需人工/专用设备）：**

- 真机**听感**：音质、打断流畅度、有无爆音/杂音/断流 → 人耳，或音频采集 + 频谱比对（半自动）。
- **触摸坐标**准确性 → 机械臂/人工，或固定点位回归。
- **配网手机端**交互（小程序）→ 手工，或 Appium 自动化 App 侧。
- **亮度/显示**观感 → 人眼或亮度计。

## 五、长稳 / 老化（量产门禁）

1. **72h 连续**：唤醒/对话/息屏/充电循环自动化跑 72h。
2. **内存**：每 10s 记录可用内部 RAM、最大空闲块、任务栈高水位；全程**不破 60KB**、无单调下降（泄漏）。
3. **温度老化**：高温(45℃)环境跑 24h，重点电源/充电/4G。
4. **看门狗**：全程无 WDT 复位；记录任何 panic/coredump。

## 六、多设备联动测试（量产关键，单机过≠批量过）

> 量产 = 成千上万台同时在网。很多问题只在并发/拥塞下暴露，单机测不出来。

| 场景 | 怎么测 | 关注 |
|---|---|---|
| N 台同时配网 | 10~50 台同时进 AP/BLE 配网 | AP 信道拥塞、BLE 广播碰撞、配网成功率、是否互相干扰 |
| N 台同时 OTA | 一批同时触发 OTA | OTA 服务器并发/带宽、下载成功率、有无半数砖机 |
| N 台同时唤醒/对话 | 多台同时唤醒连后端 | **服务商 WSS 容量/限流**、后端并发、响应延迟、掉连率 |
| 同一 WiFi 多台共存 | 多台连同一路由 | DHCP 地址池、信道争用、路由器并发连接上限 |
| 多 4G 卡并发 | 多台 4G 同时上网 | 基站/APN 并发、信号互扰、流量 |
| 批量老化 | 机架放 N 台跑 72h | 崩溃率分布、内存分布、个体差异 |

**做法**：设备农场/机架 + 统一串口日志聚合（按 MAC 标识）+ 批量烧录 + 自动化编排脚本。
**指标**：每场景的成功率、崩溃率、内存分布、后端负载曲线；任一台砖机/崩溃都要定位。

---

## 七、通过标准（门禁）

- **P0 相关用例**：零崩溃、零砖机（硬性，任一失败不予放行）。
- **P1 相关用例**：无高频崩溃；偶发记录评估。
- **72h 老化**：无内存泄漏、无 WDT、内部 RAM 不破 60KB。
- **多设备联动**：各场景成功率 ≥ 约定阈值（建议配网/OTA/唤醒 ≥99%）。
- **音频实时**：打断/连续对话/切换/校准无崩、无明显爆音断流（人工听感）。

> 注：音频实时竞态仍有 `01p2-A3`(OGG 跨页) 未修——长播放/弱网解码若复现杂音/崩溃请录日志，作为该项专项轮输入。
