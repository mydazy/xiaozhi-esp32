## 3. 漏报清单（四条硬红线维度的盲区 + 建议补审路径）

> 仅列"本该有但没人报"的盲区。不臆造编号，给补审路径。

### 红线①电源域
- **L1-充电截止/满充检测**：08-P2-E 提到固件不参与充电截止（依赖充电 IC），但**没有任何报告核实充电 IC 的 CHG_DONE/STDBY 是否接线、固件是否需读**。补审：核对 P30 原理图充电 IC 型号与 DONE 引脚走向，确认满充后是否真能降耗（关系发热/涓流）。
- **L2-休眠唤醒源 vs 状态机交界**：08 审了唤醒源配置（EXT0/EXT1/TIMER），07 审了状态机，但**两边都没审"从 deep sleep 唤醒后状态机的冷启动路径是否正确恢复"**——尤其闹钟 TIMER 唤醒后能否直达对应活动态。补审：跟踪 `HandleWakeupCause` → 状态机初始态 → 闹钟/番茄钟活动恢复链（08↔07 交界）。

### 红线②内存安全
- **L3-下游 wifi_config 定长数组**：06-P2-F 指出 SSID/PASSWD 超长风险"落在下游 wifi 子系统"，05 报告**未接住这个交界**（05 审了 SmartConfig 的 memcpy 但没审 blufi 传入的超长 SSID 如何进 wifi_config_t）。补审：`blufi.cc` → `credential_validator_` → wifi_station 的 `wifi_config_t.sta.ssid[32]` 拷贝点是否截断。
- **L4-资产→解码器的外部输入链**：10-P1-C（资产明文下载无验签）与 10-P0-A/10-P1-B（GIF 解码越界）是同一条攻击链，但 04（OTA）报告**未把资产分区下载纳入 OTA 安全治理**（04 只审固件分区）。补审：assets::Download 是否应纳入与固件同级的来源校验（GLB-01 治理域）。

### 红线③并发
- **L5-`i2c_bus_worker` 串行化下的跨子系统抢占**：01-P1-H（校准 Write 与 Read 挤 worker）、09-P3-M（sc7a20h init 持会话锁）都点到 worker 串行化，但**没有报告从全局视角审"codec/touch/accel 三类驱动同时打 worker 时的最坏排队延迟"**——这是 08↔09↔01 三方交界。补审：i2c_bus_worker 队列深度与各 driver 最坏并发占用估算。

### 红线④OTA安全
- **L6-回滚保护已被 04-P1-B 覆盖**（`APP_ANTI_ROLLBACK is not set` 已 grep 确认，line 425/3207），非漏报。
- **L7-OTA 下载 URL 来源**：04-P0-A 提到"URL 来自远程命令明文"，与 04-P0-B（远程命令无鉴权）是放大关系，已交叉引用，非漏报。

### 跨子系统交界真空（任务点名要求）
- **08 电源切片 vs 09 外设切片同在 board.cc/config.h**：已核对——08 明确声明"外设/显示/音频不在范围"，09 明确声明"电源域归 08"。两边都画了线，**未发现 board.cc 里既不属电源也不属外设的真空代码**（如 `ShutdownHandler` 里混合的外设断电+电源逻辑，08 审了断 LDO 对称性 08-P3-G，覆盖到位）。**无真空**。
- **01 audio_codec vs 02 audio_service 衔接**：已核对——01 审 codec 的 Read/Write 返回值（01-P0-A），02 审 InputData/PlaySound 的上层消费。衔接点 `AudioCodec::InputData` 见 `samples>0 即 return true` 被 01-P0-A 覆盖。**无真空**。
- **03 协议层 vs 07 application 的 protocol_ 生命周期**：**已识别为 GLB-02**（03-P1-O 是协议层内 websocket_，07-P0-A 是 application 层 protocol_），两边各审了自己一侧，**全局根因是同一个"连接对象跨线程"治理缺口**，已归并。
