**实测水位**：激活 87.2K → listening 52.2K → **AEC 通话 45.6K（破线 14.4K）**

### 破线主因（已定位，按贡献排序）

| # | 项 | internal 占用 | 证据 |
|---|---|---|---|
| 1 | **wake_word_pcm_ 唤醒前 2s PCM 缓冲，Stop() 不清、通话期白占** | **≈64KB** | afe_wake_word.cc:226-232（66×1KB 小块 <2048 走 internal）；唯一消费方 SEND_WAKE_WORD_DATA 宏未开启 → **清空零功能损失** |
| 2 | AFE VC 实例（AEC 通话）| ≈35K 实测增量 | 首次进 listening 后常驻 |
| 3 | 解码队列上限形同虚设 | 20ms 帧最坏 ~180KB（百度 PCM 960B/包全落 internal） | audio_service.h:48，3600ms 上限 |
| 4 | AFE SR 唤醒实例 | ≈49K | esp-sr v2.3.1 官方 benchmark（MR/SR/HIGH_PERF） |
| 5 | 音频任务栈 | ≈20.5K | input 6K + output 4K + VC 4K + detection 6K |

**修复第 1 项（3 行代码）即可把通话期水位拉回 ~110K，重回红线上方。**

## 二、专项答复（配置 / 唤醒词 / AEC / 音质）

### 1. 配置（sdkconfig）
- 当前是**调试态**：LOG INFO/DEBUG（+292KB flash 实测）、RUNTIME_MONITOR=y。出货：`git checkout HEAD -- sdkconfig` 一步恢复量产态。
- **防回弹机制缺失**：main/Kconfig.projbuild:11 `default y`、sdkconfig.defaults:259-261 写死 INFO/DEBUG/2M 波特率——重生成即回调试态，需 5 行对齐。
- 配置级 internal 油水已不多：WiFi TX 6→4 省 3.2K（仅 WiFi 板，defaults 已预埋目标）；`SPIRAM_MALLOC_ALWAYSINTERNAL 2048→512` 是最后的大杠杆（需 4G 真机 A/B）。
- ❌ 已否决：TASK_WDT_PANIC（主任务阻塞不饿死 idle，TWDT 根本不触发，对已知弱网卡死 P1 收益为零）；FREERTOS 函数入 flash（"必现冲突"论据查实为假但收益不抵回归成本）；D-cache 减半（伤 PSRAM 带宽=伤音频）。

### 2. 唤醒词模型
- **双模型同开**（da1zijing1ling2 + nihaoxiaozhi_tts）：flash assets 各 292KB；运行时经 libesp_audio_front_end.a 反汇编证实**双模型并行检测**：PSRAM +324KB、internal +16KB、core1 每帧 +3ms。
- 注意：这是当日 commit 5d78f8c6 刚交付的**有意产品功能**（不是调试残留）。砍不砍是产品决策：保留 = "你好小智"也能唤醒；砍掉 = 省上述资源并收窄误唤醒面（现场反馈 D2/D3 与此相关）。
- **必修 bug**：set_wakenet_threshold 索引从 0 起传，esp-sr 库要求 1..3（反汇编证实 0 直接返回 -1）→ **0.6 阈值从未生效，唤醒灵敏度一直跑模型默认值**，此前所有围绕 0.6 的调参讨论建立在未生效参数上。修复 = 索引 +1。

### 3. AEC
- internal 破线主因**不是 AEC 算法本身**（大头已走 PSRAM），而是上面表格的 wake_word_pcm_ + 队列 + 任务栈叠加。
- `CONFIG_USE_DEVICE_AEC` 条件编译已在本会话恢复一致（编译产物不变）。
- 运行时开关链路完好：SetAecMode 即时生效不重启（按键③在用）；关 AEC 省 CPU 不省内存（init 时已分配）。
- 新发现：SetFrameDuration 协议切换时 Initialize 重入 → 旧 AFE 实例整体泄漏（MB 级 PSRAM）+ 双任务竞态，5 行幂等守卫修复。

### 4. 音质参数（工作区未提交）
- vad_mode 0→1、linear_gain 4.0→3.0、at_uart RX 阈值 64、提示音重构（双 bool→指针）——**必须真机 A/B 回归后 commit 或回退，不能裸混出货构建**。

### 新发现 P1（验证通过，按伤害排序）
1. **AlarmNvsOp 含 std::string 过 FreeRTOS 队列**（memcpy UAF，每次语音设闹钟必经，短消息几乎必现脏读）→ POD 化 ~15 行
2. **i2c_bus_worker 超时弃单后写已弹栈的 caller 栈**（总线风暴下随机内存损坏；codec 失电短路 SDA 量产真实发生过）→ 堆上下文 ~15 行
3. **闹钟 ≤30min 时深睡不 arm RTC 唤醒 → 必漏响**（"20 分钟后提醒我"/贪睡场景）→ 修法采用"放弃深睡只息屏"（提前 60s 方案被数学证伪：RC 漂移 ±87s > 60s）；电源域红线流程
4. **4G modem 检测/注册失败 → 永卡 Starting 不休眠 → 整夜耗电**（叠加 08-P0-A 物理伤害）→ 失败兜底 EnterDeepSleep ~10 行

6. **过放关机最后一跳走主线程 Schedule**——弱网主线程阻塞时保护哑火 → timer 任务硬关机兜底 ~12 行
7. **pickupWake 默认开**（284ea1bd 自标"危险态待配套"）→ E1/E2/G1/E3 扰民向量默认复活；默认值 1→0（1 字符），出货门禁 blocker

10. **解码队列 cv.wait 无超时**：阻塞 4G EventTask（弱网卡死同根）+ OTA 期闹钟 PlaySound 冻结 esp_timer 任务（电池监控/过放保护停摆）→ wait_for 5s + service_stopped_ 谓词 2 行

14. **Settings 写路径 ESP_ERROR_CHECK abort**（NVS 满 → 重启循环砖机）→ 去 abort ~10 行（含 :33/:107/:116 遗留 3 处）
